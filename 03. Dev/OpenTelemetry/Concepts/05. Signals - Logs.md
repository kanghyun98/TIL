# Logs

- **Log**는 타임 스탬프가 포함된 텍스트 기록입니다. 선택적으로 metadata를 포함할 수 있습니다.

<br>

## **OpenTelemetry logs**

- OpenTelemetry는 log record를 생성하기 위한 Logs API와 SDK를 제공합니다.
  - 또한 기존 logging framework와 통합하기 위해 언어별 SDK와 logging bridge를 제공합니다.

- Logging Provider를 통해 보내는 모든 것은 log입니다. Event는 log의 특별한 유형입니다.
  - 즉, 모든 log가 event인 것은 아니지만, 모든 event는 log입니다.

- Logs API는 public API이며, application code에서 직접 사용할 수도 있고, 기존 logging library와 bridge를 통해 간접적으로 사용할 수도 있습니다.

- OpenTelemetry는 이미 생성하고 있는 log와 함께 동작하도록 설계되었습니다.

- OpenTelemetry는 다음을 위한 도구를 제공합니다.

  - log를 다른 signal과 연관시키기

  - contextual attribute 추가하기

  - 서로 다른 source를 처리 및 export를 위한 공통 표현으로 정규화하기  

<br>

### **OpenTelemetry logs in the OpenTelemetry Collector**

- OpenTelemetry Collector는 log를 다루기 위한 여러 도구를 제공합니다.

  - 특정하고 알려진 log data source에서 log를 파싱하는 여러 receiver

  - `filelogreceiver`
    - 임의의 file에서 log를 읽음
    - 여러 format에서 log를 parse하거나 regular expression을 사용할 수 있는 기능 제공

  - `transformprocessor` 같은 processor
    - nested data parse
    - nested structure flatten
    - value 추가, 제거, 업데이트 등 수행

  - OpenTelemetry 형식이 아닌 format으로 log data를 내보낼 수 있는 exporter

<br>

### **OpenTelemetry logs for applications**

- Application에서 OpenTelemetry log는 어떤 logging library나 built-in logging 기능을 통해 생성됩니다.

- Auto-instrumentation을 추가하거나 SDK를 활성화하면, OpenTelemetry는 기존 log를 active trace 및 span과 자동으로 연관시킵니다.

  - 이때 log body는 trace ID와 span ID로 감싸집니다.

  - 즉, OpenTelemetry는 log와 trace를 자동으로 correlate 합니다.  

<br>

### Structured, unstructured, and semistructured logs

- OpenTelemetry는 어떤 log format이든 허용합니다. 하지만 모든 format이 분석에 똑같이 유용한 것은 아닙니다.

- 중요한 점은, JSON으로 encoding된 log라고 해서 안정적인 schema를 가진 “structured” log가 자동으로 되는 것은 아니라는 점입니다. 이런 log는 semistructured일 수 있습니다.

- Structured log는 downstream processing이 안정적으로 의존할 수 있는 일관된 schema 또는 잘 정의된 typed field를 의미합니다.  

<br>

### **Structured logs**

- Structured log는 downstream system이 안정적으로 파싱하고 해석할 수 있는 정의된 일관된 schema 또는 typed field를 가진 log입니다.

  - 텍스트 encoding은 JSON, protobuf 또는 다른 format일 수 있습니다.

  - 하지만 log를 structured하게 만드는 것은 단순히 유효한 JSON이라는 점이 아닌, 안정적인 schema입니다.

    - field name

    - type

    - semantics


- Structured JSON log 예시는 다음과 같습니다.

  ```json
  {
    "timestamp": "2024-08-04T12:34:56.789Z",
    "level": "INFO",
    "service": "user-authentication",
    "environment": "production",
    "message": "User login successful",
    "context": {
      "userId": "12345",
      "username": "johndoe",
      "ipAddress": "192.168.1.1",
      "userAgent": "Mozilla/5.0 (Windows NT 10.0; Win64; x64) AppleWebKit/537.36 (KHTML, like Gecko) Chrome/104.0.0.0 Safari/537.36"
    },
    "transactionId": "abcd-efgh-ijkl-mnop",
    "duration": 200,
    "request": {
      "method": "POST",
      "url": "/api/v1/login",
      "headers": {
        "Content-Type": "application/json",
        "Accept": "application/json"
      },
      "body": {
        "username": "johndoe",
        "password": "******"
      }
    },
    "response": {
      "statusCode": 200,
      "body": {
        "success": true,
        "token": "jwt-token-here"
      }
    }
  }
  ```

- Infrastructure component에서는 Common Log Format(CLF)가 자주 사용됩니다.

  ```
  127.0.0.1 - johndoe [04/Aug/2024:12:34:56 -0400] "POST /api/v1/login HTTP/1.1" 200 1234
  ```

- Hybrid format이나 extended format도 흔히 볼 수 있습니다.

  - 예를 들어 CLF field와 뒤쪽의 JSON blob을 함께 사용하는 방식입니다.

  ```json
  192.168.1.1 - johndoe [04/Aug/2024:12:34:56 -0400] "POST /api/v1/login HTTP/1.1" 200 1234 "http://example.com" "Mozilla/5.0 (Windows NT 10.0; Win64; x64) AppleWebKit/537.36 (KHTML, like Gecko) Chrome/104.0.0.0 Safari/537.36" {"transactionId": "abcd-efgh-ijkl-mnop", "responseTime": 150, "requestBody": {"username": "johndoe"}, "responseHeaders": {"Content-Type": "application/json"}}
  ```

  - 이런 경우 downstream tooling이 일관되게 분석할 수 있도록 필요한 부분을 parse하거나 추출해서 normalized record로 만들어야 합니다.

- OpenTelemetry Collector의 `filelogreceiver`는 mixed format을 parse할 수 있는 helper를 제공합니다.

<br>

### **Unstructured logs**

- Unstructured log는 일관된 구조를 따르지 않는 log입니다.

  - 사람이 읽기에는 더 쉬울 수 있으며, development 환경에서 자주 사용됩니다.

  - 하지만 production observability 목적에서는 unstructured log 사용이 선호되지 않습니다.

- Unstructured log 예시는 다음과 같습니다.

  ```txt
  [ERROR] 2024-08-04 12:45:23 - Failed to connect to database. Exception: java.sql.SQLException: Timeout expired. Attempted reconnect 3 times. Server: db.example.com, Port: 5432
  System reboot initiated at 2024-08-04 03:00:00 by user: admin. Reason: Scheduled maintenance. Services stopped: web-server, database, cache. Estimated downtime: 15 minutes.
  DEBUG - 2024-08-04 09:30:15 - User johndoe performed action: file_upload. Filename: report_Q3_2024.pdf, Size: 2.3 MB, Duration: 5.2 seconds. Result: Success
  ```

<br>

### Semistructured logs

- Semistructured log는 machine-readable한 key/value pair 또는 delimited field를 포함합니다.

  - 하지만 emitter 간에 안정적인 schema를 보장하지는 않습니다.

  - 예시는 다음과 같습니다.

    - `key=value` logging

    - message마다 field name과 type이 달라지는 JSON blob


  - Semistructured log 예시는 다음과 같습니다.

  ```txt
  2024-08-04T12:45:23Z level=ERROR service=user-authentication userId=12345 action=login message="Failed login attempt" error="Invalid password" ipAddress=192.168.1.1 userAgent="Mozilla/5.0 (Windows NT 10.0; Win64; x64) AppleWebKit/537.36 (KHTML, like Gecko) Chrome/104.0.0.0 Safari/537.36"
  ```

- Semistructured log가 downstream analysis에 충분히 유용해지려면 ingestion 중에 mapping과 type coercion이 필요할 수 있습니다.  

<br>

## **OpenTelemetry logging components**

다음 개념과 component들이 OpenTelemetry의 logging 지원을 구성합니다.  

### Log Appender / Bridge

- Application developer는 Logs Bridge API를 직접 호출하면 안 됩니다.

  - 이 API는 logging library author가 log appender 또는 bridge를 만들 수 있도록 제공됩니다.

  - 대신 개발자는 선호하는 logging library를 사용하고, 해당 library가 OpenTelemetry `LogRecordExporter`로 log를 내보낼 수 있는 log appender 또는 log bridge를 사용하도록 설정하면 됩니다.

- OpenTelemetry language SDK는 이 기능을 제공합니다.  

<br>

### Logger Provider

- Logs Bridge API의 일부이며, logging library author인 경우에만 사용해야 합니다.
- Logger Provider, 또는 `LoggerProvider`는 `Logger`를 생성하는 factory입니다.

- 대부분의 경우 Logger Provider는 한 번 초기화되며, 생명주기는 application의 생명주기와 일치합니다.

- Logger Provider 초기화에는 Resource와 Exporter 초기화도 포함됩니다.  

<br>

### **Logger**

- Logs Bridge API의 일부이며, logging library author인 경우에만 사용해야 합니다.

- Logger는 log record를 생성합니다.

- Logger는 Log Provider로부터 생성됩니다.  

<br>

### **Log Record Exporter**

- Log Record Exporter는 log record를 consumer로 보냅니다.

- 이 consumer는 다음 중 하나일 수 있습니다.

  - debugging 및 development 시점에 사용하는 standard output

  - OpenTelemetry Collector

  - 선택한 open source backend

  - 선택한 vendor backend  


<br>

### **Log Record**

- Log record는 event의 기록을 나타냅니다.

- OpenTelemetry에서 log record는 두 종류의 field를 포함합니다.

  - 특정 type과 의미를 가진 named top-level field

  - 임의의 value와 type을 가진 Resource 및 attributes field


- Top-level field는 다음과 같습니다.  

| **Field Name**         | **Description**             |
| ---------------------- | --------------------------- |
| `Timestamp`            | Event가 발생한 시간         |
| `ObservedTimestamp`    | Event가 관측된 시간         |
| `TraceId`              | Request trace ID            |
| `SpanId`               | Request span ID             |
| `TraceFlags`           | W3C trace flag              |
| `SeverityText`         | Severity text, 즉 log level |
| `SeverityNumber`       | Severity의 숫자 값          |
| `Body`                 | Log record의 body           |
| `Resource`             | Log의 source 설명           |
| `InstrumentationScope` | Log를 emit한 scope 설명     |
| `Attributes`           | Event에 대한 추가 정보      |
