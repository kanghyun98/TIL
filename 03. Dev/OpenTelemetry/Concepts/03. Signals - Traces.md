# Traces

- 애플리케이션을 통과하는 요청의 경로입니다.
- Trace는 애플리케이션에 요청이 들어왔을 때 어떤 일이 발생하는지 큰 그림을 보여줍니다. 
  - 애플리케이션이 하나의 데이터베이스를 가진 모놀리스이든, 복잡한 서비스 메시 구조이든, trace는 요청이 애플리케이션 안에서 거치는 전체 “경로”를 이해하는 데 필수적입니다.  

- 이 내용을 세 개의 작업 단위로 살펴보겠습니다. 각 작업 단위는 `Span`으로 표현됩니다.

```json
// `hello` span
{
  "name": "hello",
  "context": {
    "trace_id": "5b8aa5a2d2c872e8321cf37308d69df2",
    "span_id": "051581bf3cb55c13"
  },
  "parent_id": null,
  "start_time": "2022-04-29T18:52:58.114201Z",
  "end_time": "2022-04-29T18:52:58.114687Z",
  "attributes": {
    "http.route": "some_route1"
  },
  "events": [
    {
      "name": "Guten Tag!",
      "timestamp": "2022-04-29T18:52:58.114561Z",
      "attributes": {
        "event_attributes": 1
      }
    }
  ]
}
```

- 이 span은 root span입니다. 전체 operation의 시작과 끝을 나타냅니다.
  - `trace_id` 필드는 trace를 나타내지만, `parent_id`는 없습니다. 그래서 이것이 root span이라는 것을 알 수 있습니다.

```json
// `hello-greetings` span
{
  "name": "hello-greetings",
  "context": {
    "trace_id": "5b8aa5a2d2c872e8321cf37308d69df2",
    "span_id": "5fb397be34d26b51"
  },
  "parent_id": "051581bf3cb55c13",
  "start_time": "2022-04-29T18:52:58.114304Z",
  "end_time": "2022-04-29T22:52:58.114561Z",
  "attributes": {
    "http.route": "some_route2"
  },
  "events": [
    {
      "name": "hey there!",
      "timestamp": "2022-04-29T18:52:58.114561Z",
      "attributes": {
        "event_attributes": 1
      }
    },
    {
      "name": "bye now!",
      "timestamp": "2022-04-29T18:52:58.114585Z",
      "attributes": {
        "event_attributes": 1
      }
    }
  ]
}
```

- 이 span은 인사말을 말하는 것과 같은 특정 작업을 캡슐화합니다. 그리고 부모는 `hello` span입니다.
  - root span과 같은 `trace_id`를 공유하므로 같은 trace의 일부임을 알 수 있습니다. 또한 `parent_id`는 `hello` span의 `span_id`와 일치합니다.

```json
// `hello-salutations` span
{
  "name": "hello-salutations",
  "context": {
    "trace_id": "5b8aa5a2d2c872e8321cf37308d69df2",
    "span_id": "93564f51e1abe1c2"
  },
  "parent_id": "051581bf3cb55c13",
  "start_time": "2022-04-29T18:52:58.114492Z",
  "end_time": "2022-04-29T18:52:58.114631Z",
  "attributes": {
    "http.route": "some_route3"
  },
  "events": [
    {
      "name": "hey there!",
      "timestamp": "2022-04-29T18:52:58.114561Z",
      "attributes": {
        "event_attributes": 1
      }
    }
  ]
}
```

- 이 span은 이 trace 안의 세 번째 operation을 나타냅니다. 
  - 앞의 span과 마찬가지로 `hello` span의 child입니다. 
  - 따라서 `hello-greetings` span과는 sibling 관계입니다.

- 이 세 JSON 블록은 모두 같은 `trace_id`를 공유하고, `parent_id` 필드는 계층 구조를 나타냅니다. 이것이 Trace입니다.

- 각 Span이 구조화된 로그처럼 보입니다.
- Trace를 이해하는 한 가지 방식은, context, correlation, hierarchy 등이 포함된 구조화 로그들의 모음으로 보는 것입니다.

- 다만 이 “구조화 로그”들은 서로 다른 process, service, VM, data center 등에서 올 수 있습니다. 이것이 tracing이 어떤 시스템이든 end-to-end 관점으로 표현할 수 있게 하는 이유입니다.

## Tracer Provider

- Tracer Provider는 `Tracer`를 생성하는 factory입니다.

- 대부분의 애플리케이션에서 Tracer Provider는 한 번 초기화되며, 생명주기는 애플리케이션의 생명주기와 일치합니다.
  - Tracer Provider 초기화에는 Resource와 Exporter 초기화도 포함됩니다.

## Tracer

- Tracer는 특정 operation에서 무슨 일이 일어나는지에 대한 더 많은 정보를 담은 span을 생성합니다.
  - 예를 들어 서비스의 요청이 하나의 operation이 될 수 있습니다.

- Tracer는 Tracer Provider로부터 생성됩니다.  

## **Trace Exporters**

- Trace Exporter는 trace를 consumer로 보냅니다.

- 이 consumer는 다음 중 하나일 수 있습니다.

  - 디버깅 및 개발 시점에 사용하는 standard output

  - OpenTelemetry Collector

  - 선택한 open source backend

  - 선택한 vendor backend  

## Context Propagation

- Context Propagation은 Distributed Tracing을 가능하게 하는 핵심 개념입니다.

- Context Propagation을 통해 Span이 어디에서 생성되었든 서로 연관될 수 있고, 하나의 trace로 조립될 수 있습니다.

## Spans

- Span은 작업 단위 또는 operation을 나타냅니다.

- Span은 Trace를 구성하는 building block입니다.

- OpenTelemetry에서 span은 다음 정보를 포함합니다.

  - Name
    - Parent span ID
      - root span의 경우 비어 있음

  - Start Timestamp

  - End Timestamp

  - Span Context

  - Attributes

  - Span Events

  - Span Links

  - Span Status

- Span은 중첩될 수 있습니다. 이는 parent span ID의 존재로 암시됩니다.

- Child span은 하위 operation을 나타냅니다. 이를 통해 span은 애플리케이션에서 수행된 작업을 더 정확하게 포착할 수 있습니다.  

### Span Context

- Span Context는 모든 span에 존재하는 immutable object입니다.

- Span Context는 다음 정보를 포함합니다.

  - 해당 span이 속한 trace를 나타내는 Trace ID

  - span의 Span ID
    - Trace Flags
      - trace에 대한 정보를 담는 binary encoding

  - Trace State
    - vendor-specific trace 정보를 전달할 수 있는 key-value pair 목록

- Span Context는 span에서 직렬화되어 Distributed Context 및 Baggage와 함께 전파되는 부분입니다.

- Span Context에는 Trace ID가 포함되므로, Span Links를 만들 때 사용됩니다.  

### **Attributes**

- Attributes는 key-value pair입니다.

- Span이 추적하는 operation에 대한 정보를 담기 위해, span에 metadata를 붙이는 데 사용할 수 있습니다.

  - 예를 들어 eCommerce 시스템에서 사용자의 장바구니에 상품을 추가하는 operation을 span이 추적한다고 가정해보겠습니다.

  - 이 경우 다음 정보를 기록할 수 있습니다.

    - 사용자 ID

    - 장바구니에 추가할 상품 ID

    - 장바구니 ID

- Attributes는 span 생성 중이나 생성 후에 추가할 수 있습니다.

- SDK sampling에서 attributes를 사용할 수 있도록, 가능하면 span 생성 시점에 attributes를 추가하는 것이 좋습니다.
  - span 생성 후 값을 추가해야 한다면, 해당 값으로 span을 업데이트합니다.

- Semantic Attributes

  - Semantic Attributes는 일반적인 operation에서 보통 나타나는 metadata에 대한 알려진 naming convention입니다.

  - 가능하다면 semantic attribute naming을 사용하는 것이 좋습니다. 이렇게 하면 시스템 전반에서 공통 metadata 유형을 표준화할 수 있습니다.  

### **Span Events**

- Span Event는 Span 위에 기록되는 구조화된 log message 또는 annotation으로 생각할 수 있습니다.

- 일반적으로 Span이 지속되는 동안 발생한 의미 있는 단일 시점을 나타내는 데 사용됩니다.

  - 예를 들어 웹 브라우저에서 다음 두 시나리오를 생각해볼 수 있습니다.
    1. page load 추적
    2. page가 interactive해지는 시점 표시

  - 첫 번째 시나리오는 시작과 끝이 있는 operation이므로 Span으로 추적하는 것이 가장 적절합니다.

  - 두 번째 시나리오는 의미 있는 단일 시점을 나타내므로 Span Event로 추적하는 것이 가장 적절합니다.  

#### **When to use span events versus span attributes**

- Span event도 attributes를 포함하므로, 언제 event를 사용하고 언제 attribute를 사용할지 항상 명확하지 않을 수 있습니다.

- 판단할 때는 특정 timestamp가 의미 있는지를 고려합니다.

  - 예를 들어 어떤 operation을 span으로 추적하고 있고, 해당 operation이 완료되었을 때 operation의 데이터를 telemetry에 추가하고 싶다고 가정합니다.

  - operation이 완료된 timestamp가 의미 있거나 관련이 있다면, 데이터를 span event에 붙입니다.

  - timestamp가 의미 없다면, 데이터를 span attributes로 붙입니다.  

### **Span Links**

- Link는 하나의 span을 하나 이상의 span과 연결하여 인과 관계를 나타내기 위해 존재합니다.

- 예를 들어 일부 operation이 trace로 추적되는 distributed system이 있다고 가정합니다.

  - 이 operation들에 대한 응답으로 추가 operation이 queue에 들어가고, 나중에 비동기적으로 실행될 수 있습니다.

  - 이 후속 operation도 trace로 추적할 수 있습니다.

  - 후속 operation의 trace를 첫 번째 trace와 연결하고 싶지만, 후속 operation이 언제 시작될지는 예측할 수 없습니다.
  - 두 trace를 연결해야 하므로 span link를 사용합니다.

  - 첫 번째 trace의 마지막 span을 두 번째 trace의 첫 번째 span에 link할 수 있습니다.

  - 그러면 두 span은 서로 인과 관계를 갖게 됩니다.

- Link는 선택 사항이지만, trace span들을 서로 연결하는 좋은 방법입니다.

### **Span Status**

- 각 span에는 status가 있습니다.

  - `Unset`

  - `Error`

  - `Ok`

- 기본값은 `Unset`입니다.
- span status가 `Unset`이면, 해당 span이 추적한 operation이 에러 없이 성공적으로 완료되었음을 의미합니다.
- span status가 `Error`이면, 해당 span이 추적한 operation에서 어떤 에러가 발생했음을 의미합니다.
  - 예를 들어 서버가 요청을 처리하는 중 HTTP 500 에러가 발생한 경우가 여기에 해당할 수 있습니다.

- span status가 `Ok`이면, 애플리케이션 개발자가 해당 span을 명시적으로 에러 없음으로 표시했다는 의미입니다.
  - 직관적이지 않을 수 있지만, span이 에러 없이 완료되었다는 것을 알고 있더라도 반드시 `Ok`로 설정할 필요는 없습니다. 이 경우는 `Unset`으로도 충분히 표현됩니다.
  - `Ok`는 사용자가 명시적으로 설정한 span status에 대해 모호하지 않은 “최종 판단”을 나타냅니다.

- 다시 정리하면 다음과 같습니다.

  - `Unset`: span이 에러 없이 완료됨

  - `Ok`: 개발자가 span을 명시적으로 성공으로 표시함

### **Span Kind**

- Span이 생성될 때, span은 다음 중 하나의 종류를 가집니다.

  - `Client`

  - `Server`

  - `Internal`

  - `Producer`

  - `Consumer`

- 이 span kind는 tracing backend가 trace를 어떻게 조립해야 하는지에 대한 힌트를 제공합니다.
- OpenTelemetry specification에 따르면 server span의 parent는 종종 remote client span입니다.
- client span의 child는 일반적으로 server span입니다.

- 마찬가지로 consumer span의 parent는 항상 producer이고, producer span의 child는 항상 consumer입니다.

- 제공되지 않으면 span kind는 internal로 간주됩니다.



#### **Client**

- Client span은 synchronous(동기식) outbound remote call을 나타냅니다.

- 예시는 다음과 같습니다.

  - outbound HTTP request

  - database call

- 여기서 “synchronous”는 `async/await`를 의미하지 않습니다.
- 나중에 처리하기 위해 queue에 들어가는 작업이 아니라는 의미입니다.

#### **Server**

- Server span은 synchronous inbound remote call을 나타냅니다.

- 예시는 다음과 같습니다.

  - incoming HTTP request

  - remote procedure call

#### **Internal**

- Internal span은 process boundary를 넘지 않는 operation을 나타냅니다.

- 예를 들어 function call을 instrumenting(계측)하거나 Express middleware를 instrumenting할 때 internal span을 사용할 수 있습니다.

#### **Producer**

- Producer span은 나중에 비동기적으로 처리될 수 있는 job의 생성을 나타냅니다.

- 이는 job queue에 삽입된 remote job일 수도 있고, event listener가 처리하는 local job일 수도 있습니다.

#### **Consumer**

- Consumer span은 producer가 만든 job의 처리를 나타냅니다.
- Consumer span은 producer span이 이미 종료된 뒤 한참 후에 시작될 수도 있습니다.
