# Context propagation

> "Distributed Tracing"을 가능하게 하는 개념을 알아봅니다.

> : 분산 시스템이나 비동기 환경에서 트레이스 ID, 사용자 정보 등 핵심 데이터(컨텍스트)를 서비스 간 또는 스레드 간에 유지하며 전달하는 기술

- Context propagation(컨텍스트 전파)을 사용하면 signal, 즉 trace, metric, log가 어디에서 생성되었는지와 관계없이 서로 연관될 수 있습니다.

- Context propagation은 tracing에만 한정되지는 않습니다. 하지만 trace가 임의로 분산된 서비스, 프로세스, 네트워크 경계를 넘어 시스템의 인과 관계 정보를 구성할 수 있게 해줍니다.

- Context propagation을 이해하려면 두 가지 개념을 이해해야 합니다.

  - Context

  - Propagation  

## Context

- Context는 보내는 서비스와 받는 서비스, 또는 execution unit이 하나의 signal을 다른 signal과 연관시키기 위해 필요한 정보를 담는 객체입니다.
  - Service A가 Service B를 호출하면, Service A는 context의 일부로 trace ID와 span ID를 포함합니다.
  - Service B는 이 값을 사용해 같은 trace에 속하는 새 span을 생성하고, Service A의 span을 부모로 설정합니다.
  - 이를 통해 서비스 경계를 넘어 요청의 전체 흐름을 추적할 수 있습니다.  

## Propagation

- Propagation은 서비스와 프로세스 사이에서 context를 이동시키는 메커니즘입니다.

- Propagation은 context 객체를 직렬화하거나 역직렬화하고, 한 서비스에서 다른 서비스로 전파해야 하는 관련 정보를 제공합니다.

- Propagation은 보통 instrumentation library가 처리하며, 사용자에게는 투명하게 동작합니다.

- Context를 수동으로 전파해야 하는 경우에는 Propagators API를 사용할 수 있습니다.
  - OpenTelemetry는 여러 공식 propagator를 유지합니다.
  - 기본 propagator는 W3C TraceContext 명세에서 정의한 header를 사용합니다.  

## Example

- `Frontend`라는 서비스가 있다고 가정합니다. 이 서비스는 다음과 같은 여러 HTTP endpoint를 제공합니다.

  - `POST /cart/add`

  - `GET /checkout/`

- 그리고 사용자가 장바구니에 추가하려는 상품이나 checkout에 포함된 상품의 상세 정보를 받기 위해, downstream service인 `Product Catalog`의 HTTP endpoint `GET /product`를 호출합니다.

- `Frontend`에서 들어온 요청의 context 안에서 `Product Catalog` 서비스의 활동을 이해하려면 context가 전파되어야 합니다.

  - 여기서 context는 Trace ID와 Span ID를 Parent ID로 사용하는 정보입니다.

  - 이 context는 W3C TraceContext 명세에 정의된 `traceparent` header를 통해 전파됩니다.

  - 즉, ID들은 header의 field 안에 포함됩니다.

```txt
<version>-<trace-id>-<parent-id>-<trace-flags>
```

- 예시

```txt
00-a0892f3577b34da6a3ce929d0e0e4736-f03067aa0ba902b7-01
```

### Traces

- context propagation은 trace가 서비스 간 인과 관계 정보를 구성할 수 있게 해줍니다.

- 이 예시에서는 `Product Catalog` 서비스의 HTTP endpoint `GET /product`에 대한 두 번의 호출을 `Frontend` 서비스의 upstream 호출과 연관시킬 수 있습니다.

- 이를 위해 `traceparent` header에서 remote context를 추출하고, 이를 local context에 주입하여 Trace ID와 Parent ID를 설정합니다.

- 그 결과 Jaeger 같은 backend에서 두 요청을 하나의 trace에 속한 span으로 볼 수 있습니다.  

### Logs

- OpenTelemetry SDK는 log를 trace와 자동으로 연관시킬 수 있습니다.
  - 즉, context, 예를 들어 Trace ID와 Span ID를 log record에 주입할 수 있습니다.

- 이를 통해 해당 log가 속한 trace와 span의 context 안에서 log를 볼 수 있습니다.

- 또한 서비스 또는 execution unit 경계를 넘어 서로 관련된 log를 함께 볼 수 있습니다.  

### Metrics

- Metric의 경우 context propagation은 해당 context 안에서 measurement를 집계할 수 있게 해줍니다.

- 예를 들어 모든 `GET /product` 요청의 response time만 보는 대신, 다음 조합에 대한 metric도 볼 수 있습니다.

  - `POST /cart/add > GET /product`

  - `GET /checkout < GET /product`

| **Name**                        | **Calls Per Second** | **Average Response Time** |
| ------------------------------- | -------------------- | ------------------------- |
| `* > GET /product`              | 370                  | 300ms                     |
| `POST /card/add > GET /product` | 330                  | 130ms                     |
| `GET /checkout > GET /product`  | 40                   | 1703ms                    |

## Custom Context Propagation

- 대부분의 사용 사례에서는 context propagation을 대신 처리해주는 instrumentation library 또는 native library instrumentation을 찾을 수 있습니다. 하지만 일부 경우에는 이런 지원이 없어서 직접 만들어야 할 수 있습니다.

- 이를 위해 앞서 언급한 Propagators API를 활용해야 합니다.

- 보내는 쪽에서는 context가 carrier에 injected 됩니다.
  - 예: HTTP 요청 header에 context를 주입
  - 다른 경우에는 request의 metadata를 저장할 수 있는 위치를 찾아야 합니다.
- 받는 쪽에서는 context가 carrier에서 extracted 됩니다.
  - HTTP의 경우 header에서 context를 가져옵니다.
  - 다른 경우에는 보내는 쪽에서 context를 저장하기로 선택한 위치에서 가져옵니다.

- Metadata 전용 field가 없는 protocol에서도 context를 전파할 수 있습니다.
  - 다만 받는 쪽에서 데이터가 처리되기 전에 context를 추출하고 제거해야 합니다.
  - 그렇지 않으면 undefined behavior가 발생할 수 있습니다.

## **Security best practices**

- Propagation은 서비스 경계를 넘어 데이터를 보내고 받는 것을 포함하므로, 보안상 영향을 줄 수 있습니다.

### **External services**

- 서비스가 외부 서비스, 즉 소유하거나 신뢰하지 않는 서비스와 상호작용하는 경우 다음 사항을 고려해야 합니다.

**Incoming context**

- 외부 source에서 context를 받을 때는 주의해야 합니다.
- 악의적인 사용자가 위조된 trace header를 보내 tracing data를 조작하거나, context parsing의 취약점을 악용할 수 있습니다.
- 신뢰할 수 없는 source에서 들어오는 context는 무시하거나 sanitize하는 것이 좋을 수 있습니다.

**Outgoing context**

- 외부 서비스로 무엇을 전파할지 주의해야 합니다.
- Internal trace ID, span ID, baggage item은 내부 architecture나 business logic에 대한 민감한 정보를 드러낼 수 있습니다.
- External endpoint 또는 public-facing endpoint로 context를 보내지 않도록 propagator를 설정하는 것이 좋을 수 있습니다.  

### **Baggage**

- Baggage는 임의의 key-value pair를 전파할 수 있게 해줍니다.
- 이 데이터는 서비스 경계를 넘어 전파되므로, baggage에 민감한 정보를 넣지 않아야 합니다.

- 예시는 다음과 같습니다.

  - 사용자 credential

  - API key

  - PII, 즉 개인 식별 정보

- 이런 정보는 log에 기록되거나 신뢰할 수 없는 downstream service로 전송될 수 있습니다.
