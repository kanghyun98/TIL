# Metrics

- 런타임에 캡처되는 측정값입니다.

- 측정값을 캡처하는 순간을 metric event라고 하며, metric event는 측정값 자체뿐만 아니라 측정된 시간과 관련 metadata도 포함합니다.

- application metric과 request metric은 availability와 performance를 나타내는 중요한 indicator입니다.

- custom metric은 availability indicator가 사용자 경험이나 비즈니스에 어떤 영향을 주는지에 대한 insight를 제공할 수 있습니다.

- 수집된 데이터는 다음과 같은 용도로 사용할 수 있습니다.

  - 장애 발생 시 alert

  - 수요가 높을 때 deployment를 자동으로 scale up하기 위한 scheduling decision trigger

## Meter Provider

- Meter Provider, 또는 `MeterProvider`는 `Meter`를 생성하는 factory입니다.

- 대부분의 애플리케이션에서 Meter Provider는 한 번 초기화되며, 생명주기는 애플리케이션의 생명주기와 일치합니다.

  - Meter Provider 초기화에는 Resource와 Exporter 초기화도 포함됩니다.

  - OpenTelemetry에서 metering을 시작할 때 일반적으로 가장 먼저 수행하는 단계입니다.

## Meter

- Meter는 metric instrument를 생성합니다.

- Metric instrument는 런타임에 서비스에 대한 측정값을 캡처합니다.

- Meter는 Meter Provider로부터 생성됩니다.  

## Metric Exporter

- Metric Exporter는 metric data를 consumer로 보냅니다.

- 이 consumer는 다음 중 하나일 수 있습니다.

  - 개발 중 debugging을 위한 standard output

  - OpenTelemetry Collector

  - 선택한 open source backend

  - 선택한 vendor backend  

## Metric Instruments

- OpenTelemetry에서 측정값은 metric instrument에 의해 캡처됩니다.

- Metric instrument는 다음 항목으로 정의됩니다.

  - Name

  - Kind
    - Unit
      - optional

  - Description
    - optional

- Name, unit, description은 개발자가 선택하거나, request metric이나 process metric처럼 일반적인 항목의 경우 semantic conventions를 통해 정의됩니다.

- Instrument kind는 다음 중 하나입니다.

  - **Counter**
    - 시간이 지남에 따라 누적되는 값입니다.
    - 자동차의 주행 거리계처럼 생각할 수 있습니다.
    - 값은 항상 증가하기만 합니다.

  - **Asynchronous Counter**
    - Counter와 동일하지만, export마다 한 번씩 수집됩니다.
    - 지속적인 증가량에는 접근할 수 없고, 집계된 값에만 접근할 수 있는 경우 사용할 수 있습니다.

  - **UpDownCounter**
    - 시간이 지남에 따라 누적되는 값이지만, 다시 감소할 수도 있습니다.
    - 예를 들어 queue length가 될 수 있습니다.
    - queue에 있는 work item 수에 따라 증가하거나 감소합니다.

  - **Asynchronous UpDownCounter**
    - UpDownCounter와 동일하지만, export마다 한 번씩 수집됩니다.
    - 지속적인 변화에는 접근할 수 없고, 집계된 값에만 접근할 수 있는 경우 사용할 수 있습니다.
    - 예: 현재 queue size

  - **Gauge**
    - 읽히는 시점의 현재 값을 측정합니다.
    - 예를 들어 차량의 fuel gauge가 있습니다.
    - Gauge는 synchronous입니다.

  - **Asynchronous Gauge**
    - Gauge와 동일하지만, export마다 한 번씩 수집됩니다.
    - 지속적인 변화에는 접근할 수 없고, 집계된 값에만 접근할 수 있는 경우 사용할 수 있습니다.

  - **Histogram**
    - request latency 같은 값들을 client-side에서 집계한 것입니다.
    - 값의 통계에 관심이 있을 때 적합합니다.
    - 예: 1초보다 적게 걸리는 요청은 몇 개인가?

## **Aggregation**

- Metric instrument 외에도 aggregation(집합) 개념을 이해하는 것이 중요합니다.
- Aggregation은 많은 수의 측정값을 결합하여, 특정 time window 동안 발생한 metric event에 대한 정확하거나 추정된 통계를 만드는 기법입니다.
- OpenTelemetry API는 각 instrument에 대한 default aggregation을 제공합니다.
  - 이 default aggregation은 Views를 사용해 override할 수 있습니다.

- OpenTelemetry 프로젝트는 visualizer와 telemetry backend에서 지원되는 default aggregation을 제공하는 것을 목표로 합니다.

- Request tracing은 request lifecycle을 캡처하고 요청의 개별 부분에 context를 제공하는 데 목적이 있습니다.

- 반면 metric은 aggregate된 통계 정보를 제공하는 데 목적이 있습니다.

- Metric use case의 예시는 다음과 같습니다.

  - protocol type별로 서비스가 읽은 총 byte 수 report

  - 총 byte 수와 request당 byte 수 report

  - system call duration report

  - trend를 파악하기 위한 request size report

  - process의 CPU 또는 memory usage report

  - account의 average balance value report

  - 현재 처리 중인 active request report  

## **Views**

- View는 SDK 사용자가 SDK가 출력하는 metric을 customize할 수 있는 유연성을 제공합니다.

- 다음을 customize할 수 있습니다.

  - 어떤 metric instrument를 처리할지

  - 어떤 metric instrument를 무시할지

  - aggregation

  - metric에서 어떤 attribute를 report할지
