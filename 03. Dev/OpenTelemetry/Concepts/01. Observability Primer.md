# Observability Primer

## 1. What is Observability?

- Observability는 시스템 내부 구현을 몰라도 외부에서 시스템을 이해할 수 있게 해주는 개념이다.

- Observability를 통해 시스템에 대해 질문할 수 있으며, 다음 문제를 다룰 수 있다.

  - 새로운 문제를 쉽게 추적하고 처리

  - “unknown unknowns” 문제 대응

  - “왜 이런 일이 발생했는가?”에 대한 답 찾기


- 시스템에 이런 질문을 하려면 애플리케이션이 적절히 계측 되어야 한다.

- OpenTelemetry는 애플리케이션 코드를 계측하여 시스템을 observable하게 만드는 메커니즘이다.

<br>

## 2. Reliability and Metrics

- Telemetry는 시스템과 시스템 동작에서 emit되는 데이터를 의미한다.

  - traces

  - metrics

  - logs


- Reliability(신뢰성)는 다음 질문에 답한다.

  - 서비스가 사용자가 기대하는 일을 하고 있는가?

  - 시스템이 100% uptime을 유지하더라도 reliable하지 않을 수 있다.
    - 예를 들어, 사용자가 “Add to Cart”를 클릭해 검은색 신발을 장바구니에 담으려 했는데, 시스템이 항상 검은색 신발을 담지 못한다면 해당 시스템은 reliable하지 않을 수 있다.

- Metrics는 일정 기간 동안 수집된 numeric data의 aggregation(집합)이다.

  - system error rate

  - CPU utilization

  - 특정 service의 request rate


- "SLI(Service Level Indicator)"는 service behavior에 대한 measurement(측정)를 나타낸다.
  - 좋은 SLI는 사용자의 관점에서 서비스를 측정한다.
    - ex) web page가 load되는 속도


- "SLO(Service Level Objective)"는 reliability를 조직 또는 다른 팀에 전달하는 수단이다.
  - SLO는 하나 이상의 SLI를 business value에 연결해 reliability를 표현한다.

<br>

## 3. Understanding Distributed Tracing

- "distributed tracing"은 request가 복잡한 distributed system을 따라 propagation되는 과정을 관찰할 수 있게 해준다.

  - application 또는 system health에 대한 visibility 향상

  - local 환경에서 재현하기 어려운 behavior debugging


- distributed tracing은 distributed system에 필수적이다.

- distributed system은 일반적으로 다음 문제를 가진다.

  - nondeterministic problem

    : 같은 입력값이 주어져도 실행할 때마다 과정이나 결과가 달라질 수 있는 특성(불확실성)

  - local 환경에서 재현하기에는 너무 복잡한 문제


- Distributed tracing을 이해하려면 각 component의 역할을 알아야 한다.

  - logs

  - spans

  - traces


<br>

### 3.1 Logs

- "Log"는 service 또는 다른 component에서 emit되는 timestamped message이다.

- Trace와 달리 log는 반드시 특정 user request 또는 transaction과 연결되지 않는다.

- Log만으로는 code execution을 추적하기에 충분하지 않다.
  - 일반적으로 log에는 log가 어디에서 호출되었는지와 같은 contextual information이 부족하기 때문이다.

- Log는 다음 경우 훨씬 더 유용해진다.

  - span의 일부로 포함될 때

  - trace 및 span과 correlated 될 때


<br>

### 3.2 Spans

- "Span"은 single unit of work 또는 operation(작업)을 나타낸다.

- Span은 request가 수행하는 특정 operation을 추적한다.
  - 이를 통해 operation이 실행되는 동안 어떤 일이 발생했는지 보여준다.

- Span은 다음 데이터를 포함한다.

  - name

  - time-related data

  - structured log messages

  - Attributes로 표현되는 metadata


<br>

- Span attributes는 span에 attached 되는 metadata이다.

| Key                         | Value                                                        |
| --------------------------- | ------------------------------------------------------------ |

### 3.3 Distributed Traces

- "distributed trace"는 일반적으로 "trace"라고 부른다.

- "Trace"는 application 또는 end user가 만든 single request가 architecture의 여러 service를 통과하는 path를 기록한다.

- Trace는 하나 이상의 span으로 구성된다.
  - 	첫 번째 span은 root span을 나타낸다.
  - 	각 root span은 request의 시작부터 끝까지를 나타낸다.

- Parent 아래의 span들은 request 중 발생하는 일을 더 깊이 있는 context로 제공한다.

  - 즉, request를 구성하는 step들을 보여준다.

  - 예를 들어 사용자가 web page를 load하면 initial HTTP request는 다음을 통과할 수 있다.

    - API gateway

    - backend service

    - database


  - 각 step은 span으로 표현된다.

  - 이 span들이 함께 하나의 trace를 구성하며, request의 end-to-end journey를 보여준다.

- Tracing이 없으면 distributed system에서 performance problem의 root cause를 찾기 어렵다.
  - Tracing은 request가 distributed system을 통과하는 동안 발생하는 일을 breakdown한다.
  - 이를 통해 distributed system debugging과 이해를 덜 어렵게 만든다.

- 많은 Observability backend는 trace를 waterfall diagram으로 시각화한다.
  - Waterfall diagram은 root span과 child span 사이의 parent-child relationship을 보여준다.

- 하나의 span이 다른 span을 감싸는 경우, 이는 nested relationship도 나타낸다.
