# Profiles

- **profile**은 애플리케이션이 실행 중에 어디에서 resource를 소비하는지 보여주는 samples와 관련 metadata의 모음입니다.

- Sample은 특정 program context에서 관찰된 값을 기록합니다. 
  - 보통 stack trace가 여기에 해당하며, 상위 수준 request에 해당하는 trace ID 같은 보조 정보가 선택적으로 추가될 수 있습니다.

- Sample을 캡처하는 순간을 sample event라고 합니다. 

  - Sample event는 관측된 data point뿐만 아니라, 해당 data point가 캡처된 시간도 함께 포함합니다.

  - 예를 들어 On-CPU profile은 sample이 캡처된 시점에 CPU에서 실행 중이던 code에 대한 samples, 즉 집계된 stack traces를 포함합니다. 또한 각 stack trace가 관측된 timestamp와 관측 횟수도 함께 포함합니다.  

<br>

## Profiles overview

- Profiles는 logs, metrics, traces와 함께 observability의 네 번째 핵심 signal로 부상하고 있습니다.

- Profiles는 system과 application behavior에 대한 매우 깊은 insight를 제공하며, 다른 signal에서는 놓칠 수 있는 performance bottleneck을 찾아내는 경우가 많습니다.

- Profiles는 resource consumption과 code execution에 대한 세밀한 time-based view를 제공합니다. 여기에는 다음이 포함됩니다.

  - **Application-level profiling**
    - software function이 CPU, memory 및 기타 resource를 어떻게 소비하는지 보여줍니다.
    - 느리거나 비효율적인 code를 드러냅니다.

  - **System-level profiling**
    - infrastructure에 대한 전체적인 view를 제공합니다.
      - operating system call, kernel operation, I/O에서 발생하는 문제를 찾아냅니다.

- 이런 performance picture는 다음으로 이어질 수 있습니다.

  - **Faster Root Cause Analysis**

    - performance degradation의 정확한 원인을 빠르게 식별합니다.

    - **Proactive Optimization**
      - 사용자에게 영향이 가기 전에 잠재적 문제를 식별합니다.

    - **Improved Resource Utilization**
      - cost saving과 efficiency를 위해 infrastructure를 최적화합니다.

  - **Enhanced Developer Productivity**
    - 개발자가 code performance를 검증하고 regression을 방지할 수 있도록 돕습니다.

## How profiles complement other signals

- 각 OpenTelemetry signal은 서로 다른 질문에 답합니다.

| **Signal** | **Question**                                                 |
| ---------- | ------------------------------------------------------------ |
| Logs       | 어떤 개별 event가 발생했는가? system behavior에 대한 insight |
| Metrics    | system level에서 무슨 일이 발생하고 있는가? 예: CPU usage가 90% |
| Traces     | request가 distributed system을 어떻게 이동하는가?            |
| Profiles   | 어떤 code가 resource를 소비하고 있는가?                      |

- OpenTelemetry profiles는 다른 signal과의 bi-directional link를 지원합니다.

- 이 correlation은 두 가지 차원에서 동작합니다.

  - **Request context correlation**
    - profiling data를 특정 trace 또는 span에 연결합니다.
    - 특정 request 중 어떤 code가 실행되고 있었는지 볼 수 있습니다.

  - **Resource context correlation**
    - profiling data를 관련 metrics, logs, traces를 emit한 동일한 resource에 연결합니다.
      - 예를 들어 같은 service instance와 연결할 수 있습니다.

- Profiles는 다른 signal과 연관될 때 특히 강력해집니다.

  - **Logs to profiles**

    - out of memory log entry에서 memory pressure를 유발한 code path를 찾습니다.

    - **Metrics to profiles**
      - CPU 또는 memory usage spike에서 해당 resource를 소비한 function으로 바로 이동합니다.

  - **Traces to profiles**
    - trace 안의 느린 span에서 해당 profile을 확인해 latency를 유발한 code를 식별합니다.

## Profile types

- Profiling은 다양한 종류의 resource usage를 캡처할 수 있습니다.

- 일반적인 profile type은 다음과 같습니다.

  - **On-CPU**

    - 어떤 function이 processor time을 소비하고 있는가?

    - **Off-CPU**
      - thread가 실행되는 대신 어디에서 blocked 또는 waiting 상태인가?
      - 예: lock, I/O

    - **Heap (memory)**
      - 어떤 function이 아직 사용 중인 memory를 allocate했는가?

  - **Allocations (memory)**
    - 어떤 code path가 가장 많은 memory allocation을 발생시키는가?
      - 해당 memory가 해제되었는지 여부와는 무관합니다.

- OpenTelemetry profiles signal은 이 모든 유형을 수용할 만큼 유연합니다.

- 다만 실제로 사용할 수 있는 profile type은 사용하는 language runtime과 profiler에 따라 달라집니다.  

<br>

## **How profiling works**

- Profile을 수집하는 방법은 여러 가지가 있으며, OpenTelemetry profiles는 이 모든 방식을 지원하도록 설계되었습니다.

- **Sampling-based profiling**
  - profiler가 주기적으로 program을 interrupt합니다.
  - 예를 들어 timer-based interrupt를 사용합니다.
  - 그리고 현재 stack trace를 기록합니다.
  - CPU profiling에서 가장 일반적인 접근 방식입니다.
  - Linux에서는 profiler가 eBPF를 사용해 userspace application을 전혀 수정하지 않고 kernel에서 stack trace를 캡처할 수 있습니다.

- 이 접근 방식은 zero-instrumentation, whole-system profiling을 가능하게 합니다.
  - 여기에는 runtime support 없이 compiled language로 생성된 code도 포함됩니다.
  - 또한 continuous하고 low-overhead인 production 사용을 염두에 두고 설계되었습니다.

- **Instrumentation-based profiling**
  - runtime hook 또는 bytecode instrumentation이 memory allocation, lock acquisition, garbage collection 같은 event를 관련 stack trace와 함께 report합니다.

- 수집 방식과 관계없이, 결과 data는 OpenTelemetry의 common profile data model로 직렬화되고 OTLP를 통해 export됩니다.
