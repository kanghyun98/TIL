# 기존 아키텍처에서 FSD로의 마이그레이션

> https://feature-sliced.design/kr/docs/guides/migration/from-custom
> 

### 시**작 전 체크리스트 (**정말 필요한지 먼저 확인)

1. 신규 팀원이 프로젝트에 적응하기 어려워하는 경우
2. 코드 일부를 수정할 때, 관련 없는 다른 코드에 오류가 발생하는 경우가 **잦은** 경우
3. 새 기능을 추가할 때 고려해야 할 사항이 너무 많아 어려움을 겪는 경우

### **1단계: 페이지 단위로 코드 분리**

- `📁 pages` 폴더가 있다면 이 단계를 건너뛰어도 됨
- `📁 routes`만 있다면
    1. `📁 pages` 폴더를 새로 만듦
    2. `📁 routes`에 있던 **페이지용 컴포넌트**를 가능한 한 모두 `📁 pages` 폴더로 옮김
    3. 코드를 옮길 때마다 해당 페이지 전용 폴더를 만들고 그 안에 `index.tsx` 파일을 추가해 **진입점(entry point)** 를 노출

### **2단계: 페이지 외부 코드를 분리**

- **📁 src/shared 폴더를 만들고,** 📁 pages 또는 📁 routes를 import하지 않는 모든(파일)은 이 폴더로 모음
- **📁 src/app 폴더를 만들고,** 📁 pages 또는 📁 routes를 import하는 모듈과 라우트 정의 파일은 이 폴더에 배치

### **3단계: 페이지 간 cross-imports 해결**

- 한 페이지가 다른 페이지의 코드를 직접 import하고 있다면, 아래 두 가지 방식 중 하나로 의존성을 정리
    - **A. 코드 복사하여 독립시키기**
        - 페이지별로 로직이 달라질 가능성이 높거나, 재사용성이 낮은 경우
    - **B. Shared로 이동하여 공통화하기**
        - 여러 페이지에서 반복적으로 사용되는 경우

<aside>

경우에 따라서는 **중복을 허용하더라도 페이지 간 의존성을 줄이는 것**이 더 중요합니다.

다만, 비즈니스 로직처럼 변경 가능성이 큰 핵심 부분은 중복을 피하고, 복사할 때에도 가능한 한 DRY 원칙을 고려합니다.

</aside>

### **4단계: Shared Layer 정리하기**

- 한 페이지에서만 사용되는 코드는 해당 페이지의 slice로 이동
    - `actions, reducers, selectors` 역시 예외가 아니며, **사용되는 위치와 가까운 곳**에 두는 것이 가장 좋음
- Shared는 모든 layer가 의존할 수 있는 공통 의존 지점이기 때문에, 이곳에 코드를 과도하게 쌓아두지 않고 최소한으로 유지하는 것이 변경 위험을 줄이는 핵심 원칙

### **5단계: 기술적 목적별 segment 정리**

| **segment** | **용도 예시** |
| --- | --- |
| `ui` | Components, formatters, styles |
| `api` | Backend requests, DTOs, mappers |
| `model` | Store, schema, business logic |
| `lib` | Shared utilities / helpers |
| `config` | Configuration files, feature flags |

> **무엇인지**가 아니라 **무엇을 위해 존재하는지**를 기준으로 폴더를 구분
> 
> 
>  `components`, `utils`, `types`처럼 목적이 모호한 폴더 이름은 지양
> 
1. **각 페이지 내부**에서, 필요한 `segment(ui, model, api 등)`를 구성
2. Shared 폴더는 공통 기능만 남기도록 정리
    - `components/containers` → `shared/ui`
    - `helpers/utils` → `shared/lib` (기능별 그룹화 후)
    - `constants` → `shared/config`

⬇️ 여기서부터 선택 단계

### **6단계: 여러 페이지에서 재사용되는 Redux slice를 Entities / Features layer로 분리**

- 명확한 business entity 를 갖는 slice는 Entities layer로 이동하며, entity마다 별도의 폴더를 구성
- 사용자의 특정 행동(action) 을 중심으로 한 slice는 Features layer로 옮겨 독립적으로 관리
- Entities와 Features는 서로 의존하지 않고 사용할 수 있도록 설계

### **7단계: modules 폴더 리팩터링**

- `📁 modules`는 과거에 비즈니스 로직을 모아두던 공간으로, 성격상 **Features layer**와 비슷

### **8단계: shared/ui에 presentational UI 기반 마련**

- `shared/ui`에는 비즈니스 로직이 전혀 없는, 재사용 가능한 presentational UI 컴포넌트만
- 기존 `📁 components / 📁 containers`에 있던 컴포넌트에서 비즈니스 로직을 분리해 상위 layer로 이동
    - 여러 곳에서 쓰이지 않는 부분은 **복사(paste)** 해서 각 layer에서 독립적으로 관리해도 문제 없음

## ▶️ V2

> https://github.com/feature-sliced/documentation
> 

### **v2 도입 배경**

- v1에는 다음과 같은 한계가 존재
    - 과도한 **boilerplate** 발생
    - 추상화 규칙이 모호해 **코드베이스 복잡도** 상승
    - 암묵적 설계로 **확장/온보딩 어려움**

### **v2 마이그레이션 이유**

- **직관적 구조 제공**
    - v2는 **layer → slice → segment** 라는 세 가지 개념만 이해하면 구조적 결정을 쉽게 내릴 수 있음
- **유연한 모듈화**
    - 독립 영역은 slice 단위로, 전역 흐름은 Processes layer로 분리해 확장성을 확보
    - 새로운 module을 추가할 때 *(layer → slice → segment)* 규칙만 따르면 폴더 재배치나 리팩터링 부담이 크게 줄어듦

### **주요 변경 사항**

**Layer 구조 명확화**

- `/app` > `/processes` > **`/pages`** > **`/features`** > `/entities` > `/shared`
- 상위 layer는 더 넓은 context를 제공하며, 하위 layer는 더 낮은 변경 리스크와 높은 재사용성을 가짐

**Shared 통합**

- `src` 루트에 흩어져 있던 UI, lib, API 인프라 추상화를 `/src/shared`로 통합
    - `shared/ui` - 공통 UI components (선택 사항)
    - `shared/lib` - 재사용 가능한 helper libraries
    - `shared/api` - API entry points
        - *각 feature/page 내 local 정의 가능하지만, 전역 entry point 집중을 권장*
    - `shared` 폴더에는 business logic 의존을 두지 않음

**Entities / Processes Layer 추가**

- 로직 복잡성과 높은 결합을 줄이기 위한 **새로운 추상화**가 추가
- `/entities` : 프론트엔드에서 사용되는 business entities(예: `user`, `order`, `i18n`, `blog`)를 담당하는 layer
- `/processes` : 애플리케이션 전반에 걸친 비즈니스 process(예: `payment`, `auth`, `quick-tour`)를 캡슐화하는 선택적 layer

## ▶️ v2.1

### **v2.0 접근 방식**

- v2.0에서는 애플리케이션을 Entity와 Feature 단위로 세분화하여 구성
- 화면을 이루는 가장 작은 단위(entity 표현, 상호작용 요소 등)를 잘게 나눈 뒤, 이를 Widget으로 조합하고, 최종적으로 Page 를 구성하는 방식
- ➡️ 비즈니스 로직이 대부분 entity/feature layer에 과도하게 집중되었고, Page는 단순한 조합 계층으로 남아 고유한 책임이 약해지는 문제

### **v2.1 접근 방식**

- **Pages-First** 사고방식을 도입
- **Entity 간 cross-import**를 위한 `@x` 표기법이 **표준화**

**핵심 원칙**

- Page 내부에 주요 UI와 비즈니스 로직을 배치
- Shared layer에는 순수 재사용 요소만 유지
- 여러 Page에서 실제로 공유되는 로직만 Feature/Entity로 분리

**장점**

- **Page가 명확한 책임 단위**가 되어, 코드의 역할이 분명
- **Shared** layer가 불필요하게 비대해지는 것을 방지해 의존성이 간결
- 공통 로직을 실제로 재사용할 때만 분리하므로 과도한 추상화가 줄어듦

### **마이그레이션 프로세스**

**1. Slice 병합**

- v2.1 Page-First 모델에서는 **여러 Page에서 재사용되지 않는** slice는 Page 내부로 병합하는 것을 권장

> 프로젝트 루트에서 [Steiger](https://github.com/feature-sliced/steiger)를 실행하면 v2.1 mental model에 맞추어 slice 사용 여부를 자동으로 분석
> 

**2. Cross Import 표준화**

- v2.1에서는 Entity 간 cross-import를 위해 `@x-` 표기법을 사용

```jsx
import type { EntityA } from "entities/A/@x/B";
```
