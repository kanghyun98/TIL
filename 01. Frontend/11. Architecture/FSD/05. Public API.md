# Public API

> https://feature-sliced.design/kr/docs/reference/public-api
> 

- Public API는 **Slice 기능을 외부에서 사용할 수 있는 공식 경로**
- 외부 코드는 반드시 이 경로를 통해서만 Slice 내부의 특정 객체에 접근할 수 있음
- **Slice와 외부 코드 간의 계약(Contract)** 이자 **접근 게이트(Gate)** 역할
- 일반적으로 Public API는 **Re-export를 모아둔 `index` 파일**로 만듦

```jsx
// pages/auth/index.js

export { LoginPage } from "./ui/LoginPage";
export { RegisterPage } from "./ui/RegisterPage";
```

## ▶️ **좋은 Public API의 조건**

- **내부 구조 변경에 영향 없음**
    - Slice 내부 폴더 구조를 바꾸더라도, 외부 코드는 그대로 동작해야 합니다.
- **주요 동작 변경 = API 변경**
    - Slice의 동작이 크게 바뀌어 기존 기대가 깨진다면, Public API도 함께 변경되어야 합니다.
- **필요한 부분만 노출**
    - Slice 전체 구현을 공개하는 것이 아니라, 외부에서 꼭 필요한 기능만 선별해서 노출합니다.

**안 좋은 예: 무분별한 Wildcard Re-export**

```jsx
// ❌
export * from "./ui/Comment"; // 👎 무분별한 UI export
export * from "./model/comments"; // 💩 내부 모델 노출
```

- **발견 가능성 저하**
    - Public API에서 어떤 기능을 제공하는지 한눈에 파악하기 어렵습니다.
- **내부 구현 노출**
    - 원래 외부에서 알 필요가 없는 내부 코드를 외부에서 직접 사용하게 되고,이 코드에 대한 의존성이 생기면 리팩터링이 매우 어려워집니다.

## ▶️ **Cross-Import를 위한 Public API**

- **Cross-import:** 같은 Layer 안에서 한 Slice가 다른 Slice를 import하는 것을 말합니다.
- [Layer Import Rule](https://feature-sliced.design/kr/docs/reference/layers#import-rule-on-layers)에 따라 원칙적으로는 금지되지만, **ntity 간 참조**처럼 현실적으로 불가피한 경우가 있음
- 이럴 때는 `@x` 표기를 사용해 교차 참조 전용 Public API를 명시적으로 만듦

```jsx
- 📂 entities
    - 📂 artist
        - 📂 @x
            - song.ts — entities/song 전용 Public API
        - index.ts — 일반 Public API
```

```jsx
// entities/song에서는 다음과 같이 import

import type { Artist } from "entities/artist/@x/song";
```

## ▶️ **Index File 사용 시 주의사항**

### ✅ **Circular Import (순환 참조)**

- 이 구조는 번들러가 처리하기 어렵고, 디버그하기도 힘든 런타임 오류를 만들 수 있음
- 순환 참조는 Index 파일이 없어도 발생할 수 있지만, Index 파일은 특히 이런 실수를 만들기 쉬운 환경을 제공

**예방 원칙**

- **같은 Slice 내부**에서 가져올 때는
    - 상대 경로(`../api/loadUserStatistics`)를 사용해서 어느 파일을 참조하는지 명확히 작성
- **다른 Slice**에서 가져올 때는
    - 절대 경로(예: `@/features/...`)나 Alias를 사용
- Index에서 export한 모듈이 다시 Index를 참조하지 않도록 주의

### ✅ **Large Bundle & Tree-shaking 문제**

- 일부 Bundler는 Index 파일에서 여러 모듈을 한 번에 export할 경우, 실제로 사용하지 않는 코드(Dead code)를 제대로 제거(Tree-shaking)하지 못할 수 있음
- 대부분의 Public API에서는 모듈 간 연관성이 높아 크게 문제되지 않지만, `shared/ui`, `shared/lib`처럼 **서로 관련성이 낮은 모듈 묶음**에서는 문제가 커짐

**해결 방법**

- 각 컴포넌트/라이브러리별로 **별도의 작은 Index 파일**을 만듦
- 그리고 사용하는 쪽에서는 `@/shared/ui/button`, `@/shared/ui/text-field`와 같이 **컴포넌트 단위로 직접 import**

```jsx
import { Button } from "@/shared/ui/button";
import { TextField } from "@/shared/ui/text-field";
```

→ 이렇게 하면 Bundler가 사용되지 않는 컴포넌트 코드를 제거할 수 있어 Tree-shaking이 더 잘 동작

### ✅ **Public API 우회 방지의 한계**

- Slice에 Index 파일을 만들어도, 개발자가 직접 내부 경로를 입력해 import하는 것을 완전히 막을 수는 없음

**해결 방법**

- [Steiger](https://github.com/feature-sliced/steiger) 같은 **FSD 전용 아키텍처 린터**를 사용해 프로젝트의 import 경로를 검사하고, 규칙을 강제

### ✅ **대규모 프로젝트에서의 Bundler 성능 문제**

- Index 파일(일명 **barrel 파일**)이 너무 많아지면 개발 서버 실행 속도나 HMR(Hot Module Replacement) 성능이 저하될 수 있음
    
    > https://tkdodo.eu/blog/please-stop-using-barrel-files
    > 

**최적화 방법**

- [Large Bundle & Tree-shaking 문제](https://feature-sliced.design/kr/docs/reference/public-api#large-bundles)에서 설명한 것처럼, `shared/ui`, `shared/lib`에 있는 큰 Index를 없애고 컴포넌트/모듈 단위로 쪼갠 작은 Index를 사용
- Segment 단위로 불필요한 Index 파일을 만들지 않음
- 큰 프로젝트는 **기능 단위 Chunk 또는 패키지**로 나눔
