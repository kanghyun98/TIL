# Queries

<br>

## 1. Query Basics

쿼리는 고유한 키에 연결된 비동기 데이터 소스에 대한 선언적 의존성입니다.

쿼리는 서버에서 데이터를 가져오기 위해 모든 Promise 기반 메서드와 함께 사용할 수 있습니다. 여기에는 `GET`, `POST` 메서드도 포함됩니다. 서버의 데이터를 수정하는 메서드라면 `Mutations`를 사용하는 것을 권장합니다.

컴포넌트나 커스텀 훅에서 쿼리를 구독하려면 `useQuery` 훅을 최소한 다음과 함께 호출합니다.

- 쿼리를 위한 고유한 키
- 다음 중 하나를 수행하는 Promise를 반환하는 함수
  - 데이터를 resolve 한다
  - 에러를 throw 한다

```tsx
import { useQuery } from '@tanstack/react-query'

function App() {
  const info = useQuery({ queryKey: ['todos'], queryFn: fetchTodoList })
}
```

<br>

제공한 고유 키는 애플리케이션 전반에서 쿼리를 다시 가져오고, 캐싱하고, 공유하는 데 내부적으로 사용됩니다.

`useQuery`가 반환하는 쿼리 결과에는 템플릿 구성과 데이터 활용에 필요한 정보가 모두 들어 있습니다.

```tsx
const result = useQuery({ queryKey: ['todos'], queryFn: fetchTodoList })
```



결과 객체에는 알아두어야 할 중요한 상태가 있습니다. 쿼리는 어느 시점이든 다음 상태 중 하나만 가질 수 있습니다.

- `isPending` 또는 `status === 'pending'` - 쿼리에 아직 데이터가 없습니다
- `isError` 또는 `status === 'error'` - 쿼리에서 에러가 발생했습니다
- `isSuccess` 또는 `status === 'success'` - 쿼리가 성공했고 데이터를 사용할 수 있습니다



이 기본 상태 외에도 쿼리 상태에 따라 다음 정보가 제공됩니다.

- `error` - 쿼리가 `isError` 상태라면 `error` 프로퍼티로 에러를 사용할 수 있습니다
- `data` - 쿼리가 `isSuccess` 상태라면 `data` 프로퍼티로 데이터를 사용할 수 있습니다
- `isFetching` - 어떤 상태에서든 쿼리가 가져오는 중이라면, 배경 리패칭을 포함해 `isFetching`은 `true`가 됩니다



대부분의 쿼리에서는 보통 `isPending`을 확인하고, 그다음 `isError`를 확인한 뒤, 마지막으로 데이터를 사용할 수 있다고 가정하고 성공 상태를 렌더링하면 충분합니다.

```tsx
function Todos() {
  const { isPending, isError, data, error } = useQuery({
    queryKey: ['todos'],
    queryFn: fetchTodoList,
  })

  if (isPending) {
    return <span>Loading...</span>
  }

  if (isError) {
    return <span>Error: {error.message}</span>
  }

  // We can assume by this point that `isSuccess === true`
  return (
    <ul>
      {data.map((todo) => (
        <li key={todo.id}>{todo.title}</li>
      ))}
    </ul>
  )
}
```



불리언보다 상태값이 더 편하다면 `status` 상태를 사용할 수도 있습니다.

```tsx
function Todos() {
  const { status, data, error } = useQuery({
    queryKey: ['todos'],
    queryFn: fetchTodoList,
  })

  if (status === 'pending') {
    return <span>Loading...</span>
  }

  if (status === 'error') {
    return <span>Error: {error.message}</span>
  }

  // also status === 'success', but "else" logic works, too
  return (
    <ul>
      {data.map((todo) => (
        <li key={todo.id}>{todo.title}</li>
      ))}
    </ul>
  )
}
```

`TypeScript`는 `pending`과 `error`를 확인한 뒤 `data`에 접근하면 데이터 타입도 올바르게 좁혀 줍니다.

<br>

### 1.1 FetchStatus

`status` 필드 외에도 `fetchStatus` 프로퍼티가 추가로 제공되며, 다음 옵션을 가집니다.

- `fetchStatus === 'fetching'` - 쿼리가 현재 데이터를 가져오는 중입니다
- `fetchStatus === 'paused'` - 쿼리가 데이터를 가져오려 했지만 일시 중지된 상태입니다
- `fetchStatus === 'idle'` - 쿼리가 현재 아무 작업도 하지 않고 있습니다

<br>

### 1.2 Why two different states?

배경 리패치와 stale-while-revalidate 로직 때문에 `status`와 `fetchStatus`의 모든 조합이 가능해집니다. 예를 들면 다음과 같습니다.

- `success` 상태의 쿼리는 보통 `idle` `fetchStatus`에 있지만, 배경 리패치가 진행 중이라면 `fetching`일 수도 있습니다
- 마운트되었고 데이터가 없는 쿼리는 보통 `pending` 상태와 `fetching` `fetchStatus`를 가지지만, 네트워크 연결이 없다면 `paused`일 수도 있습니다

따라서 쿼리는 실제로 데이터를 가져오고 있지 않더라도 `pending` 상태일 수 있습니다. 경험적으로는 다음처럼 이해할 수 있습니다.

- `status`는 데이터에 대한 정보를 제공합니다. 데이터가 있는가, 없는가?
- `fetchStatus`는 `queryFn`에 대한 정보를 제공합니다. 실행 중인가, 아닌가?
