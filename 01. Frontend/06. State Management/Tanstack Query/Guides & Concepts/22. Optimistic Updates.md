# Optimistic Updates

React Query는 mutation이 완료되기 전에 UI를 낙관적으로 업데이트하는 두 가지 방법을 제공합니다.

- `onMutate` 옵션으로 캐시를 직접 업데이트할 수 있습니다.
- `useMutation` 결과에서 반환되는 `variables`를 활용해 UI를 업데이트할 수 있습니다.

<br>

## 1. Via the UI

이 방식은 캐시를 직접 건드리지 않기 때문에 더 단순합니다.

```tsx
const addTodoMutation = useMutation({
  mutationFn: (newTodo: string) => axios.post('/api/data', { text: newTodo }),
  // query invalidation에서 Promise를 반드시 반환해야 합니다.
  // 그래야 refetch가 끝날 때까지 mutation이 `pending` 상태를 유지합니다.
  onSettled: () => queryClient.invalidateQueries({ queryKey: ['todos'] }),
})

const { isPending, submittedAt, variables, mutate, isError } = addTodoMutation
```

이제 `addTodoMutation.variables`로 추가한 todo에 접근할 수 있습니다.

쿼리를 렌더링하는 UI 목록에서 mutation이 `isPending`인 동안 임시 항목을 하나 더 렌더링할 수 있습니다.

```tsx
<ul>
  {todoQuery.items.map((todo) => (
    <li key={todo.id}>{todo.text}</li>
  ))}
  {isPending && <li style={{ opacity: 0.5 }}>{variables}</li>}
</ul>
```

- mutation이 `pending`인 동안, 다른 opacity를 가진 임시 항목을 렌더링합니다.
- mutation이 완료되면 이 항목은 자동으로 사라집니다.
- refetch가 성공했다면, 같은 항목이 이제 일반 항목으로 목록에 표시됩니다.

mutation이 실패하면 이 항목도 함께 사라집니다.

하지만 `isError` 상태를 확인해서 계속 보여줄 수도 있습니다. `variables`는 mutation 에러가 발생해도 지워지지 않기 때문에, 여전히 접근할 수 있고 재시도 버튼도 보여줄 수 있습니다.

```tsx
{
  isError && (
    <li style={{ color: 'red' }}>
      {variables}
      <button onClick={() => mutate(variables)}>Retry</button>
    </li>
  )
}
```

<br>

### 1.1 If the mutation and the query don't live in the same component

이 방식은 mutation과 query가 같은 컴포넌트에 있을 때 잘 동작합니다.

하지만 전용 `useMutationState` 훅을 사용하면 다른 컴포넌트에 있는 mutation에도 접근할 수 있습니다. 이 경우 `mutationKey`와 함께 사용하는 것이 가장 좋습니다.

```tsx
// 앱의 어딘가
const { mutate } = useMutation({
  mutationFn: (newTodo: string) => axios.post('/api/data', { text: newTodo }),
  onSettled: () => queryClient.invalidateQueries({ queryKey: ['todos'] }),
  mutationKey: ['addTodo'],
})

// 다른 곳에서 variables 접근
const variables = useMutationState<string>({
  filters: { mutationKey: ['addTodo'], status: 'pending' },
  select: (mutation) => mutation.state.variables,
})
```

`variables`는 배열이 됩니다.

- 동시에 여러 mutation이 실행될 수 있기 때문입니다.
- 항목에 고유 key가 필요하다면 `mutation.state.submittedAt`도 선택할 수 있습니다.
- 이렇게 하면 동시 optimistic update도 쉽게 표시할 수 있습니다.

<br>

## 2. Via the cache

mutation을 실행하기 전에 상태를 낙관적으로 업데이트하면 mutation이 실패할 가능성이 있습니다.

대부분의 실패 상황에서는 optimistic query를 refetch해서 실제 서버 상태로 되돌릴 수 있습니다. 하지만 어떤 경우에는 refetch가 제대로 동작하지 않거나, mutation 에러가 서버 문제를 의미해서 refetch 자체가 불가능할 수도 있습니다.

이럴 때는 업데이트를 rollback하도록 선택할 수 있습니다.

이를 위해 `useMutation`의 `onMutate` 핸들러는 값을 반환할 수 있고, 이 값은 나중에 `onError`와 `onSettled` 핸들러의 마지막 인자로 전달됩니다.

대부분의 경우 rollback 함수나 rollback에 필요한 값을 전달하는 것이 가장 유용합니다.

<br>

### 2.1 Updating a list of todos when adding a new todo

```tsx
const queryClient = useQueryClient()

useMutation({
  mutationFn: updateTodo,
  // mutate가 호출되면:
  onMutate: async (newTodo, context) => {
    // 진행 중인 refetch를 취소합니다.
    // 그래야 optimistic update를 덮어쓰지 않습니다.
    await context.client.cancelQueries({ queryKey: ['todos'] })

    // 이전 값을 스냅샷으로 저장합니다.
    const previousTodos = context.client.getQueryData(['todos'])

    // 새 값으로 낙관적 업데이트를 수행합니다.
    context.client.setQueryData(['todos'], (old) => [...old, newTodo])

    // 스냅샷 값을 포함한 결과를 반환합니다.
    return { previousTodos }
  },
  // mutation이 실패하면
  // onMutate에서 반환한 결과로 rollback합니다.
  onError: (err, newTodo, onMutateResult, context) => {
    context.client.setQueryData(['todos'], onMutateResult.previousTodos)
  },
  // 성공 여부와 관계없이 항상 refetch합니다.
  onSettled: (data, error, variables, onMutateResult, context) =>
    context.client.invalidateQueries({ queryKey: ['todos'] }),
})
```

### 2.2 Updating a single todo

```tsx
useMutation({
  mutationFn: updateTodo,
  // mutate가 호출되면:
  onMutate: async (newTodo, context) => {
    // 진행 중인 refetch를 취소합니다.
    // 그래야 optimistic update를 덮어쓰지 않습니다.
    await context.client.cancelQueries({ queryKey: ['todos', newTodo.id] })

    // 이전 값을 스냅샷으로 저장합니다.
    const previousTodo = context.client.getQueryData(['todos', newTodo.id])

    // 새 값으로 낙관적 업데이트를 수행합니다.
    context.client.setQueryData(['todos', newTodo.id], newTodo)

    // 이전 todo와 새 todo를 함께 반환합니다.
    return { previousTodo, newTodo }
  },
  // mutation이 실패하면 위에서 반환한 결과를 사용합니다.
  onError: (err, newTodo, onMutateResult, context) => {
    context.client.setQueryData(
      ['todos', onMutateResult.newTodo.id],
      onMutateResult.previousTodo,
    )
  },
  // 성공 여부와 관계없이 항상 refetch합니다.
  onSettled: (newTodo, error, variables, onMutateResult, context) =>
    context.client.invalidateQueries({ queryKey: ['todos', newTodo.id] }),
})
```

원한다면 `onError`와 `onSuccess`를 따로 두는 대신 `onSettled`에서 처리할 수도 있습니다.

```tsx
useMutation({
  mutationFn: updateTodo,
  // ...
  onSettled: async (newTodo, error, variables, onMutateResult, context) => {
    if (error) {
      // do something
    }
  },
})
```

## 3. When to use what

optimistic result를 보여줘야 하는 위치가 하나뿐이라면, `variables`를 사용해 UI를 직접 업데이트하는 방식이 더 적은 코드로 구현할 수 있고 이해하기도 쉽습니다. 예를 들어 rollback을 전혀 처리하지 않아도 됩니다.

반면, 화면의 여러 위치가 이 업데이트를 함께 알아야 한다면 캐시를 직접 조작하는 방식이 이를 자동으로 처리해줍니다.
