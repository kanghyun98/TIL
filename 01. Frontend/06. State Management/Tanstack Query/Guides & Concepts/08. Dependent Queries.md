# Dependent Queries

<br>

## 1. useQuery dependent Query

Dependent queries 또는 serial queries는 이전 query가 끝난 뒤 실행됩니다.

이를 위해 `enabled` 옵션으로 query가 실행 가능한 시점을 지정합니다.

```tsx
// Get the user
const { data: user } = useQuery({
  queryKey: ['user', email],
  queryFn: getUserByEmail,
})

const userId = user?.id

// Then get the user's projects
const {
  status,
  fetchStatus,
  data: projects,
} = useQuery({
  queryKey: ['projects', userId],
  queryFn: getProjectsByUser,
  // The query will not execute until the userId exists
  enabled: !!userId,
})
```

`projects` query는 처음에 아래 상태로 시작합니다.

```tsx
status: 'pending'
isPending: true
fetchStatus: 'idle'
```

`user`가 준비되면 `projects` query가 활성화되고, 아래 상태로 전환됩니다.

```tsx
status: 'pending'
isPending: true
fetchStatus: 'fetching'
```

`projects`를 가져오면 아래 상태가 됩니다.

```tsx
status: 'success'
isPending: false
fetchStatus: 'idle'
```

## 2. useQueries dependent Query

Dynamic parallel query인 `useQueries`도 이전 query에 의존할 수 있습니다.

```tsx
// Get the users ids
const { data: userIds } = useQuery({
  queryKey: ['users'],
  queryFn: getUsersData,
  select: (users) => users.map((user) => user.id),
})

// Then get the users messages
const usersMessages = useQueries({
  queries: userIds
    ? userIds.map((id) => {
        return {
          queryKey: ['messages', id],
          queryFn: () => getMessagesByUsers(id),
        }
      })
    : [], // if userIds is undefined, an empty array will be returned
})
```

`useQueries`는 query result 배열을 반환합니다.

<br>

## 3. A note about performance

Dependent queries는 정의상 request waterfall 형태이므로 성능에 불리합니다.

- 두 query가 같은 시간만큼 걸린다고 가정하면
  - 병렬 실행보다 직렬 실행이 항상 2배의 시간이 걸립니다
- 이 문제는 client의 latency가 높을수록 더 크게 드러납니다

가능하다면 backend API를 재구성해 두 query를 병렬로 가져오는 편이 좋습니다.

- 다만, 항상 현실적으로 가능한 것은 아닙니다
- 위 예시에서는 먼저 `getUserByEmail`을 호출한 뒤 `getProjectsByUser`를 호출하는 대신
  - `getProjectsByUserEmail` query를 새로 두어 waterfall을 평탄화할 수 있습니다
