# Caching Examples

<br>

## 1. Basic Example

이 캐싱 예시는 다음 항목의 흐름과 생명주기를 설명합니다.

- 캐시 데이터가 있는 Query Instance와 없는 Query Instance
- Background Refetching
- Inactive Queries
- Garbage Collection

<br>

기본값은 다음과 같다고 가정합니다.

- `gcTime`: 5분
- `staleTime`: `0`

<br>

새로운 `useQuery({ queryKey: ['todos'], queryFn: fetchTodos })` 인스턴스가 마운트됩니다.

- 아직 `['todos']` query key로 만들어진 쿼리가 없습니다.
- 이 쿼리는 hard loading state를 보여줍니다.
- 데이터를 가져오기 위해 네트워크 요청을 실행합니다.
- 네트워크 요청이 완료되면, 반환된 데이터는 `['todos']` key 아래에 캐시됩니다.
- 데이터는 설정된 `staleTime` 이후 stale 상태로 표시됩니다.
  - 기본값은 `0`입니다.
  - 즉, 즉시 stale 상태가 됩니다.

<br>

다른 위치에서 두 번째 `useQuery({ queryKey: ['todos'], queryFn: fetchTodos })` 인스턴스가 마운트됩니다.

- 첫 번째 쿼리로 인해 `['todos']` key에 대한 캐시 데이터가 이미 존재합니다.
- 해당 데이터가 즉시 캐시에서 반환됩니다.
- 새 인스턴스는 자신의 query function으로 새로운 네트워크 요청을 트리거합니다.
  - 두 `fetchTodos` query function이 동일한지 여부와 관계없이, 두 쿼리의 `status`가 함께 업데이트됩니다.
  - `isFetching`, `isPending` 및 관련 값도 함께 업데이트됩니다.
  - 이유는 두 쿼리가 같은 query key를 사용하기 때문입니다.
- 요청이 성공적으로 완료되면, `['todos']` key 아래의 캐시 데이터가 새 데이터로 업데이트됩니다.
- 두 인스턴스 모두 새 데이터로 업데이트됩니다.

<br>

두 `useQuery({ queryKey: ['todos'], queryFn: fetchTodos })` 인스턴스가 언마운트되고 더 이상 사용되지 않습니다.

- 이 쿼리의 active instance가 더 이상 없습니다.
- `gcTime`을 기준으로 garbage collection timeout이 설정됩니다.
- 이 timeout은 쿼리를 삭제하고 garbage collect하기 위한 것입니다.
- 기본값은 5분입니다.

<br>

캐시 timeout인 `gcTime`이 완료되기 전에, 또 다른 `useQuery({ queryKey: ['todos'], queryFn: fetchTodos })` 인스턴스가 마운트됩니다.

- 쿼리는 사용 가능한 캐시 데이터를 즉시 반환합니다.
- 동시에 `fetchTodos` function이 백그라운드에서 실행됩니다.
- 요청이 성공적으로 완료되면, 캐시는 fresh data로 채워집니다.

<br>

마지막 `useQuery({ queryKey: ['todos'], queryFn: fetchTodos })` 인스턴스가 언마운트됩니다.

<br>

5분 안에 `useQuery({ queryKey: ['todos'], queryFn: fetchTodos })` 인스턴스가 더 이상 나타나지 않습니다.

- `['todos']` key 아래의 캐시 데이터가 삭제됩니다.
- 해당 데이터는 garbage collect됩니다.
