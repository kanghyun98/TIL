# Advanced Server Rendering

Advanced Server Rendering 가이드는 React Query를 streaming, Server Components, Next.js app router와 함께 사용하는 방법을 다룹니다.

이 가이드를 보기 전에 Server Rendering & Hydration 가이드를 먼저 읽는 것이 좋습니다. SSR에서 React Query를 사용하는 기본 개념을 다룹니다.

Performance & Request Waterfalls, Prefetching & Router Integration 가이드도 유용한 배경 지식을 제공합니다.

SSR 가이드의 `initialData` 접근 방식도 Server Components와 함께 사용할 수 있습니다. 하지만 이 가이드에서는 hydration API에 집중합니다.

<br>

## 1. Server Components & Next.js app router

Server Components를 깊게 다루지는 않습니다.

짧게 정리하면 Server Components는 항상 서버에서만 실행되는 컴포넌트입니다.

- 초기 페이지 뷰에서도 서버에서 실행됩니다.
- 페이지 전환에서도 서버에서 실행됩니다.
- Next.js의 `getServerSideProps`, `getStaticProps`, Remix의 `loader`와 비슷합니다.
- 다만 `getServerSideProps`, `getStaticProps`, `loader`는 데이터만 반환할 수 있습니다.
- Server Components는 그보다 더 많은 일을 할 수 있습니다.

React Query에서는 데이터가 핵심이므로, Server Components를 “또 다른 framework loader”로 생각하는 것이 좋습니다.

<br>

### 1.1 A quick note on terminology

지금까지 가이드에서는 server와 client를 기준으로 설명했습니다.

하지만 이 구분은 Server Components와 Client Components에 1:1로 대응되지 않습니다.

- Server Components
  - 항상 서버에서만 실행됩니다.

- Client Components
  - 브라우저에서만 실행되는 것은 아닙니다.
  - 초기 서버 렌더링 단계에서도 실행될 수 있습니다.

이를 다음처럼 생각할 수 있습니다.

- Server Components는 “loader phase”에서 실행됩니다.
  - 이 단계는 항상 서버에서 발생합니다.

- Client Components는 “application phase”에서 실행됩니다.
  - 이 application은 SSR 중 서버에서 실행될 수도 있습니다.
  - 브라우저에서 실행될 수도 있습니다.
  - 실제 실행 위치와 SSR 여부는 framework마다 다를 수 있습니다.

<br>

### 1.2 Initial setup

React Query 설정의 첫 단계는 항상 같습니다.

- `queryClient`를 생성합니다.
- 애플리케이션을 `QueryClientProvider`로 감쌉니다.

Server Components를 사용할 때도 대부분 동일합니다. 차이는 framework의 파일 이름 규칙입니다.

```tsx
// In Next.js, this file would be called: app/providers.tsx
'use client'

// Since QueryClientProvider relies on useContext under the hood, we have to put 'use client' on top
import {
  isServer,
  QueryClient,
  QueryClientProvider,
} from '@tanstack/react-query'

function makeQueryClient() {
  return new QueryClient({
    defaultOptions: {
      queries: {
        // With SSR, we usually want to set some default staleTime
        // above 0 to avoid refetching immediately on the client
        staleTime: 60 * 1000,
      },
    },
  })
}

let browserQueryClient: QueryClient | undefined = undefined

function getQueryClient() {
  if (isServer) {
    // Server: always make a new query client
    return makeQueryClient()
  } else {
    // Browser: make a new query client if we don't already have one
    // This is very important, so we don't re-make a new client if React
    // suspends during the initial render. This may not be needed if we
    // have a suspense boundary BELOW the creation of the query client
    if (!browserQueryClient) browserQueryClient = makeQueryClient()
    return browserQueryClient
  }
}

export default function Providers({ children }: { children: React.ReactNode }) {
  // NOTE: Avoid useState when initializing the query client if you don't
  //       have a suspense boundary between this and the code that may
  //       suspend because React will throw away the client on the initial
  //       render if it suspends and there is no boundary
  const queryClient = getQueryClient()

  return (
    <QueryClientProvider client={queryClient}>{children}</QueryClientProvider>
  )
}
```

```tsx
// In Next.js, this file would be called: app/layout.tsx
import Providers from './providers'

export default function RootLayout({
  children,
}: {
  children: React.ReactNode
}) {
  return (
    <html lang="en">
      <head />
      <body>
        <Providers>{children}</Providers>
      </body>
    </html>
  )
}
```

이 부분은 SSR 가이드와 거의 비슷합니다.

다만 Server Components에서는 두 개의 파일로 나누어야 합니다.

<br>

### 1.3 Prefetching and de/hydrating data

먼저 데이터를 prefetch한 뒤, dehydrate하고 hydrate하는 방법을 살펴봅니다.

Next.js Pages Router에서는 다음과 같은 형태였습니다.

```tsx
// pages/posts.tsx
import {
  dehydrate,
  HydrationBoundary,
  QueryClient,
  useQuery,
} from '@tanstack/react-query'

// This could also be getServerSideProps
export async function getStaticProps() {
  const queryClient = new QueryClient()

  await queryClient.prefetchQuery({
    queryKey: ['posts'],
    queryFn: getPosts,
  })

  return {
    props: {
      dehydratedState: dehydrate(queryClient),
    },
  }
}

function Posts() {
  // This useQuery could just as well happen in some deeper child to
  // the <PostsRoute>, data will be available immediately either way
  //
  // Note that we are using useQuery here instead of useSuspenseQuery.
  // Because this data has already been prefetched, there is no need to
  // ever suspend in the component itself. If we forget or remove the
  // prefetch, this will instead fetch the data on the client, while
  // using useSuspenseQuery would have had worse side effects.
  const { data } = useQuery({ queryKey: ['posts'], queryFn: getPosts })

  // This query was not prefetched on the server and will not start
  // fetching until on the client, both patterns are fine to mix
  const { data: commentsData } = useQuery({
    queryKey: ['posts-comments'],
    queryFn: getComments,
  })

  // ...
}

export default function PostsRoute({ dehydratedState }) {
  return (
    <HydrationBoundary state={dehydratedState}>
      <Posts />
    </HydrationBoundary>
  )
}
```

App Router로 변환해도 구조는 비슷합니다.

먼저 prefetch를 수행하는 Server Component를 만듭니다.

```tsx
// app/posts/page.tsx
import {
  dehydrate,
  HydrationBoundary,
  QueryClient,
} from '@tanstack/react-query'
import Posts from './posts'

export default async function PostsPage() {
  const queryClient = new QueryClient()

  await queryClient.prefetchQuery({
    queryKey: ['posts'],
    queryFn: getPosts,
  })

  return (
    // Neat! Serialization is now as easy as passing props.
    // HydrationBoundary is a Client Component, so hydration will happen there.
    <HydrationBoundary state={dehydrate(queryClient)}>
      <Posts />
    </HydrationBoundary>
  )
}
```

다음은 Client Component입니다.

```tsx
// app/posts/posts.tsx
'use client'

export default function Posts() {
  // This useQuery could just as well happen in some deeper
  // child to <Posts>, data will be available immediately either way
  const { data } = useQuery({
    queryKey: ['posts'],
    queryFn: () => getPosts(),
  })

  // This query was not prefetched on the server and will not start
  // fetching until on the client, both patterns are fine to mix.
  const { data: commentsData } = useQuery({
    queryKey: ['posts-comments'],
    queryFn: getComments,
  })

  // ...
}
```

위 예제에서 Next.js에만 특화된 부분은 파일 이름뿐입니다.

그 외 구조는 Server Components를 지원하는 다른 framework에서도 동일하게 적용할 수 있습니다.

SSR 가이드에서는 모든 route마다 `<HydrationBoundary>`를 두는 boilerplate를 제거할 수 있다고 설명했습니다.

하지만 Server Components에서는 이것이 불가능합니다.

> TypeScript 5.1.3 미만과 `@types/react` 18.2.8 미만에서 async Server Components 사용 시 타입 에러가 발생할 수 있습니다. 권장 방법은 두 버전을 최신으로 업데이트하는 것입니다. 임시 workaround로는 이 컴포넌트를 다른 곳에서 호출할 때 `{/* @ts-expect-error Server Component */}`를 추가할 수 있습니다.

> `Only plain objects, and a few built-ins, can be passed to Server Actions. Classes or null prototypes are not supported.` 에러가 발생하면 `queryFn`에 함수 참조를 직접 전달하지 않았는지 확인해야 합니다. `queryFn` 인자에는 여러 속성이 있고, 그중 일부는 serializable하지 않을 수 있으므로 함수를 호출하는 형태로 작성해야 합니다.

<br>

### 1.4 Nesting Server Components

Server Components의 장점은 React tree의 여러 레벨에 중첩될 수 있다는 점입니다.

덕분에 데이터를 애플리케이션 최상단에서만 prefetch하지 않고, 실제로 데이터가 사용되는 위치에 더 가깝게 prefetch할 수 있습니다.

이는 Remix loader와 비슷한 장점입니다.

아래 예제에서는 간결함을 위해 Client Components는 제외하고, Server Component가 다른 Server Component를 렌더링하는 형태를 사용합니다.

```tsx
// app/posts/page.tsx
import {
  dehydrate,
  HydrationBoundary,
  QueryClient,
} from '@tanstack/react-query'
import Posts from './posts'
import CommentsServerComponent from './comments-server'

export default async function PostsPage() {
  const queryClient = new QueryClient()

  await queryClient.prefetchQuery({
    queryKey: ['posts'],
    queryFn: getPosts,
  })

  return (
    <HydrationBoundary state={dehydrate(queryClient)}>
      <Posts />
      <CommentsServerComponent />
    </HydrationBoundary>
  )
}
```

```tsx
// app/posts/comments-server.tsx
import {
  dehydrate,
  HydrationBoundary,
  QueryClient,
} from '@tanstack/react-query'
import Comments from './comments'

export default async function CommentsServerComponent() {
  const queryClient = new QueryClient()

  await queryClient.prefetchQuery({
    queryKey: ['posts-comments'],
    queryFn: getComments,
  })

  return (
    <HydrationBoundary state={dehydrate(queryClient)}>
      <Comments />
    </HydrationBoundary>
  )
}
```

`<HydrationBoundary>`를 여러 위치에서 사용하는 것은 문제 없습니다.

또한 prefetch를 위해 여러 `queryClient`를 만들고 각각 dehydrate해도 괜찮습니다.

다만 위 예제에서는 `CommentsServerComponent`를 렌더링하기 전에 `getPosts`를 `await`합니다.

이 때문에 서버 측 waterfall이 발생합니다.

```plaintext
1. |> getPosts()
2.   |> getComments()
```

서버에서 데이터 소스까지의 latency가 낮다면 큰 문제가 아닐 수 있습니다.

그래도 이 구조가 waterfall을 만들 수 있다는 점은 알아야 합니다.

Next.js에서는 `page.tsx`뿐 아니라 다음 위치에서도 데이터를 prefetch할 수 있습니다.

- `layout.tsx`
- parallel routes

이들은 routing의 일부이므로 Next.js는 이들을 병렬로 가져올 수 있습니다.

따라서 위 `CommentsServerComponent`가 parallel route로 표현된다면 waterfall은 자동으로 flatten됩니다.

다른 framework가 Server Components를 지원하기 시작하면 서로 다른 routing convention을 가질 수 있습니다.

자세한 내용은 사용하는 framework 문서를 확인해야 합니다.

<br>

### 1.5 Alternative: Use a single queryClient for prefetching

위 예제에서는 데이터를 가져오는 각 Server Component마다 새로운 `queryClient`를 만들었습니다.

이 방식이 권장됩니다.

하지만 원한다면 모든 Server Components에서 재사용되는 단일 `queryClient`를 만들 수도 있습니다.

```tsx
// app/getQueryClient.tsx
import { QueryClient } from '@tanstack/react-query'
import { cache } from 'react'

// cache() is scoped per request, so we don't leak data between requests
const getQueryClient = cache(() => new QueryClient())

export default getQueryClient
```

이 방식의 장점은 Server Component에서 호출되는 어디서든 `getQueryClient()`로 client를 얻을 수 있다는 점입니다.

예를 들어 utility function 안에서도 사용할 수 있습니다.

단점도 있습니다.

- `dehydrate(getQueryClient())`를 호출할 때마다 전체 `queryClient`가 serialize됩니다.
- 이미 serialize된 query도 다시 포함됩니다.
- 현재 Server Component와 관련 없는 query도 포함됩니다.
- 이로 인해 불필요한 overhead가 생깁니다.

Next.js는 `fetch()`를 사용하는 요청을 이미 dedupe합니다.

하지만 다음 상황에서는 단일 `queryClient` 방식이 의미 있을 수 있습니다.

- `queryFn`에서 `fetch()`가 아닌 다른 방식을 사용하는 경우
- framework가 요청 dedupe를 자동으로 처리하지 않는 경우

이 경우 중복 serialization이 있더라도 단일 `queryClient`를 사용하는 방식이 적합할 수 있습니다.

향후 개선으로, 마지막 호출 이후 새로 추가된 query만 dehydrate하는 `dehydrateNew()` 같은 함수를 만들 수도 있습니다.

<br>

### 1.6 Data ownership and revalidation

Server Components를 사용할 때는 data ownership과 revalidation을 고려해야 합니다.

다음은 앞선 예제를 수정한 형태입니다.

```tsx
// app/posts/page.tsx
import {
  dehydrate,
  HydrationBoundary,
  QueryClient,
} from '@tanstack/react-query'
import Posts from './posts'

export default async function PostsPage() {
  const queryClient = new QueryClient()

  // Note we are now using fetchQuery()
  const posts = await queryClient.fetchQuery({
    queryKey: ['posts'],
    queryFn: getPosts,
  })

  return (
    <HydrationBoundary state={dehydrate(queryClient)}>
      {/* This is the new part */}
      <div>Nr of posts: {posts.length}</div>
      <Posts />
    </HydrationBoundary>
  )
}
```

이제 `getPosts` query의 데이터를 Server Component와 Client Component에서 모두 렌더링합니다.

초기 페이지 렌더에서는 괜찮습니다.

하지만 `staleTime`이 지난 뒤 클라이언트에서 query가 revalidate되면 문제가 생길 수 있습니다.

React Query는 Server Component를 어떻게 revalidate해야 하는지 알 수 없습니다.

따라서 클라이언트에서 데이터를 refetch하고 posts list가 다시 렌더링되더라도, 서버에서 렌더링한 `Nr of posts: {posts.length}`는 동기화되지 않을 수 있습니다.

`staleTime: Infinity`를 설정하면 React Query가 revalidate하지 않으므로 이 문제는 피할 수 있습니다.

하지만 React Query를 사용하는 목적을 생각하면, 일반적으로 원하는 방식은 아닐 수 있습니다.

React Query와 Server Components를 함께 사용하는 것이 적합한 경우는 다음과 같습니다.

- 이미 React Query를 사용하는 앱이 있고, 데이터 fetching을 모두 다시 작성하지 않고 Server Components로 마이그레이션하고 싶은 경우
- 익숙한 programming paradigm을 유지하면서 Server Components의 이점을 필요한 곳에만 적용하고 싶은 경우
- React Query가 해결하는 use case가 있지만, 선택한 framework가 이를 제공하지 않는 경우

React Query와 Server Components를 언제 함께 사용하는 것이 적절한지에 대한 일반적인 조언은 어렵습니다.

새 Server Components 앱을 처음 시작한다면 다음 접근이 권장됩니다.

- framework가 제공하는 data fetching 도구부터 사용합니다.
- 실제로 React Query가 필요해질 때까지 도입하지 않습니다.
- React Query가 끝내 필요하지 않을 수도 있습니다.
- 작업에 맞는 도구를 사용하는 것이 중요합니다.

React Query를 사용한다면 좋은 기준은 다음과 같습니다.

- error를 catch해야 하는 경우가 아니라면 `queryClient.fetchQuery`를 피합니다.
- `fetchQuery`를 사용하더라도 그 결과를 서버에서 렌더링하지 않습니다.
- 그 결과를 다른 컴포넌트에 전달하지 않습니다.
  - Client Component에 전달하는 것도 피합니다.

React Query 관점에서 Server Components는 데이터를 prefetch하는 장소로만 다루는 것이 좋습니다.

물론 Server Components가 일부 데이터를 소유하고, Client Components가 다른 데이터를 소유하는 것은 괜찮습니다.

중요한 것은 두 현실이 서로 어긋나지 않도록 하는 것입니다.

## 2. Streaming with Server Components

Next.js app router는 표시할 준비가 된 애플리케이션의 일부를 가능한 빨리 브라우저로 자동 streaming합니다.

덕분에 아직 pending 상태인 콘텐츠를 기다리지 않고, 준비된 콘텐츠를 즉시 표시할 수 있습니다.

이 동작은 `<Suspense>` boundary를 기준으로 이루어집니다.

`loading.tsx` 파일을 만들면 내부적으로 `<Suspense>` boundary가 자동 생성됩니다.

위에서 설명한 prefetching pattern을 사용하면 React Query는 이런 streaming 방식과 잘 호환됩니다.

각 Suspense boundary의 데이터가 resolve되면 Next.js는 완료된 콘텐츠를 렌더링하고 브라우저로 stream할 수 있습니다.

위에서 설명한 것처럼 `useQuery`를 사용해도 동작합니다.

그 이유는 실제 suspend가 prefetch를 `await`하는 시점에 발생하기 때문입니다.

React Query v5.40.0부터는 streaming을 위해 모든 prefetch를 `await`할 필요가 없습니다.

pending query도 dehydrate되어 클라이언트로 전송될 수 있습니다.

이를 통해 다음이 가능해집니다.

- prefetch를 가능한 한 일찍 시작합니다.
- 전체 Suspense boundary를 block하지 않습니다.
- query가 완료되는 시점에 데이터를 클라이언트로 stream합니다.

이 방식이 유용한 예시는 다음과 같습니다.

- 사용자 상호작용 이후에만 보이는 콘텐츠를 미리 prefetch하고 싶은 경우
- infinite query의 첫 페이지는 `await`하고 렌더링하되, 두 번째 페이지는 렌더링을 막지 않고 prefetch하고 싶은 경우

이를 사용하려면 `queryClient`가 pending query도 dehydrate하도록 설정해야 합니다.

이 설정은 전역으로 지정할 수도 있고, `dehydrate`에 직접 옵션을 전달할 수도 있습니다.

또한 `getQueryClient()` 함수를 `app/providers.tsx`에서 분리해야 합니다.

이 함수는 Server Component와 Client Provider 양쪽에서 사용되어야 하기 때문입니다.

```tsx
// app/get-query-client.ts
import {
  isServer,
  QueryClient,
  defaultShouldDehydrateQuery,
} from '@tanstack/react-query'

function makeQueryClient() {
  return new QueryClient({
    defaultOptions: {
      queries: {
        staleTime: 60 * 1000,
      },
      dehydrate: {
        // include pending queries in dehydration
        shouldDehydrateQuery: (query) =>
          defaultShouldDehydrateQuery(query) ||
          query.state.status === 'pending',
        shouldRedactErrors: (error) => {
          // We should not catch Next.js server errors
          // as that's how Next.js detects dynamic pages
          // so we cannot redact them.
          // Next.js also automatically redacts errors for us
          // with better digests.
          return false
        },
      },
    },
  })
}

let browserQueryClient: QueryClient | undefined = undefined

export function getQueryClient() {
  if (isServer) {
    // Server: always make a new query client
    return makeQueryClient()
  } else {
    // Browser: make a new query client if we don't already have one
    // This is very important, so we don't re-make a new client if React
    // suspends during the initial render. This may not be needed if we
    // have a suspense boundary BELOW the creation of the query client
    if (!browserQueryClient) browserQueryClient = makeQueryClient()
    return browserQueryClient
  }
}
```

> 이 방식은 Next.js와 Server Components에서 동작합니다. React가 Promise를 Client Components로 전달할 때 wire를 통해 serialize할 수 있기 때문입니다.

이제 `HydrationBoundary`만 제공하면 됩니다.

prefetch를 더 이상 `await`할 필요가 없습니다.

```tsx
// app/posts/page.tsx
import { dehydrate, HydrationBoundary } from '@tanstack/react-query'
import { getQueryClient } from './get-query-client'
import Posts from './posts'

// the function doesn't need to be `async` because we don't `await` anything
export default function PostsPage() {
  const queryClient = getQueryClient()

  // look ma, no await
  queryClient.prefetchQuery({
    queryKey: ['posts'],
    queryFn: getPosts,
  })

  return (
    <HydrationBoundary state={dehydrate(queryClient)}>
      <Posts />
    </HydrationBoundary>
  )
}
```

클라이언트에서는 Promise가 QueryCache에 들어갑니다.

따라서 `Posts` 컴포넌트 안에서 `useSuspenseQuery`를 호출해 서버에서 만들어진 Promise를 사용할 수 있습니다.

```tsx
// app/posts/posts.tsx
'use client'

export default function Posts() {
  const { data } = useSuspenseQuery({ queryKey: ['posts'], queryFn: getPosts })

  // ...
}
```

> `useSuspenseQuery` 대신 `useQuery`를 사용할 수도 있습니다. 이 경우 Promise는 여전히 올바르게 선택됩니다. 하지만 Next.js는 이 경우 suspend하지 않습니다. 컴포넌트는 pending 상태로 렌더링되며, 콘텐츠의 server rendering에서도 opt out됩니다.

non-JSON 데이터 타입을 사용하고, 서버에서 query result를 serialize한다면 다음 옵션을 지정할 수 있습니다.

- `dehydrate.serializeData`
- `hydrate.deserializeData`

이 옵션을 사용하면 boundary 양쪽에서 데이터를 serialize / deserialize할 수 있습니다.

그 결과 서버와 클라이언트 cache의 데이터 형식을 동일하게 유지할 수 있습니다.

```tsx
// app/get-query-client.ts
import { QueryClient, defaultShouldDehydrateQuery } from '@tanstack/react-query'
import { deserialize, serialize } from './transformer'

function makeQueryClient() {
  return new QueryClient({
    defaultOptions: {
      // ...
      hydrate: {
        deserializeData: deserialize,
      },
      dehydrate: {
        serializeData: serialize,
      },
    },
  })
}

// ...
```

```tsx
// app/posts/page.tsx
import {
  dehydrate,
  HydrationBoundary,
  QueryClient,
} from '@tanstack/react-query'
import { getQueryClient } from './get-query-client'
import { serialize } from './transformer'
import Posts from './posts'

export default function PostsPage() {
  const queryClient = getQueryClient()

  // look ma, no await
  queryClient.prefetchQuery({
    queryKey: ['posts'],
    queryFn: () => getPosts().then(serialize), // <-- serialize the data on the server
  })

  return (
    <HydrationBoundary state={dehydrate(queryClient)}>
      <Posts />
    </HydrationBoundary>
  )
}
```

```tsx
// app/posts/posts.tsx
'use client'

  const { data } = useSuspenseQuery({ queryKey: ['posts'], queryFn: getPosts })

}
```

이제 `getPosts` 함수는 예를 들어 Temporal datetime object 같은 값을 반환할 수 있습니다.

단, transformer가 해당 데이터 타입을 serialize / deserialize할 수 있어야 합니다.

### 2.1 Using the Persist Adapter with Streaming

persist adapter를 Streaming with Server Components 기능과 함께 사용할 때는 Promise를 storage에 저장하지 않도록 주의해야 합니다.

pending query는 dehydrate되어 클라이언트로 stream될 수 있습니다.

따라서 persister는 successful query만 persist하도록 설정해야 합니다.

```tsx
<PersistQueryClientProvider
  client={queryClient}
  persistOptions={{
    persister,
    // We don't want to save promises into the storage, so we only persist successful queries
    dehydrateOptions: { shouldDehydrateQuery: defaultShouldDehydrateQuery },
  }}
>
  {children}
</PersistQueryClientProvider>
```

이 설정은 성공적으로 resolve된 query만 storage에 persist되도록 보장합니다.

그 결과 pending promise로 인한 serialization 문제를 방지할 수 있습니다.

## 3. Experimental streaming without prefetching in Next.js

위에서 설명한 prefetching solution이 권장됩니다.

그 이유는 다음 두 상황 모두에서 request waterfall을 flatten하기 때문입니다.

- 초기 페이지 로드
- 이후 페이지 navigation

하지만 prefetching을 완전히 건너뛰면서 streaming SSR을 동작하게 하는 experimental 방식도 있습니다.

이를 위해 `@tanstack/react-query-next-experimental`을 사용할 수 있습니다.

이 package를 사용하면 Client Component에서 `useSuspenseQuery`를 호출하는 것만으로 서버에서 데이터를 fetch할 수 있습니다.

결과는 SuspenseBoundary가 resolve될 때 서버에서 클라이언트로 stream됩니다.

`useSuspenseQuery`를 `<Suspense>` boundary 없이 호출하면 HTML response는 fetch가 resolve될 때까지 시작되지 않습니다.

상황에 따라 원하는 동작일 수 있지만, TTFB에는 좋지 않습니다.

이를 사용하려면 앱을 `ReactQueryStreamedHydration` 컴포넌트로 감쌉니다.

```tsx
// app/providers.tsx
'use client'

import {
  isServer,
  QueryClient,
  QueryClientProvider,
} from '@tanstack/react-query'
import * as React from 'react'
import { ReactQueryStreamedHydration } from '@tanstack/react-query-next-experimental'

function makeQueryClient() {
  return new QueryClient({
    defaultOptions: {
      queries: {
        // With SSR, we usually want to set some default staleTime
        // above 0 to avoid refetching immediately on the client
        staleTime: 60 * 1000,
      },
    },
  })
}

let browserQueryClient: QueryClient | undefined = undefined

function getQueryClient() {
  if (isServer) {
    // Server: always make a new query client
    return makeQueryClient()
  } else {
    // Browser: make a new query client if we don't already have one
    // This is very important, so we don't re-make a new client if React
    // suspends during the initial render. This may not be needed if we
    // have a suspense boundary BELOW the creation of the query client
    if (!browserQueryClient) browserQueryClient = makeQueryClient()
    return browserQueryClient
  }
}

export function Providers(props: { children: React.ReactNode }) {
  // NOTE: Avoid useState when initializing the query client if you don't
  //       have a suspense boundary between this and the code that may
  //       suspend because React will throw away the client on the initial
  //       render if it suspends and there is no boundary
  const queryClient = getQueryClient()

  return (
    <QueryClientProvider client={queryClient}>
      <ReactQueryStreamedHydration>
        {props.children}
      </ReactQueryStreamedHydration>
    </QueryClientProvider>
  )
}
```

이 방식의 큰 장점은 SSR을 동작시키기 위해 query를 수동으로 prefetch할 필요가 없다는 점입니다.

결과도 stream됩니다.

따라서 DX가 좋고 코드 복잡도가 낮아집니다.

단점은 request waterfall과 관련이 있습니다.

Server Components와 prefetching을 함께 사용하면 다음 두 상황에서 request waterfall을 효과적으로 제거할 수 있습니다.

- 초기 페이지 로드
- 이후 navigation

하지만 prefetch 없는 접근 방식은 초기 페이지 로드에서만 waterfall을 flatten합니다.

페이지 navigation에서는 다음처럼 깊은 waterfall이 다시 발생할 수 있습니다.

```plaintext
1. |> JS for <Feed>
2.   |> getFeed()
3.     |> JS for <GraphFeedItem>
4.       |> getGraphDataById()
```

이는 `getServerSideProps`나 `getStaticProps`를 사용하는 방식보다도 나쁠 수 있습니다.

그 방식에서는 적어도 data fetching과 code fetching을 병렬화할 수 있었기 때문입니다.

다음 조건에 해당한다면 이 tradeoff가 괜찮을 수 있습니다.

- performance보다 DX, iteration, shipping speed, 낮은 코드 복잡도를 더 중요하게 보는 경우
- 깊게 중첩된 query가 없는 경우
- `useSuspenseQueries` 같은 도구로 병렬 fetching을 사용해 request waterfall을 잘 관리하는 경우

> 두 접근 방식을 조합하는 것도 가능할 수 있습니다. 하지만 아직 공식적으로 시도된 방식은 아닙니다.
