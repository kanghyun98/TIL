# Performance & Request Waterfalls

애플리케이션 성능은 넓고 복잡한 영역입니다. React Query가 API 자체를 더 빠르게 만들 수는 없지만, React Query를 사용할 때 좋은 성능을 유지하기 위해 주의해야 할 지점이 있습니다.

React Query처럼 컴포넌트 안에서 데이터를 가져올 수 있는 라이브러리에서 가장 큰 성능 문제는 **Request Waterfall**입니다.

<br>

## 1. What is a Request Waterfall?

Request Waterfall은 한 리소스(code, CSS, images, data) 요청이 끝나야 다음 리소스 요청이 시작되는 상황을 말합니다.

웹 페이지를 예로 들면, 브라우저는 CSS나 JS 등을 로드하기 전에 먼저 markup을 로드해야 합니다.

```plaintext
1. |-> Markup
2.   |-> CSS
2.   |-> JS
2.   |-> Image
```

JS 파일 안에서 CSS를 가져오면 double waterfall이 됩니다.

```plaintext
1. |-> Markup
2.   |-> JS
3.     |-> CSS
```

그 CSS가 background image를 사용하면 triple waterfall이 됩니다.

```plaintext
1. |-> Markup
2.   |-> JS
3.     |-> CSS
4.       |-> Image
```

Request Waterfall을 발견하고 분석하는 가장 좋은 방법은 보통 브라우저 DevTools의 `Network` 탭을 여는 것입니다.

각 waterfall은 리소스가 로컬 캐시에 있지 않은 이상 서버와의 최소 한 번의 roundtrip을 의미합니다. 실제로는 브라우저가 연결을 설정하기 위해 여러 번 오가야 하므로, 일부 waterfall은 하나 이상의 roundtrip을 의미할 수도 있습니다.

Request Waterfall의 부정적인 영향은 사용자의 latency에 크게 의존합니다.

예를 들어 triple waterfall은 실제로 4번의 서버 roundtrip을 나타냅니다.

- latency가 250ms인 경우
  - 3G 네트워크나 좋지 않은 네트워크 환경에서 드물지 않은 수치입니다.
- latency만 계산해도 `4 * 250 = 1000ms`가 걸립니다.
- 이를 첫 번째 예시처럼 2번의 roundtrip으로 flatten할 수 있다면 `500ms`가 됩니다.
- background image를 절반에 가까운 시간으로 로드할 수도 있습니다.

<br>

## 2. Request Waterfalls & React Query

Server Rendering이 없는 경우를 먼저 생각해보겠습니다.

query를 시작하기 전에도 JS를 먼저 로드해야 합니다. 따라서 데이터를 화면에 보여주기 전까지 double waterfall이 발생합니다.

```plaintext
1. |-> Markup
2.   |-> JS
3.     |-> Query
```

이를 기준으로 React Query에서 Request Waterfall을 만들 수 있는 몇 가지 패턴과 피하는 방법을 살펴봅니다.

- Single Component Waterfalls / Serial Queries
- Nested Component Waterfalls
- Code Splitting

<br>

### 2.1. Single Component Waterfalls / Serial Queries

하나의 컴포넌트가 먼저 하나의 query를 가져오고, 그 다음 다른 query를 가져오면 request waterfall이 됩니다.

두 번째 query가 첫 번째 query의 데이터에 의존하는 Dependent Query일 때 이런 상황이 발생할 수 있습니다.

```tsx
// Get the user
const { data: user } = useQuery({
  queryKey: ['user', email],
  queryFn: getUserByEmail,
})

const userId = user?.id

// Then get the user's projects
const {
  status,
  fetchStatus,
  data: projects,
} = useQuery({
  queryKey: ['projects', userId],
  queryFn: getProjectsByUser,
  // The query will not execute until the userId exists
  enabled: !!userId,
})
```

항상 가능한 것은 아니지만, 최적의 성능을 위해서는 API를 재구성해 두 요청을 하나의 query로 가져오는 것이 좋습니다.

위 예시에서는 다음 흐름이 waterfall을 만듭니다.

- 먼저 `getUserByEmail` 호출
- 응답에서 `userId` 획득
- 이후 `getProjectsByUser` 호출

이를 flatten하려면 `getProjectsByUserEmail` 같은 새로운 query를 도입할 수 있습니다.

API를 재구성하지 않고 dependent query를 완화하는 또 다른 방법은 waterfall을 서버로 옮기는 것입니다.

서버는 latency가 더 낮기 때문에, 이 방식은 Server Components의 기본 아이디어와 연결됩니다.

<br>

Suspense와 함께 React Query를 사용할 때도 serial query가 발생할 수 있습니다.

```tsx
function App() {
  // The following queries will execute in serial, causing separate roundtrips to the server:
  const usersQuery = useSuspenseQuery({ queryKey: ['users'], queryFn: fetchUsers })
  const teamsQuery = useSuspenseQuery({ queryKey: ['teams'], queryFn: fetchTeams })
  const projectsQuery = useSuspenseQuery({ queryKey: ['projects'], queryFn: fetchProjects })

  // Note that since the queries above suspend rendering, no data
  // gets rendered until all of the queries finished
  ...
}
```

일반 `useQuery`를 사용하면 이 query들은 병렬로 실행됩니다. 여러 개의 suspenseful query가 하나의 컴포넌트에 있다면, `useSuspenseQueries`를 사용해 쉽게 해결할 수 있습니다.

```tsx
const [usersQuery, teamsQuery, projectsQuery] = useSuspenseQueries({
  queries: [
    { queryKey: ['users'], queryFn: fetchUsers },
    { queryKey: ['teams'], queryFn: fetchTeams },
    { queryKey: ['projects'], queryFn: fetchProjects },
  ],
})
```

### 2.2. Nested Component Waterfalls

Nested Component Waterfall은 부모와 자식 컴포넌트가 모두 query를 가지고 있고, 부모의 query가 끝나기 전까지 자식을 렌더링하지 않는 경우를 말합니다.

이 현상은 다음 모두에서 발생할 수 있습니다.

- `useQuery`
- `useSuspenseQuery`

자식이 부모의 데이터에 따라 조건부로 렌더링되거나, 자식 query가 부모로부터 전달받은 값을 필요로 한다면 dependent nested component waterfall이 됩니다.

먼저 자식이 부모에 의존하지 않는 예시를 보겠습니다.

```tsx
function Article({ id }) {
  const { data: articleData, isPending } = useQuery({
    queryKey: ['article', id],
    queryFn: getArticleById,
  })

  if (isPending) {
    return 'Loading article...'
  }

  return (
    <>
      <ArticleHeader articleData={articleData} />
      <ArticleBody articleData={articleData} />
      <Comments id={id} />
    </>
  )
}

function Comments({ id }) {
  const { data, isPending } = useQuery({
    queryKey: ['article-comments', id],
    queryFn: getArticleCommentsById,
  })

  ...
}
```

`<Comments>`는 부모로부터 `id` prop을 받지만, 이 `id`는 `<Article>`이 렌더링될 때 이미 사용할 수 있습니다.

따라서 comments를 article과 동시에 가져오지 못할 이유가 없습니다.

실제 애플리케이션에서는 자식이 부모보다 훨씬 깊게 중첩되어 있을 수 있습니다.

이런 waterfall은 발견하고 고치기 더 어렵습니다.

이 예시에서는 comments query를 부모로 끌어올려 waterfall을 flatten할 수 있습니다.

```tsx
function Article({ id }) {
  const { data: articleData, isPending: articlePending } = useQuery({
    queryKey: ['article', id],
    queryFn: getArticleById,
  })

  const { data: commentsData, isPending: commentsPending } = useQuery({
    queryKey: ['article-comments', id],
    queryFn: getArticleCommentsById,
  })

  if (articlePending) {
    return 'Loading article...'
  }

  return (
    <>
      <ArticleHeader articleData={articleData} />
      <ArticleBody articleData={articleData} />
      {commentsPending ? (
        'Loading comments...'
      ) : (
        <Comments commentsData={commentsData} />
      )}
    </>
  )
}
```

이제 두 query는 병렬로 fetch됩니다.

Suspense를 사용한다면 두 query를 하나의 `useSuspenseQueries`로 합치는 것이 좋습니다.

이 waterfall을 flatten하는 또 다른 방법도 있습니다.

- `<Article>` 컴포넌트에서 comments를 prefetch
- 페이지 로드나 페이지 navigation 시 router level에서 두 query를 모두 prefetch

<br>

다음은 Dependent Nested Component Waterfall 예시입니다.

```tsx
function Feed() {
  const { data, isPending } = useQuery({
    queryKey: ['feed'],
    queryFn: getFeed,
  })

  if (isPending) {
    return 'Loading feed...'
  }

  return (
    <>
      {data.map((feedItem) => {
        if (feedItem.type === 'GRAPH') {
          return <GraphFeedItem key={feedItem.id} feedItem={feedItem} />
        }

        return <StandardFeedItem key={feedItem.id} feedItem={feedItem} />
      })}
    </>
  )
}

function GraphFeedItem({ feedItem }) {
  const { data, isPending } = useQuery({
    queryKey: ['graph', feedItem.id],
    queryFn: getGraphDataById,
  })

  ...
}
```

두 번째 query인 `getGraphDataById`는 두 가지 방식으로 부모에 의존합니다.

- `feedItem`이 graph일 때만 실행됩니다.
- 부모로부터 `id`를 받아야 합니다.

```plaintext
1. |> getFeed()
2.   |> getGraphDataById()
```

이 예시에서는 query를 부모로 끌어올리거나 prefetch를 추가하는 것만으로 waterfall을 간단히 flatten할 수 없습니다.

이 가이드의 앞부분에서 본 dependent query 예시처럼, 한 가지 방법은 `getFeed` query에 graph data를 포함하도록 API를 리팩터링하는 것입니다.

더 고급 해결책은 Server Components를 활용해 waterfall을 latency가 더 낮은 서버로 옮기는 것입니다.

다만 이는 매우 큰 아키텍처 변경이 될 수 있습니다.

몇 개의 query waterfall이 있더라도 좋은 성능을 낼 수는 있습니다.

하지만 query waterfall은 흔한 성능 이슈이므로, 이를 인지하고 주의해야 합니다.

특히 Code Splitting이 포함되면 더 교묘한 문제가 될 수 있습니다.

<br>

### 2.3. Code Splitting

애플리케이션의 JS 코드를 작은 chunk로 나누고 필요한 부분만 로드하는 것은 좋은 성능을 얻기 위한 중요한 단계입니다.

하지만 단점도 있습니다. Code Splitting은 종종 request waterfall을 만듭니다. code split된 코드 안에 query까지 포함되어 있으면 문제가 더 커집니다.

다음은 앞서 본 Feed 예시를 약간 수정한 형태입니다.

```tsx
// This lazy loads the GraphFeedItem component, meaning
// it wont start loading until something renders it
const GraphFeedItem = React.lazy(() => import('./GraphFeedItem'))

function Feed() {
  const { data, isPending } = useQuery({
    queryKey: ['feed'],
    queryFn: getFeed,
  })

  if (isPending) {
    return 'Loading feed...'
  }

  return (
    <>
      {data.map((feedItem) => {
        if (feedItem.type === 'GRAPH') {
          return <GraphFeedItem key={feedItem.id} feedItem={feedItem} />
        }

        return <StandardFeedItem key={feedItem.id} feedItem={feedItem} />
      })}
    </>
  )
}

// GraphFeedItem.tsx
function GraphFeedItem({ feedItem }) {
  const { data, isPending } = useQuery({
    queryKey: ['graph', feedItem.id],
    queryFn: getGraphDataById,
  })

  ...
}
```

이 예시는 double waterfall을 가집니다.

```plaintext
1. |> getFeed()
2.   |> JS for <GraphFeedItem>
3.     |> getGraphDataById()
```

하지만 첫 페이지 로드를 기준으로 보면, graph를 렌더링하기 전에 서버와 5번의 roundtrip을 완료해야 합니다.

```plaintext
1. |> Markup
2.   |> JS for <Feed>
3.     |> getFeed()
4.       |> JS for <GraphFeedItem>
5.         |> getGraphDataById()
```

Server Rendering을 사용하면 이 모습은 조금 달라집니다.

또한 `<Feed>`를 포함하는 route 자체도 code split되는 경우가 드물지 않습니다.

이 경우에는 또 하나의 hop이 추가될 수 있습니다.

code split 상황에서는 `getGraphDataById` query를 `<Feed>` 컴포넌트로 끌어올리고 조건부로 실행하거나, 조건부 prefetch를 추가하는 것이 도움이 될 수 있습니다.

그러면 query가 code와 병렬로 fetch될 수 있습니다.

```plaintext
1. |> getFeed()
2.   |> getGraphDataById()
2.   |> JS for <GraphFeedItem>
```

다만 이는 tradeoff입니다.

`getGraphDataById`를 위한 data fetching code를 `<Feed>`와 같은 bundle에 포함하게 됩니다.

따라서 자신의 상황에 무엇이 더 적합한지 평가해야 합니다.

tradeoff는 다음과 같습니다.

- data fetching code를 main bundle에 모두 포함
  - 거의 사용하지 않는 경우에도 포함됩니다.
- data fetching code를 code split bundle에 넣음
  - request waterfall이 발생합니다.

이 tradeoff는 Server Components가 등장한 동기 중 하나입니다.

Server Components를 사용하면 두 문제를 모두 피할 수 있습니다.

<br>

## 3. Summary and takeaways

Request Waterfall은 매우 흔하고 복잡한 성능 문제입니다. 또한 여러 tradeoff를 가집니다.

애플리케이션에서는 다음과 같은 방식으로 request waterfall이 의도치 않게 생길 수 있습니다.

- 부모에 이미 query가 있다는 것을 모르고 자식에 query를 추가하는 경우
- 자식에 이미 query가 있다는 것을 모르고 부모에 query를 추가하는 경우
- query를 가진 descendants가 있는 컴포넌트를 query가 있는 ancestor 아래의 새 부모로 옮기는 경우
- 그 외 여러 경우



이런 우발적인 복잡성 때문에 waterfall을 의식하고, 애플리케이션에서 정기적으로 확인하는 것이 좋습니다. 좋은 방법 중 하나는 가끔씩 `Network` 탭을 살펴보는 것입니다. 좋은 성능을 위해 모든 waterfall을 flatten할 필요는 없습니다. 하지만 영향이 큰 waterfall은 주의 깊게 살펴봐야 합니다.
