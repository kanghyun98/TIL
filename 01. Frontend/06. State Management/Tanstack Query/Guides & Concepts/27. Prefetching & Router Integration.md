# Prefetching & Router Integration

특정 데이터가 필요할 것을 알고 있거나 예상할 수 있다면, prefetching을 사용해 해당 데이터를 미리 캐시에 채울 수 있습니다.  이렇게 하면 사용자가 데이터를 필요로 하는 시점에 더 빠른 경험을 제공할 수 있습니다.

Prefetching에는 몇 가지 패턴이 있습니다.

1. In event handlers
2. In components
3. Via router integration
4. During Server Rendering (another form of router integration)

<br>

## 1. prefetchQuery & prefetchInfiniteQuery

구체적인 prefetch 패턴을 보기 전에 `prefetchQuery`와 `prefetchInfiniteQuery`를 먼저 살펴봅니다.

기본 동작은 다음과 같습니다.

- 기본적으로 이 함수들은 `queryClient`에 설정된 기본 `staleTime`을 사용합니다.
  - 캐시에 있는 기존 데이터가 fresh한지 확인합니다.
  - stale 상태라면 다시 fetch가 필요한지 판단합니다.

- 특정 `staleTime`을 직접 전달할 수도 있습니다.

```tsx
prefetchQuery({
  queryKey: ['todos'],
  queryFn: fn,
  staleTime: 5000,
})
```

- 이 `staleTime`은 prefetch에만 사용됩니다.
  - `useQuery` 호출에도 동일하게 적용하려면 `useQuery`에도 따로 설정해야 합니다.

- `staleTime`을 무시하고 캐시에 데이터가 있으면 항상 반환하고 싶다면 `ensureQueryData`를 사용할 수 있습니다.

- 서버에서 prefetch하는 경우:
  - 해당 `queryClient`의 기본 `staleTime`을 `0`보다 크게 설정하는 것이 좋습니다.
  - 이렇게 하면 각 prefetch 호출마다 `staleTime`을 따로 넘길 필요가 줄어듭니다.

- Prefetch된 query에 대해 `useQuery` 인스턴스가 나타나지 않으면:
  - `gcTime`에 지정된 시간이 지난 뒤 삭제되고 garbage collected 됩니다.

- 이 함수들은 `Promise<void>`를 반환합니다.
  - 따라서 query data를 반환하지 않습니다.
  - 데이터가 필요하다면 `fetchQuery` 또는 `fetchInfiniteQuery`를 사용해야 합니다.

- Prefetch 함수는 error를 throw하지 않습니다.
  - 보통 이후 `useQuery`에서 다시 fetch를 시도할 수 있기 때문입니다.
  - error를 catch해야 한다면 `fetchQuery` 또는 `fetchInfiniteQuery`를 사용해야 합니다.

`prefetchQuery`는 다음처럼 사용할 수 있습니다.

```tsx
const prefetchTodos = async () => {
  // 이 query의 결과는 일반 query처럼 캐시됩니다.
  await queryClient.prefetchQuery({
    queryKey: ['todos'],
    queryFn: fetchTodos,
  })
}
```

Infinite Query도 일반 Query처럼 prefetch할 수 있습니다.

기본적으로는 다음과 같이 동작합니다.

- 첫 번째 페이지만 prefetch됩니다.
- 해당 결과는 지정한 `QueryKey` 아래에 저장됩니다.

두 페이지 이상을 prefetch하려면 `pages` 옵션을 사용할 수 있습니다.  
이 경우 `getNextPageParam` 함수도 함께 제공해야 합니다.

```tsx
const prefetchProjects = async () => {
  // 이 query의 결과는 일반 query처럼 캐시됩니다.
  await queryClient.prefetchInfiniteQuery({
    queryKey: ['projects'],
    queryFn: fetchProjects,
    initialPageParam: 0,
    getNextPageParam: (lastPage, pages) => lastPage.nextCursor,
    pages: 3, // 처음 3개의 페이지를 prefetch합니다.
  })
}
```

## 2. Prefetch in event handlers

가장 단순한 prefetch 방식은 사용자가 어떤 요소와 상호작용할 때 prefetch를 실행하는 것입니다.

아래 예시는 `queryClient.prefetchQuery`를 사용해 `onMouseEnter` 또는 `onFocus` 시점에 prefetch를 시작합니다.

```tsx
function ShowDetailsButton() {
  const queryClient = useQueryClient()

  const prefetch = () => {
    queryClient.prefetchQuery({
      queryKey: ['details'],
      queryFn: getDetailsData,
      // Prefetch는 데이터가 staleTime보다 오래되었을 때만 실행됩니다.
      // 이런 경우 staleTime을 설정하는 것이 좋습니다.
      staleTime: 60000,
    })
  }

  return (
    <button onMouseEnter={prefetch} onFocus={prefetch} onClick={...}>
      Show Details
    </button>
  )
}
```

## 3. Prefetch in components

Component lifecycle 중 prefetching은 다음 상황에서 유용합니다.

- 어떤 child 또는 descendant가 특정 데이터를 필요로 할 것을 알고 있습니다.
- 하지만 다른 query의 loading이 끝나기 전까지는 해당 컴포넌트를 렌더링할 수 없습니다.

예를 들어 다음과 같은 구조가 있습니다.

```tsx
function Article({ id }) {
  const { data: articleData, isPending } = useQuery({
    queryKey: ['article', id],
    queryFn: getArticleById,
  })

  if (isPending) {
    return 'Loading article...'
  }

  return (
    <>
      <ArticleHeader articleData={articleData} />
      <ArticleBody articleData={articleData} />
      <Comments id={id} />
    </>
  )
}

function Comments({ id }) {
  const { data, isPending } = useQuery({
    queryKey: ['article-comments', id],
    queryFn: getArticleCommentsById,
  })

  ...
}
```

이 구조는 다음과 같은 request waterfall을 만듭니다.

```plaintext
1. |> getArticleById()
2.   |> getArticleCommentsById()
```

이 waterfall을 평탄화하고 성능을 개선하는 방법 중 하나는 `getArticleCommentsById` query를 parent로 끌어올린 뒤 결과를 prop으로 전달하는 것입니다.

하지만 다음과 같은 경우에는 이 방식이 적합하지 않을 수 있습니다.

- 컴포넌트들이 서로 관련이 없습니다.
- 컴포넌트 사이에 여러 계층이 있습니다.
- query 결과를 prop으로 전달하는 구조가 바람직하지 않습니다.

이 경우 parent에서 query를 prefetch할 수 있습니다.  
가장 단순한 방법은 query를 사용하되 결과는 무시하는 것입니다.

```tsx
function Article({ id }) {
  const { data: articleData, isPending } = useQuery({
    queryKey: ['article', id],
    queryFn: getArticleById,
  })

  // Prefetch
  useQuery({
    queryKey: ['article-comments', id],
    queryFn: getArticleCommentsById,
    // Optional optimization to avoid rerenders when this query changes:
    notifyOnChangeProps: [],
  })

  if (isPending) {
    return 'Loading article...'
  }

  return (
    <>
      <ArticleHeader articleData={articleData} />
      <ArticleBody articleData={articleData} />
      <Comments id={id} />
    </>
  )
}

function Comments({ id }) {
  const { data, isPending } = useQuery({
    queryKey: ['article-comments', id],
    queryFn: getArticleCommentsById,
  })

  ...
}
```

이렇게 하면 `'article-comments'` fetch가 즉시 시작되고 waterfall이 평탄화됩니다.

```plaintext
1. |> getArticleById()
1. |> getArticleCommentsById()
```

Suspense와 함께 prefetch하려면 조금 다르게 처리해야 합니다.

- `useSuspenseQueries`로 prefetch할 수 없습니다.
  - prefetch가 component rendering을 block하기 때문입니다.

- 같은 컴포넌트에서 prefetch 목적으로 `useQuery`를 사용할 수도 없습니다.
  - suspenseful query가 resolve된 뒤에야 prefetch가 시작되기 때문입니다.

이 경우 라이브러리에서 제공하는 다음 hook을 사용할 수 있습니다.

- `usePrefetchQuery`
- `usePrefetchInfiniteQuery`

실제로 데이터가 필요한 컴포넌트에서는 `useSuspenseQuery`를 사용할 수 있습니다.  
나중에 렌더링되는 컴포넌트는 별도의 `<Suspense>` boundary로 감싸는 것이 좋을 수 있습니다.  
이렇게 하면 prefetch하는 “secondary” query가 “primary” data 렌더링을 막지 않습니다.

```tsx
function ArticleLayout({ id }) {
  usePrefetchQuery({
    queryKey: ['article-comments', id],
    queryFn: getArticleCommentsById,
  })

  return (
    <Suspense fallback="Loading article">
      <Article id={id} />
    </Suspense>
  )
}

function Article({ id }) {
  const { data: articleData, isPending } = useSuspenseQuery({
    queryKey: ['article', id],
    queryFn: getArticleById,
  })

  ...
}
```

또 다른 방법은 query function 내부에서 prefetch하는 것입니다.  
이 방식은 article을 fetch할 때마다 comments도 필요할 가능성이 높다는 것을 알고 있을 때 적합합니다.

```tsx
const queryClient = useQueryClient()

const { data: articleData, isPending } = useQuery({
  queryKey: ['article', id],
  queryFn: (...args) => {
    queryClient.prefetchQuery({
      queryKey: ['article-comments', id],
      queryFn: getArticleCommentsById,
    })

    return getArticleById(...args)
  },
})
```

Effect 안에서 prefetch하는 것도 가능합니다.  
다만 같은 컴포넌트에서 `useSuspenseQuery`를 사용하고 있다면 주의해야 합니다.

- effect는 query가 끝난 뒤에야 실행됩니다.
- 따라서 원하는 시점보다 늦게 prefetch가 시작될 수 있습니다.

```tsx
const queryClient = useQueryClient()

useEffect(() => {
  queryClient.prefetchQuery({
    queryKey: ['article-comments', id],
    queryFn: getArticleCommentsById,
  })
}, [queryClient, id])
```

Component lifecycle 중 query를 prefetch하는 방법은 여러 가지입니다.  
상황에 맞는 방식을 선택하면 됩니다.

- Suspense boundary 이전에 `usePrefetchQuery` 또는 `usePrefetchInfiniteQuery`로 prefetch
- `useQuery` 또는 `useSuspenseQueries`를 사용하고 결과 무시
- Query function 내부에서 prefetch
- Effect 안에서 prefetch

### 3.1 Dependent Queries & Code Splitting

때로는 다른 fetch 결과에 따라 조건부로 prefetch하고 싶을 수 있습니다.

다음 예시를 살펴봅니다.

```tsx
// This lazy loads the GraphFeedItem component, meaning
// it wont start loading until something renders it
const GraphFeedItem = React.lazy(() => import('./GraphFeedItem'))

function Feed() {
  const { data, isPending } = useQuery({
    queryKey: ['feed'],
    queryFn: getFeed,
  })

  if (isPending) {
    return 'Loading feed...'
  }

  return (
    <>
      {data.map((feedItem) => {
        if (feedItem.type === 'GRAPH') {
          return <GraphFeedItem key={feedItem.id} feedItem={feedItem} />
        }

        return <StandardFeedItem key={feedItem.id} feedItem={feedItem} />
      })}
    </>
  )
}

// GraphFeedItem.tsx
function GraphFeedItem({ feedItem }) {
  const { data, isPending } = useQuery({
    queryKey: ['graph', feedItem.id],
    queryFn: getGraphDataById,
  })

  ...
}
```

이 예시는 다음과 같은 double request waterfall을 만듭니다.

```plaintext
1. |> getFeed()
2.   |> JS for <GraphFeedItem>
3.     |> getGraphDataById()
```

API를 재구성해서 `getFeed()`가 필요한 경우 `getGraphDataById()` 데이터까지 반환하도록 만들 수 없다면, `getFeed -> getGraphDataById` waterfall 자체를 제거할 수는 없습니다.

하지만 conditional prefetching을 활용하면 code와 data를 병렬로 load할 수 있습니다.

이 예시에서는 query function 내부에서 처리합니다.

```tsx
function Feed() {
  const queryClient = useQueryClient()

  const { data, isPending } = useQuery({
    queryKey: ['feed'],
    queryFn: async (...args) => {
      const feed = await getFeed(...args)

      for (const feedItem of feed) {
        if (feedItem.type === 'GRAPH') {
          queryClient.prefetchQuery({
            queryKey: ['graph', feedItem.id],
            queryFn: getGraphDataById,
          })
        }
      }

      return feed
    },
  })

  ...
}
```

이렇게 하면 code와 data를 병렬로 load할 수 있습니다.

```plaintext
1. |> getFeed()
2.   |> JS for <GraphFeedItem>
2.   |> getGraphDataById()
```

다만 tradeoff가 있습니다.

- `getGraphDataById` 코드가 `<GraphFeedItem>`의 JS가 아니라 parent bundle에 포함됩니다.
- 따라서 어떤 성능 tradeoff가 더 나은지는 case by case로 판단해야 합니다.

판단 기준은 다음과 같습니다.

- `GraphFeedItem`이 자주 나타날 가능성이 높다면:
  - parent에 코드를 포함하는 것이 좋을 수 있습니다.

- `GraphFeedItem`이 매우 드물게 나타난다면:
  - parent에 포함하지 않는 편이 나을 수 있습니다.

## 4. Router Integration

Component tree 내부에서 data fetching을 처리하면 request waterfall이 쉽게 발생할 수 있습니다.  
또한 이를 해결하기 위한 방식들이 애플리케이션 전반에 누적되면 번거로워질 수 있습니다.

이 때문에 router level에서 prefetching을 통합하는 방식이 매력적입니다.

이 접근 방식에서는 route별로 해당 component tree에 필요한 데이터를 미리 명시합니다.

Server Rendering에서는 전통적으로 rendering을 시작하기 전에 모든 데이터가 load되어야 했기 때문에, SSR 앱에서는 오랫동안 이 방식이 많이 사용되어 왔습니다.

여기서는 client side case에 집중합니다.

Router level에서 통합할 때는 다음 방식 중 선택할 수 있습니다.

- 모든 데이터가 준비될 때까지 route rendering을 block
- prefetch를 시작하되 결과를 기다리지 않음
  - 이 경우 route rendering을 가능한 빨리 시작할 수 있습니다.
- 두 방식을 혼합
  - 중요한 데이터는 기다립니다.
  - secondary data는 완료되기 전에 rendering을 시작할 수 있습니다.

아래 예시는 `/article` route를 구성합니다.

- article data가 load될 때까지 route를 렌더링하지 않습니다.
- comments는 가능한 빨리 prefetch합니다.
- comments가 아직 완료되지 않았더라도 route rendering은 block하지 않습니다.

```tsx
const queryClient = new QueryClient()

const routerContext = new RouterContext()

const rootRoute = routerContext.createRootRoute({
  component: () => { ... },
})

const articleRoute = new Route({
  getParentRoute: () => rootRoute,
  path: 'article',
  beforeLoad: () => {
    return {
      articleQueryOptions: { queryKey: ['article'], queryFn: fetchArticle },
      commentsQueryOptions: { queryKey: ['comments'], queryFn: fetchComments },
    }
  },
  loader: async ({
    context: { queryClient },
    routeContext: { articleQueryOptions, commentsQueryOptions },
  }) => {
    // Fetch comments asap, but don't block
    queryClient.prefetchQuery(commentsQueryOptions)

    // Don't render the route at all until article has been fetched
    await queryClient.prefetchQuery(articleQueryOptions)
  },
  component: ({ useRouteContext }) => {
    const { articleQueryOptions, commentsQueryOptions } = useRouteContext()

    const articleQuery = useQuery(articleQueryOptions)
    const commentsQuery = useQuery(commentsQueryOptions)

    return (
      ...
    )
  },
  errorComponent: () => 'Oh crap!',
})
```

다른 router와의 integration도 가능합니다.

## 5. Manually Priming a Query

Query에 필요한 데이터를 이미 동기적으로 가지고 있다면 prefetch할 필요가 없습니다.

이 경우 Query Client의 `setQueryData` method를 사용해 query의 cached result를 key 기준으로 직접 추가하거나 업데이트할 수 있습니다.

```tsx
queryClient.setQueryData(['todos'], todos)
```
