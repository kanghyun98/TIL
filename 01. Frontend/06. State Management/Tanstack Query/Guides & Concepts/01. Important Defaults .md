# Important Defaults

TanStack Query는 기본적으로 공격적이지만 합리적인 기본값을 제공합니다. 이 기본값은 처음에는 예상 밖으로 느껴질 수 있으므로, 학습과 디버깅을 위해 아래 내용을 알고 있는 것이 중요합니다.

- `useQuery` 또는 `useInfiniteQuery`의 Query 인스턴스는 기본적으로 캐시된 데이터를 stale 상태로 간주합니다.

  - 이 동작은 `staleTime` 옵션으로 전역 또는 쿼리별로 변경할 수 있습니다. `staleTime`이 길수록 refetch 빈도는 줄어듭니다.

- `staleTime`이 설정된 Query는 해당 시간이 지나기 전까지 fresh 상태로 간주됩니다.

  - `staleTime: 2 * 60 * 1000`
    - 2분 동안, 또는 Query가 수동으로 무효화될 때까지 refetch 없이 캐시 데이터를 사용합니다.
  - `staleTime: Infinity`
    - Query가 수동으로 무효화될 때까지 refetch가 발생하지 않습니다.
  - `staleTime: 'static'`
    - Query가 수동으로 무효화되더라도 refetch가 발생하지 않습니다.

  > `'static'`과 `Infinity`는 모두 stale 기반 refetch를 막지만, `'static'`이 더 엄격합니다.
  >
  > - `staleTime: Infinity`인 쿼리는 `queryClient.invalidateQueries()`로 무효화할 수 있습니다.
  > - `staleTime: 'static'`은 `queryClient.invalidateQueries()`의 영향을 받지 않습니다.
  > - `refetchOnMount`, `refetchOnWindowFocus`, `refetchOnReconnect`가 `"always"`여도 `'static'`에서는 refetch가 발생하지 않습니다.
  > - `'static'`은 앱이 실행되는 동안 바뀔 수 없는 데이터에 사용합니다.
  >   - 부팅 시 가져오는 feature flags
  >   - 로그인 시 불러오는 사용자 권한
  >   - 정적인 참조 테이블
  >
  > 수동 무효화를 유지하고 싶다면 `Infinity`를 사용합니다.

- stale 상태인 쿼리는 다음 시점에 백그라운드에서 자동 refetch 됩니다.

  - 새로운 쿼리 인스턴스가 마운트될 때
  - 윈도우가 다시 포커스될 때
  - 네트워크가 다시 연결될 때

  > 과도한 refetch를 줄이려면 `staleTime` 설정이 권장됩니다. 필요하면 `refetchOnMount`, `refetchOnWindowFocus`, `refetchOnReconnect`로 refetch 시점을 직접 조정할 수도 있습니다.

- 쿼리는 선택적으로 `refetchInterval`을 설정해 주기적으로 refetch 하도록 구성할 수 있으며, 이는 `staleTime`과는 독립적으로 동작합니다.

- 더 이상 활성 상태인 `useQuery`, `useInfiniteQuery`, 또는 query observer 인스턴스가 없는 Query 결과는 `"inactive"` 상태가 되며, 다시 사용될 수 있도록 캐시에 남아 있습니다.

- 기본적으로 `"inactive"` 쿼리는 5분 후 가비지 컬렉션됩니다.

  > 이 동작은 기본 `gcTime` 값을 변경해 조정할 수 있습니다.

- 실패한 쿼리는 UI에 에러를 표시하기 전에 기본적으로 지수 백오프 지연과 함께 조용히 3번 재시도됩니다.

  > 이 동작은 기본 `retry`와 `retryDelay` 옵션으로 변경할 수 있습니다.

- 기본적으로 Query 결과에는 structural sharing이 적용됩니다.

  - 데이터가 실제로 바뀌지 않았다면 참조를 유지합니다.
  - 이로 인해 `useMemo`, `useCallback` 관점에서 값 안정화에 도움이 됩니다.
  - 대부분의 경우 비활성화할 필요가 없으며, 추가 비용 없이 성능 향상에 도움이 됩니다.
  
  > Structural sharing은 JSON 호환 값에서만 동작합니다.
  >
  > - JSON과 호환되지 않는 값은 항상 변경된 것으로 간주됩니다.
  > - 큰 응답 때문에 성능 문제가 있다면 `config.structuralSharing` 플래그로 비활성화할 수 있습니다.
  > - JSON과 호환되지 않는 값을 응답으로 사용하면서도 변경 감지를 원한다면, `config.structuralSharing`에 사용자 정의 함수를 제공해 이전 응답과 새 응답으로부터 값을 계산하고 필요한 참조를 유지할 수 있습니다.
