# Testing

React Query는 hook으로 동작합니다.

- React Query가 제공하는 hook
- React Query hook을 감싼 custom hook

React 17 이하에서는 custom hook의 unit test를 `@testing-library/react-hooks`로 작성할 수 있습니다.

설치 명령어는 다음과 같습니다.

```sh
npm install @testing-library/react-hooks react-test-renderer --save-dev
```

`react-test-renderer`는 `@testing-library/react-hooks`의 peer dependency입니다.

- 사용 중인 React 버전에 맞는 버전이 필요합니다.
- React 18 이상에서는 `renderHook`을 `@testing-library/react`에서 직접 사용할 수 있습니다.
- React 18 이상에서는 `@testing-library/react-hooks`가 더 이상 필요하지 않습니다.

<br>

## 1. Our First Test

설치가 끝나면 간단한 테스트를 작성할 수 있습니다.

다음 custom hook이 있다고 가정합니다.

```tsx
export function useCustomHook() {
  return useQuery({ queryKey: ['customHook'], queryFn: () => 'Hello' })
}
```

테스트는 다음과 같이 작성할 수 있습니다.

```tsx
import { renderHook, waitFor } from '@testing-library/react'

const queryClient = new QueryClient()

const wrapper = ({ children }) => (
  <QueryClientProvider client={queryClient}>{children}</QueryClientProvider>
)

const { result } = renderHook(() => useCustomHook(), { wrapper })

await waitFor(() => expect(result.current.isSuccess).toBe(true))

expect(result.current.data).toEqual('Hello')
```

이 테스트에서는 custom wrapper를 제공합니다.

- wrapper는 `QueryClient`와 `QueryClientProvider`를 구성합니다.
- 이를 통해 테스트가 다른 테스트와 완전히 격리되도록 돕습니다.

wrapper를 한 번만 작성하는 것도 가능합니다.

단, 이 경우 다음 조건을 지켜야 합니다.

- 각 테스트 전에 `QueryClient`를 clear해야 합니다.
- 테스트가 병렬로 실행되지 않도록 해야 합니다.
- 그렇지 않으면 한 테스트가 다른 테스트 결과에 영향을 줄 수 있습니다.

<br>

## 2. Turn off retries

React Query는 기본적으로 exponential backoff와 함께 3번 retry합니다.

오류 query를 테스트할 때 이 기본 동작 때문에 테스트가 timeout될 수 있습니다.

retry를 끄는 가장 쉬운 방법은 `QueryClientProvider`를 통해 설정하는 것입니다.

```tsx
const queryClient = new QueryClient({
  defaultOptions: {
    queries: {
      // ✅ turns retries off
      retry: false,
    },
  },
})

const wrapper = ({ children }) => (
  <QueryClientProvider client={queryClient}>{children}</QueryClientProvider>
)
```

이 설정은 component tree 안의 모든 query 기본값을 `no retries`로 설정합니다.

단, 이 설정은 실제 `useQuery`에 명시적인 retry 설정이 없을 때만 동작합니다.

- 특정 query가 `5 retries`를 명시하면 해당 설정이 우선합니다.
- default 값은 fallback으로만 사용됩니다.

<br>

## 3. Set gcTime to Infinity with Jest

Jest를 사용하는 경우 `gcTime`을 `Infinity`로 설정할 수 있습니다.

이 설정은 다음 오류 메시지를 방지하는 데 사용할 수 있습니다.

```txt
Jest did not exit one second after the test run completed
```

이 동작은 server에서 기본값입니다.

따라서 `gcTime`을 명시적으로 설정한 경우에만 필요합니다.

<br>

## 4. Testing Network Calls

React Query의 주요 용도는 network request를 cache하는 것입니다.

따라서 코드가 올바른 network request를 수행하는지 테스트할 수 있어야 합니다.

이를 테스트하는 방법은 여러 가지가 있습니다.

이 예시에서는 `nock`을 사용합니다.

다음 custom hook이 있다고 가정합니다.

```tsx
function useFetchData() {
  return useQuery({
    queryKey: ['fetchData'],
    queryFn: () => request('/api/data'),
  })
}
```

테스트는 다음과 같이 작성할 수 있습니다.

```tsx
const queryClient = new QueryClient()

const wrapper = ({ children }) => (
  <QueryClientProvider client={queryClient}>{children}</QueryClientProvider>
)

const expectation = nock('example-origin').get('/api/data').reply(200, {
  answer: 42,
})

const { result } = renderHook(() => useFetchData(), { wrapper })

await waitFor(() => expect(result.current.isSuccess).toBe(true))

expect(result.current.data).toEqual({ answer: 42 })
```

여기서는 `waitFor`를 사용합니다.

- query status가 성공 상태가 될 때까지 기다립니다.
- 이를 통해 hook 실행이 완료되었고 올바른 data를 가졌다는 것을 확인할 수 있습니다.

React 18을 사용하는 경우 `waitFor`의 semantics가 변경되었습니다.

## 5. Testing Load More / Infinite Scroll

먼저 API response를 mock합니다.

```tsx
function generateMockedResponse(page) {
  return {
    page: page,
    items: [...]
  }
}
```

그다음 `nock` 설정에서 page에 따라 응답을 다르게 처리해야 합니다.

이를 위해 `uri`를 사용합니다.

`uri` 값은 다음과 같은 형태입니다.

```txt
/?page=1
/?page=2
```

```tsx
const expectation = nock('example-origin')
  .persist()
  .query(true)
  .get('/api/data')
  .reply(200, (uri) => {
    const url = new URL(`example-origin${uri}`)
    const { page } = Object.fromEntries(url.searchParams)

    return generateMockedResponse(page)
  })
```

`.persist()`를 사용하는 이유는 같은 endpoint를 여러 번 호출하기 때문입니다.

이제 테스트를 안전하게 실행할 수 있습니다.

핵심은 data assertion이 통과할 때까지 기다리는 것입니다.

```tsx
const { result } = renderHook(() => useInfiniteQueryCustomHook(), {
  wrapper,
})

await waitFor(() => expect(result.current.isSuccess).toBe(true))

expect(result.current.data.pages).toStrictEqual(generateMockedResponse(1))

result.current.fetchNextPage()

await waitFor(() =>
  expect(result.current.data.pages).toStrictEqual([
    ...generateMockedResponse(1),
    ...generateMockedResponse(2),
  ]),
)

expectation.done()
```

React 18을 사용하는 경우 `waitFor`의 semantics가 변경되었습니다.
