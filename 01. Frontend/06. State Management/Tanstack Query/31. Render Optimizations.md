# Render Optimizations

React Query는 컴포넌트가 실제로 필요할 때만 다시 렌더링되도록 몇 가지 최적화를 자동으로 적용합니다.

이 최적화는 다음 방식으로 동작합니다.

<br>

## 1. structural sharing

React Query는 `structural sharing`이라는 기법을 사용합니다.

이 기법은 리렌더링 사이에서 가능한 많은 참조를 그대로 유지합니다.

네트워크에서 데이터를 가져오면 일반적으로 JSON 파싱으로 인해 완전히 새로운 참조가 만들어집니다.

하지만 React Query는 다음처럼 동작합니다.

- 데이터가 변경되지 않았다면 기존 참조를 유지합니다.
- 일부 데이터만 변경되었다면 변경되지 않은 부분은 그대로 유지합니다.
- 변경된 부분만 새 값으로 교체합니다.

> Note: 이 최적화는 `queryFn`이 JSON 호환 데이터를 반환할 때만 동작합니다.
>
> 다음 방식으로 설정할 수 있습니다.
>
> - 전역 또는 쿼리 단위로 `structuralSharing: false`를 설정해 비활성화
> - `structuralSharing`에 함수를 전달해 직접 구현

<br>

### 1.1 referential identity

`useQuery`, `useInfiniteQuery`, `useMutation`이 반환하는 최상위 객체는 참조적으로 안정적이지 않습니다.

`useQueries`가 반환하는 배열도 마찬가지입니다.

즉, 매 렌더링마다 새로운 참조가 됩니다.

하지만 이 훅들이 반환하는 `data` 프로퍼티는 가능한 한 안정적인 참조를 유지합니다.

<br>

## 2. tracked properties

React Query는 `useQuery`가 반환한 프로퍼티 중 실제로 사용된 프로퍼티가 변경될 때만 리렌더링을 발생시킵니다.

이 동작은 `Proxy` 객체를 사용해 구현됩니다.

이를 통해 불필요한 리렌더링을 줄일 수 있습니다.

예를 들어 다음 값들은 자주 변경될 수 있습니다.

- `isFetching`
- `isStale`

하지만 컴포넌트에서 해당 값을 사용하지 않는다면, 이 값이 변경되어도 리렌더링되지 않습니다.

이 기능은 다음 방식으로 설정할 수 있습니다.

- 전역 또는 쿼리 단위로 `notifyOnChangeProps`를 직접 설정
- `notifyOnChangeProps: 'all'`을 설정해 기능 비활성화

> Note: Proxy의 `get` trap은 프로퍼티에 접근할 때 실행됩니다.
>
> 프로퍼티 접근 방식은 다음을 포함합니다.
>
> - 구조 분해 할당
> - 직접 접근
>
> 단, object rest destructuring을 사용하면 이 최적화가 비활성화됩니다.
>
> 이 문제를 방지하기 위한 lint rule이 제공됩니다.

<br>

## 3. select

`select` 옵션을 사용하면 컴포넌트가 구독해야 하는 데이터의 일부만 선택할 수 있습니다.

이 기능은 다음 경우에 유용합니다.

- 고도로 최적화된 데이터 변환
- 불필요한 리렌더링 방지

```ts
export const useTodos = (select) => {
  return useQuery({
    queryKey: ['todos'],
    queryFn: fetchTodos,
    select,
  })
}

export const useTodoCount = () => {
  return useTodos((data) => data.length)
}
```

`useTodoCount` 커스텀 훅을 사용하는 컴포넌트는 `todos`의 길이가 변경될 때만 리렌더링됩니다.

예를 들어 todo의 이름만 변경되는 경우에는 리렌더링되지 않습니다.

> Note: `select`는 성공적으로 캐시된 데이터에 대해 동작합니다.
>
> 따라서 `select`는 에러를 던지기에 적절한 위치가 아닙니다.
>
> 에러의 source of truth는 `queryFn`입니다.
>
> `select` 함수가 에러를 반환하면 다음 상태가 됩니다.
>
> - `data`는 `undefined`
> - `isSuccess`는 `true`
>
> 잘못된 데이터로 인해 쿼리를 실패 처리하고 싶다면 `queryFn`에서 에러를 처리하는 것을 권장합니다.
>
> 캐싱과 관련 없는 에러 케이스라면 쿼리 훅 외부에서 처리하는 것을 권장합니다.

<br>

### 3.1 memoization

`select` 함수는 다음 경우에만 다시 실행됩니다.

- `select` 함수 자체의 참조가 변경된 경우
- `data`가 변경된 경우

위 예시처럼 인라인으로 `select` 함수를 작성하면 매 렌더링마다 실행됩니다.

이를 피하려면 다음 중 하나를 사용할 수 있습니다.

- `useCallback`으로 `select` 함수 감싸기
- 의존성이 없다면 안정적인 함수 참조로 추출하기

```ts
// wrapped in useCallback
export const useTodoCount = () => {
  return useTodos(useCallback((data) => data.length, []))
}
```

```ts
// extracted to a stable function reference
const selectTodoCount = (data) => data.length

export const useTodoCount = () => {
  return useTodos(selectTodoCount)
}
```
