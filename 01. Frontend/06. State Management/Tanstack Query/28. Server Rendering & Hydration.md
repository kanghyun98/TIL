# Server Rendering & Hydration

이 글은 React Query를 서버 렌더링과 함께 사용하는 방법을 정리합니다.

서버 렌더링을 이해하기 전에 다음 개념을 함께 보면 좋습니다.

- Prefetching & Router Integration
- Performance & Request Waterfalls
- Dependent Queries & Code Splitting
- Advanced Server Rendering
  - streaming
  - Server Components
  - Next.js app router

코드만 빠르게 보고 싶다면 다음 예제를 보면 됩니다.

- Full Next.js pages router example
- Full Remix example

<br>

## 1. Server Rendering & React Query

서버 렌더링은 서버에서 초기 HTML을 생성하는 작업입니다.

사용자는 페이지가 로드되는 즉시 콘텐츠를 볼 수 있습니다.

서버 렌더링은 다음 방식으로 발생할 수 있습니다.

- 요청 시점에 페이지를 생성하는 SSR
- 이전 요청 결과를 캐싱한 결과
- 빌드 시점에 미리 생성하는 SSG

클라이언트 렌더링 애플리케이션에서는 화면에 콘텐츠가 표시되기까지 최소 3번의 서버 왕복이 필요합니다.

```plaintext
1. |-> Markup (without content)
2.   |-> JS
3.     |-> Query
```

서버 렌더링은 이 흐름을 다음처럼 바꿉니다.

```plaintext
1. |-> Markup (with content AND initial data)
2.   |-> JS
```

1번이 완료되면 사용자는 콘텐츠를 볼 수 있습니다.

2번이 완료되면 페이지는 상호작용 가능한 상태가 됩니다.

Markup 안에 초기 데이터가 포함되어 있기 때문에, 최소한 데이터를 재검증하기 전까지는 클라이언트에서 3번 Query 요청을 실행할 필요가 없습니다.

클라이언트 관점에서는 이렇게 동작합니다.

서버 관점에서는 다음 단계가 필요합니다.

- Markup을 생성하기 전에 데이터를 prefetch
- 데이터를 직렬화 가능한 형태로 dehydrate
- Markup에 데이터를 포함
- 클라이언트에서 데이터를 React Query cache로 hydrate
- 클라이언트에서 새로운 fetch를 피함

<br>

## 2. A quick note on Suspense

이 가이드는 일반 `useQuery` API를 사용합니다.

권장하는 방식은 아니지만, 모든 query를 항상 prefetch한다면 `useSuspenseQuery`로 대체할 수 있습니다.

`useSuspenseQuery`를 사용하면 클라이언트 loading state에 `<Suspense>`를 사용할 수 있습니다.

단, `useSuspenseQuery`를 사용하면서 query prefetch를 빠뜨리면 프레임워크에 따라 결과가 달라집니다.

일부 경우에는 다음 문제가 발생할 수 있습니다.

- 데이터가 서버에서 suspend됨
- 서버에서 fetch됨
- 클라이언트로 hydrate되지 않음
- 클라이언트에서 다시 fetch됨
- 서버와 클라이언트가 서로 다른 결과를 렌더링함
- markup hydration mismatch가 발생함

<br>

## 3. Initial setup

React Query를 사용할 때 첫 단계는 항상 다음과 같습니다.

- `queryClient` 생성
- 애플리케이션을 `<QueryClientProvider>`로 감싸기

서버 렌더링에서는 `queryClient` 인스턴스를 애플리케이션 내부에서 생성하는 것이 중요합니다.

React state 안에서 생성하거나 instance ref를 사용해도 됩니다.

이 방식은 다음을 보장합니다.

- 사용자와 요청 간 데이터가 공유되지 않음
- 컴포넌트 생명주기 동안 `queryClient`를 한 번만 생성함

Next.js pages router 예시입니다.

```tsx
// _app.tsx
import { QueryClient, QueryClientProvider } from '@tanstack/react-query'

// NEVER DO THIS:
// const queryClient = new QueryClient()
//
// Creating the queryClient at the file root level makes the cache shared
// between all requests and means _all_ data gets passed to _all_ users.
// Besides being bad for performance, this also leaks any sensitive data.

export default function MyApp({ Component, pageProps }) {
  // Instead do this, which ensures each request has its own cache:
  const [queryClient] = React.useState(
    () =>
      new QueryClient({
        defaultOptions: {
          queries: {
            // With SSR, we usually want to set some default staleTime
            // above 0 to avoid refetching immediately on the client
            staleTime: 60 * 1000,
          },
        },
      }),
  )

  return (
    <QueryClientProvider client={queryClient}>
      <Component {...pageProps} />
    </QueryClientProvider>
  )
}
```

Remix 예시입니다.

```tsx
// app/root.tsx
import { Outlet } from '@remix-run/react'
import { QueryClient, QueryClientProvider } from '@tanstack/react-query'

export default function MyApp() {
  const [queryClient] = React.useState(
    () =>
      new QueryClient({
        defaultOptions: {
          queries: {
            // With SSR, we usually want to set some default staleTime
            // above 0 to avoid refetching immediately on the client
            staleTime: 60 * 1000,
          },
        },
      }),
  )

  return (
    <QueryClientProvider client={queryClient}>
      <Outlet />
    </QueryClientProvider>
  )
}
```

## 4. Get started fast with initialData

가장 빠른 시작 방법은 prefetch 단계에서 React Query를 사용하지 않는 것입니다.

이 방식은 `dehydrate` / `hydrate` API도 사용하지 않습니다.

대신 서버에서 받은 raw data를 `useQuery`의 `initialData` 옵션으로 전달합니다.

Next.js pages router에서 `getServerSideProps`를 사용하는 예시입니다.

```tsx
export async function getServerSideProps() {
  const posts = await getPosts()

  return { props: { posts } }
}

function Posts(props) {
  const { data } = useQuery({
    queryKey: ['posts'],
    queryFn: getPosts,
    initialData: props.posts,
  })

  // ...
}
```

이 방식은 다음에서도 사용할 수 있습니다.

- `getStaticProps`
- 이전 방식의 `getInitialProps`
- 동일한 역할을 하는 다른 프레임워크의 함수

Remix 예시입니다.

```tsx
export async function loader() {
  const posts = await getPosts()

  return json({ posts })
}

function Posts() {
  const { posts } = useLoaderData<typeof loader>()

  const { data } = useQuery({
    queryKey: ['posts'],
    queryFn: getPosts,
    initialData: posts,
  })

  // ...
}
```

설정은 최소화됩니다.

일부 경우에는 빠른 해결책이 될 수 있습니다.

하지만 전체 hydration 방식과 비교하면 몇 가지 tradeoff가 있습니다.

- 트리 깊은 곳의 컴포넌트에서 `useQuery`를 호출하면 해당 위치까지 `initialData`를 전달해야 합니다.
- 같은 query를 여러 위치에서 호출할 때 한 곳에만 `initialData`를 전달하면 취약할 수 있습니다.
  - `initialData`를 가진 컴포넌트를 제거하거나 이동하면 더 깊은 곳의 `useQuery`는 데이터를 받지 못할 수 있습니다.
  - 필요한 모든 query에 `initialData`를 전달하는 것도 번거로울 수 있습니다.
- 서버에서 query가 언제 fetch되었는지 알 수 없습니다.
  - `dataUpdatedAt`과 refetch 필요 여부 판단은 페이지가 로드된 시점을 기준으로 합니다.
- query cache에 이미 데이터가 있으면 `initialData`는 기존 데이터를 덮어쓰지 않습니다.
  - 새 데이터가 기존 데이터보다 최신이어도 덮어쓰지 않습니다.
  - `getServerSideProps` 예시에서 페이지를 여러 번 오가면 `getServerSideProps`는 매번 호출되고 새 데이터를 fetch합니다.
  - 하지만 `initialData` 옵션을 사용하기 때문에 클라이언트 cache와 데이터는 업데이트되지 않습니다.

전체 hydration 솔루션은 설정이 간단하고 위 단점이 없습니다.

이후 내용은 전체 hydration 방식에 집중합니다.

## 5. Using the Hydration APIs

조금 더 설정하면 `queryClient`를 사용해 preload 단계에서 query를 prefetch할 수 있습니다.

그리고 해당 `queryClient`의 직렬화된 버전을 앱의 렌더링 영역에 전달해 재사용할 수 있습니다.

이 방식은 `initialData` 방식의 단점을 피합니다.

일반적인 추가 단계는 다음과 같습니다.

- 프레임워크 loader 함수에서 `const queryClient = new QueryClient(options)` 생성
- loader 함수에서 prefetch할 각 query마다 `await queryClient.prefetchQuery(...)` 실행
  - 가능하면 `await Promise.all(...)`을 사용해 query를 병렬로 fetch
  - prefetch하지 않는 query가 있어도 괜찮음
    - 이 query는 서버 렌더링되지 않음
    - 애플리케이션이 interactive 상태가 된 뒤 클라이언트에서 fetch됨
    - 사용자 상호작용 후에만 보이는 콘텐츠나 페이지 아래쪽 콘텐츠에 유용함
    - 중요한 콘텐츠의 blocking을 피할 수 있음
- loader에서 `dehydrate(queryClient)` 반환
  - 반환 문법은 프레임워크마다 다름
- 트리를 `<HydrationBoundary state={dehydratedState}>`로 감싸기
  - `dehydratedState`는 프레임워크 loader에서 가져옴
  - 가져오는 방식은 프레임워크마다 다름
  - 각 route에서 처리할 수도 있음
  - boilerplate를 줄이기 위해 애플리케이션 상단에서 처리할 수도 있음

중요한 세부 사항은 실제로 3개의 `queryClient`가 관여한다는 점입니다.

- 프레임워크 loader는 렌더링 전 preload 단계입니다.
- preload 단계는 prefetch를 수행하는 자체 `queryClient`를 가집니다.
- 이 단계의 dehydrated 결과는 서버 렌더링 프로세스와 클라이언트 렌더링 프로세스 모두에 전달됩니다.
- 서버 렌더링과 클라이언트 렌더링은 각각 자체 `queryClient`를 가집니다.
- 둘은 같은 데이터로 시작하기 때문에 같은 markup을 반환할 수 있습니다.

Server Components도 preload 단계의 또 다른 형태입니다.

Server Components는 React component tree의 일부를 preload, 즉 pre-render할 수 있습니다.

### 5.1 Full Next.js pages router example

App router 문서는 Advanced Server Rendering에서 다룹니다.

초기 설정입니다.

```tsx
// _app.tsx
import { QueryClient, QueryClientProvider } from '@tanstack/react-query'

export default function MyApp({ Component, pageProps }) {
  const [queryClient] = React.useState(
    () =>
      new QueryClient({
        defaultOptions: {
          queries: {
            // With SSR, we usually want to set some default staleTime
            // above 0 to avoid refetching immediately on the client
            staleTime: 60 * 1000,
          },
        },
      }),
  )

  return (
    <QueryClientProvider client={queryClient}>
      <Component {...pageProps} />
    </QueryClientProvider>
  )
}
```

각 route에서는 다음처럼 구성합니다.

```tsx
// pages/posts.tsx
import {
  dehydrate,
  HydrationBoundary,
  QueryClient,
  useQuery,
} from '@tanstack/react-query'

// This could also be getServerSideProps
export async function getStaticProps() {
  const queryClient = new QueryClient()

  await queryClient.prefetchQuery({
    queryKey: ['posts'],
    queryFn: getPosts,
  })

  return {
    props: {
      dehydratedState: dehydrate(queryClient),
    },
  }
}

function Posts() {
  // This useQuery could just as well happen in some deeper child to
  // the <PostsRoute>, data will be available immediately either way
  const { data } = useQuery({ queryKey: ['posts'], queryFn: getPosts })

  // This query was not prefetched on the server and will not start
  // fetching until on the client, both patterns are fine to mix
  const { data: commentsData } = useQuery({
    queryKey: ['posts-comments'],
    queryFn: getComments,
  })

  // ...
}

export default function PostsRoute({ dehydratedState }) {
  return (
    <HydrationBoundary state={dehydratedState}>
      <Posts />
    </HydrationBoundary>
  )
}
```

초기 설정입니다.

import { QueryClient, QueryClientProvider } from '@tanstack/react-query'

각 route에서는 다음처럼 구성합니다.

Nested routes에서도 사용할 수 있습니다.

```tsx
// app/routes/posts.tsx
import { json } from '@remix-run/node'
import {
  dehydrate,
  HydrationBoundary,
  QueryClient,
  useQuery,
} from '@tanstack/react-query'

export async function loader() {
  const queryClient = new QueryClient()

  await queryClient.prefetchQuery({
    queryKey: ['posts'],
    queryFn: getPosts,
  })

  return json({ dehydratedState: dehydrate(queryClient) })
}

function Posts() {
  // This useQuery could just as well happen in some deeper child to
  // the <PostsRoute>, data will be available immediately either way
  const { data } = useQuery({ queryKey: ['posts'], queryFn: getPosts })

  // This query was not prefetched on the server and will not start
  // fetching until on the client, both patterns are fine to mix
  const { data: commentsData } = useQuery({
    queryKey: ['posts-comments'],
    queryFn: getComments,
  })

  // ...
}

export default function PostsRoute() {
  const { dehydratedState } = useLoaderData<typeof loader>()

  return (
    <HydrationBoundary state={dehydratedState}>
      <Posts />
    </HydrationBoundary>
  )
}
```

## 6. Optional - Remove boilerplate

모든 route에 다음 코드를 넣는 것은 boilerplate가 많아 보일 수 있습니다.

```tsx
export default function PostsRoute({ dehydratedState }) {
  return (
    <HydrationBoundary state={dehydratedState}>
      <Posts />
    </HydrationBoundary>
  )
}
```

이 방식에도 문제는 없습니다.

하지만 boilerplate를 제거하고 싶다면 Next.js에서 다음처럼 설정을 변경할 수 있습니다.

```tsx
// _app.tsx
import {
  HydrationBoundary,
  QueryClient,
  QueryClientProvider,
} from '@tanstack/react-query'

export default function MyApp({ Component, pageProps }) {
  const [queryClient] = React.useState(() => new QueryClient())

  return (
    <QueryClientProvider client={queryClient}>
      <HydrationBoundary state={pageProps.dehydratedState}>
        <Component {...pageProps} />
      </HydrationBoundary>
    </QueryClientProvider>
  )
}

// pages/posts.tsx
// Remove PostsRoute with the HydrationBoundary and instead export Posts directly:
export default function Posts() { ... }
```

Remix에서는 이 방식이 조금 더 복잡합니다.

`use-dehydrated-state` 패키지를 확인하는 것을 권장합니다.

## 7. Prefetching dependent queries

framework loader에서 dependent query를 prefetch하는 방법입니다.

다음 코드는 Dependent Queries 예시입니다.

```tsx
// Get the user
const { data: user } = useQuery({
  queryKey: ['user', email],
  queryFn: getUserByEmail,
})

const userId = user?.id

// Then get the user's projects
const {
  status,
  fetchStatus,
  data: projects,
} = useQuery({
  queryKey: ['projects', userId],
  queryFn: getProjectsByUser,
  // The query will not execute until the userId exists
  enabled: !!userId,
})
```

이 query를 서버 렌더링할 수 있도록 prefetch하려면 다음처럼 작성합니다.

```tsx
// For Remix, rename this to loader instead
export async function getServerSideProps() {
  const queryClient = new QueryClient()

  const user = await queryClient.fetchQuery({
    queryKey: ['user', email],
    queryFn: getUserByEmail,
  })

  if (user?.userId) {
    await queryClient.prefetchQuery({
      queryKey: ['projects', userId],
      queryFn: getProjectsByUser,
    })
  }

  // For Remix:
  // return json({ dehydratedState: dehydrate(queryClient) })
  return { props: { dehydratedState: dehydrate(queryClient) } }
}
```

이 로직은 더 복잡해질 수 있습니다.

loader 함수는 JavaScript이므로 언어의 기능을 사용해 로직을 구성할 수 있습니다.

서버 렌더링하려는 모든 query는 prefetch해야 합니다.

## 8. Error handling

React Query는 기본적으로 graceful degradation 전략을 사용합니다.

기본 동작은 다음과 같습니다.

- `queryClient.prefetchQuery(...)`는 error를 throw하지 않습니다.
- `dehydrate(...)`는 성공한 query만 포함합니다.
- 실패한 query는 포함하지 않습니다.

이 결과는 다음과 같습니다.

- 실패한 query는 클라이언트에서 다시 시도됩니다.
- 서버 렌더링 결과에는 전체 콘텐츠 대신 loading state가 포함됩니다.

이 기본값은 좋은 기본 동작입니다.

하지만 항상 원하는 동작은 아닐 수 있습니다.

중요한 콘텐츠가 누락된 경우에는 상황에 따라 404 또는 500 status code로 응답하고 싶을 수 있습니다.

이 경우 `queryClient.fetchQuery(...)`를 사용합니다.

`fetchQuery`는 실패 시 error를 throw하므로 적절한 방식으로 처리할 수 있습니다.

```tsx
let result

try {
  result = await queryClient.fetchQuery(...)
} catch (error) {
  // Handle the error, refer to your framework documentation
}

// You might also want to check and handle any invalid `result` here
```

실패한 query를 dehydrated state에 포함해 retry를 피하고 싶다면 `shouldDehydrateQuery` 옵션으로 기본 함수를 override할 수 있습니다.

```tsx
dehydrate(queryClient, {
  shouldDehydrateQuery: (query) => {
    // This will include all queries, including failed ones,
    // but you can also implement your own logic by inspecting `query`
    return true
  },
})
```

## 9. Serialization

Next.js에서 다음처럼 반환하거나,

```tsx
return { props: { dehydratedState: dehydrate(queryClient) } }
```

Remix에서 다음처럼 반환하면,

```tsx
return json({ dehydratedState: dehydrate(queryClient) })
```

프레임워크가 `queryClient`의 `dehydratedState` 표현을 직렬화합니다.

이 데이터는 markup에 포함되어 클라이언트로 전달됩니다.

기본적으로 이러한 프레임워크는 안전하게 serialize / parse할 수 있는 값만 반환할 수 있습니다.

따라서 다음 값은 지원하지 않습니다.

- `undefined`
- `Error`
- `Date`
- `Map`
- `Set`
- `BigInt`
- `Infinity`
- `NaN`
- `-0`
- regular expressions

즉, query에서도 이런 값을 반환할 수 없습니다.

이런 값을 반환해야 한다면 `superjson` 또는 유사 패키지를 확인할 수 있습니다.

Custom SSR setup을 사용한다면 이 단계를 직접 처리해야 합니다.

처음에는 `JSON.stringify(dehydratedState)`를 떠올릴 수 있습니다.

하지만 `JSON.stringify`는 기본적으로 다음과 같은 값을 escape하지 않습니다.

```html
<script>alert('Oh no..')</script>
```

따라서 애플리케이션에 XSS 취약점이 생길 수 있습니다.

`superjson`도 값을 escape하지 않습니다.

Custom SSR setup에서 단독으로 사용하면 안전하지 않습니다.

단, output을 escape하는 추가 단계를 넣는 경우는 제외됩니다.

대신 XSS injection에 기본적으로 안전한 라이브러리 사용을 권장합니다.

- Serialize JavaScript
- devalue

## 10. A note about request waterfalls

서버 렌더링이 복잡한 nested waterfall을 어떻게 바꾸는지 살펴봅니다.

예시는 다음 구조를 가정합니다.

- `<Feed>` 컴포넌트 안에 code split된 `<GraphFeedItem>` 컴포넌트가 있음
- `<GraphFeedItem>`은 feed에 graph item이 있을 때만 렌더링됨
- 두 컴포넌트 모두 자신의 데이터를 fetch함

클라이언트 렌더링에서는 다음 request waterfall이 발생합니다.

```plaintext
1. |> Markup (without content)
2.   |> JS for <Feed>
3.     |> getFeed()
4.       |> JS for <GraphFeedItem>
5.         |> getGraphDataById()
```

서버 렌더링을 사용하면 다음처럼 바꿀 수 있습니다.

```plaintext
1. |> Markup (with content AND initial data)
2.   |> JS for <Feed>
2.   |> JS for <GraphFeedItem>
```

query는 더 이상 클라이언트에서 fetch되지 않습니다.

대신 데이터가 markup에 포함됩니다.

`<GraphFeedItem>`이 서버에서 렌더링되었기 때문에 클라이언트에서도 이 JS가 필요하다는 것을 알 수 있습니다.

따라서 해당 chunk에 대한 script tag를 markup에 삽입할 수 있습니다.

그 결과 JS를 병렬로 로드할 수 있습니다.

하지만 서버에서는 여전히 다음 request waterfall이 존재합니다.

```plaintext
1. |> getFeed()
2.   |> getGraphDataById()
```

feed를 fetch하기 전에는 graph data가 필요한지 알 수 없습니다.

이 query들은 dependent query입니다.

다만 서버에서는 latency가 일반적으로 더 낮고 안정적이므로, 이 waterfall은 큰 문제가 아닐 수 있습니다.

서버 렌더링은 waterfall을 대부분 평탄화합니다.

하지만 주의할 점이 있습니다.

예를 들어 `/feed` 페이지와 `/posts` 페이지가 있다고 가정합니다.

브라우저 주소창에 직접 `/feed`를 입력하고 이동하면 서버 렌더링의 이점을 얻습니다.

하지만 `/posts`에 직접 진입한 뒤 링크를 클릭해 `/feed`로 이동하면 다음 흐름으로 돌아갑니다.

```plaintext
1. |> JS for <Feed>
2.   |> getFeed()
3.     |> JS for <GraphFeedItem>
4.       |> getGraphDataById()
```

SPA에서 서버 렌더링은 초기 페이지 로드에만 적용되고, 이후 navigation에는 적용되지 않기 때문입니다.

현대 프레임워크는 보통 초기 code와 data를 병렬로 fetch하여 이 문제를 해결하려고 합니다.

Next.js 또는 Remix에서 이 가이드의 prefetching 패턴을 사용하고 dependent query도 prefetch하면 다음처럼 동작합니다.

```plaintext
1. |> JS for <Feed>
1. |> getFeed() + getGraphDataById()
2.   |> JS for <GraphFeedItem>
```

이 방식은 훨씬 낫습니다.

더 개선하려면 Server Components를 사용해 단일 roundtrip으로 평탄화할 수 있습니다.

이 내용은 Advanced Server Rendering에서 다룹니다.

## 11. Tips, Tricks and Caveats

### 11.1 Staleness is measured from when the query was fetched on the server

query의 stale 여부는 `dataUpdatedAt` 시점을 기준으로 판단됩니다.

여기서 주의할 점은 서버 시간이 정확해야 한다는 것입니다.

UTC 시간이 사용되므로 timezone은 영향을 주지 않습니다.

`staleTime` 기본값은 `0`입니다.

따라서 query는 기본적으로 페이지 로드 시 background에서 refetch됩니다.

이중 fetch를 피하고 싶다면 더 높은 `staleTime`을 사용할 수 있습니다.

특히 markup을 cache하지 않는 경우에 유용합니다.

stale query의 refetch는 CDN에 markup을 cache할 때 잘 맞습니다.

예를 들어 다음처럼 설정할 수 있습니다.

- 페이지 자체의 cache time은 높게 설정
  - 서버에서 페이지를 다시 렌더링하는 일을 줄임
- query의 `staleTime`은 더 낮게 설정
  - 사용자가 페이지에 방문했을 때 데이터가 오래되었다면 background에서 refetch

예시입니다.

- 페이지는 1주일 cache
- 데이터가 하루보다 오래되면 페이지 로드 시 자동으로 refetch

<br>

### 11.2 High memory consumption on server

요청마다 `QueryClient`를 생성하면 React Query는 해당 client를 위한 독립 cache를 생성합니다.

이 cache는 `gcTime` 기간 동안 메모리에 유지됩니다.

해당 기간에 요청 수가 많으면 서버 메모리 사용량이 높아질 수 있습니다.

서버에서 `gcTime` 기본값은 `Infinity`입니다.

이 값은 manual garbage collection을 비활성화합니다.

요청이 끝나면 메모리는 자동으로 clear됩니다.

명시적으로 `Infinity`가 아닌 `gcTime`을 설정했다면 cache를 일찍 clear하는 책임은 사용자에게 있습니다.

`gcTime`을 `0`으로 설정하는 것은 피해야 합니다.

이 설정은 hydration error를 유발할 수 있습니다.

이유는 다음과 같습니다.

- `HydrationBoundary`가 렌더링에 필요한 데이터를 cache에 넣음
- garbage collector가 렌더링 완료 전에 데이터를 제거할 수 있음
- 이 경우 문제가 발생할 수 있음

더 짧은 `gcTime`이 필요하다면 `2 * 1000` 설정을 권장합니다.

이 정도 시간은 앱이 데이터를 참조하기에 충분합니다.

메모리 사용량을 낮추려면 요청이 처리되고 dehydrated state가 클라이언트로 전송된 뒤 `queryClient.clear()`를 호출할 수 있습니다.

또는 더 작은 `gcTime`을 설정할 수도 있습니다.

<br>

### 11.3 Caveat for Next.js rewrites

Next.js의 rewrites 기능을 Automatic Static Optimization 또는 `getStaticProps`와 함께 사용하면 주의할 점이 있습니다.

이 조합은 React Query에서 두 번째 hydration을 발생시킵니다.

Next.js는 클라이언트에서 rewrites를 parse하고 hydration 이후 params를 수집해야 합니다.

그래야 해당 params를 `router.query`에 제공할 수 있습니다.

그 결과 모든 hydration data의 referential equality가 사라집니다.

이 영향은 다음 위치에서 나타날 수 있습니다.

- 데이터를 component props로 사용하는 곳
- `useEffect` dependency array
- `useMemo` dependency array
