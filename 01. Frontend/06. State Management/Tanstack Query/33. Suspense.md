# Suspense

React Query는 React의 Suspense for Data Fetching API와 함께 사용할 수 있습니다.

- `useSuspenseQuery`
- `useSuspenseInfiniteQuery`
- `useSuspenseQueries`
- 추가로 `useQuery().promise`와 `React.use()`를 사용할 수 있습니다. 이 기능은 Experimental입니다.



Suspense 모드를 사용하면 `status` 상태와 `error` 객체가 필요하지 않습니다. 대신 다음 React 기능을 사용합니다.

- `React.Suspense`
  - `fallback` prop 사용
- React Error Boundary
  - 에러 처리



Mutation의 에러도 Query처럼 가장 가까운 Error Boundary로 전달하고 싶다면 `throwOnError` 옵션을 `true`로 설정할 수 있습니다.

Query에서 Suspense 모드를 활성화하는 예시는 다음과 같습니다.

```tsx
import { useSuspenseQuery } from '@tanstack/react-query'

const { data } = useSuspenseQuery({ queryKey, queryFn })
```

TypeScript에서 이 방식은 잘 동작합니다.

- `data`는 항상 정의되어 있음이 보장됩니다.
- 에러와 로딩 상태는 Suspense와 Error Boundary가 처리합니다.



반대로 다음 제약이 있습니다.

- Query를 조건부로 활성화하거나 비활성화할 수 없습니다.
- Suspense에서는 한 컴포넌트 안의 모든 Query가 순차적으로 fetch됩니다.
- 따라서 dependent Query에서는 일반적으로 조건부 활성화가 필요하지 않습니다.
- 이 Query에는 `placeholderData`도 존재하지 않습니다.

업데이트 중 QueryKey가 변경될 때 UI가 fallback으로 대체되는 것을 막으려면, QueryKey를 변경하는 업데이트를 `startTransition`으로 감싸야 합니다.



### 0.1 throwOnError default

기본적으로 모든 에러가 가장 가까운 Error Boundary로 전달되지는 않습니다.

React Query는 보여줄 데이터가 없을 때만 에러를 throw합니다.

즉, Query가 한 번이라도 성공해서 캐시에 데이터가 있다면 다음과 같이 동작합니다.

- 데이터가 stale 상태여도 컴포넌트를 렌더링합니다.
- 에러가 발생해도 Error Boundary로 바로 전달하지 않습니다.

따라서 `throwOnError`의 기본값은 다음과 같습니다.

```ts
throwOnError: (error, query) => typeof query.state.data === 'undefined'
```

`throwOnError`는 변경할 수 없습니다.

이 옵션을 변경하면 `data`가 잠재적으로 `undefined`가 될 수 있기 때문입니다.

모든 에러를 Error Boundary에서 처리하고 싶다면 직접 에러를 throw해야 합니다.

```tsx
import { useSuspenseQuery } from '@tanstack/react-query'

const { data, error, isFetching } = useSuspenseQuery({ queryKey, queryFn })

if (error && !isFetching) {
  throw error
}

// continue rendering data
```

## 1. Resetting Error Boundaries

Suspense를 사용하든, Query에서 `throwOnError`를 사용하든, 에러 발생 후 다시 렌더링할 때 Query에 재시도 의도를 알려줄 방법이 필요합니다.

Query 에러는 다음 방법으로 reset할 수 있습니다.

- `QueryErrorResetBoundary` 컴포넌트
- `useQueryErrorResetBoundary` Hook

`QueryErrorResetBoundary` 컴포넌트를 사용하면 해당 컴포넌트 경계 안의 Query 에러를 reset합니다.

```tsx
import { QueryErrorResetBoundary } from '@tanstack/react-query'
import { ErrorBoundary } from 'react-error-boundary'

const App = () => (
  <QueryErrorResetBoundary>
    {({ reset }) => (
      <ErrorBoundary
        onReset={reset}
        fallbackRender={({ resetErrorBoundary }) => (
          <div>
            There was an error!
            <Button onClick={() => resetErrorBoundary()}>Try again</Button>
          </div>
        )}
      >
        <Page />
      </ErrorBoundary>
    )}
  </QueryErrorResetBoundary>
)
```

`useQueryErrorResetBoundary` Hook을 사용하면 가장 가까운 `QueryErrorResetBoundary` 안의 Query 에러를 reset합니다.

가까운 boundary가 없다면 전역적으로 reset합니다.

```tsx
import { useQueryErrorResetBoundary } from '@tanstack/react-query'
import { ErrorBoundary } from 'react-error-boundary'

const App = () => {
  const { reset } = useQueryErrorResetBoundary()

  return (
    <ErrorBoundary
      onReset={reset}
      fallbackRender={({ resetErrorBoundary }) => (
        <div>
          There was an error!
          <Button onClick={() => resetErrorBoundary()}>Try again</Button>
        </div>
      )}
    >
      <Page />
    </ErrorBoundary>
  )
}
```

## 2. Fetch-on-render vs Render-as-you-fetch

React Query의 Suspense 모드는 기본적으로 추가 설정 없이 Fetch-on-render 방식으로 잘 동작합니다.

Fetch-on-render 방식에서는 다음 흐름으로 동작합니다.

- 컴포넌트가 mount를 시도합니다.
- Query fetching이 트리거됩니다.
- 컴포넌트가 suspend됩니다.
- 단, 컴포넌트를 import하고 mount한 이후에만 fetching이 시작됩니다.

Render-as-you-fetch 모델로 확장하려면 prefetching을 구현하는 것을 권장합니다.

prefetching은 다음 시점에 Query 로딩을 먼저 시작할 수 있습니다.

- routing callback
- 사용자 interaction event

이를 통해 Query를 다음보다 먼저 로딩할 수 있습니다.

- Query가 mount되기 전
- 부모 컴포넌트를 import하거나 mount하기 전

## 3. Suspense on the Server with streaming

Next.js를 사용한다면 Suspense on the Server를 위한 Experimental 통합 패키지를 사용할 수 있습니다.

패키지 이름은 다음과 같습니다.

- `@tanstack/react-query-next-experimental`

이 패키지를 사용하면 Client Component 안에서 `useSuspenseQuery`를 호출하는 것만으로 서버에서 데이터를 fetch할 수 있습니다.

결과는 Suspense Boundary가 resolve되는 시점에 서버에서 클라이언트로 stream됩니다.

이를 위해 앱을 `ReactQueryStreamedHydration` 컴포넌트로 감쌉니다.

```tsx
// app/providers.tsx
'use client'

import {
  isServer,
  QueryClient,
  QueryClientProvider,
} from '@tanstack/react-query'
import * as React from 'react'
import { ReactQueryStreamedHydration } from '@tanstack/react-query-next-experimental'

function makeQueryClient() {
  return new QueryClient({
    defaultOptions: {
      queries: {
        // With SSR, we usually want to set some default staleTime
        // above 0 to avoid refetching immediately on the client
        staleTime: 60 * 1000,
      },
    },
  })
}

let browserQueryClient: QueryClient | undefined = undefined

function getQueryClient() {
  if (isServer) {
    // Server: always make a new query client
    return makeQueryClient()
  } else {
    // Browser: make a new query client if we don't already have one
    // This is very important, so we don't re-make a new client if React
    // suspends during the initial render. This may not be needed if we
    // have a suspense boundary BELOW the creation of the query client
    if (!browserQueryClient) browserQueryClient = makeQueryClient()
    return browserQueryClient
  }
}

export function Providers(props: { children: React.ReactNode }) {
  // NOTE: Avoid useState when initializing the query client if you don't
  //       have a suspense boundary between this and the code that may
  //       suspend because React will throw away the client on the initial
  //       render if it suspends and there is no boundary
  const queryClient = getQueryClient()

  return (
    <QueryClientProvider client={queryClient}>
      <ReactQueryStreamedHydration>
        {props.children}
      </ReactQueryStreamedHydration>
    </QueryClientProvider>
  )
}
```
