# Mutations

`useMutation`은 query와 달리 데이터 생성/수정/삭제 또는 서버 사이드 이펙트 수행 작업에 사용됩니다.



새로운 todo를 서버에 추가하는 예시는 아래와 같습니다.

```tsx
function App() {
  const mutation = useMutation({
    mutationFn: (newTodo) => {
      return axios.post('/todos', newTodo)
    },
  })

  return (
    <div>
      {mutation.isPending ? (
        'Adding todo...'
      ) : (
        <>
          {mutation.isError ? (
            <div>An error occurred: {mutation.error.message}</div>
          ) : null}

          {mutation.isSuccess ? <div>Todo added!</div> : null}

          <button
            onClick={() => {
              mutation.mutate({ id: new Date(), title: 'Do Laundry' })
            }}
          >
            Create Todo
          </button>
        </>
      )}
    </div>
  )
}
```

<br>

mutation은 한 시점에 아래 상태 중 하나만 가질 수 있습니다.

- `isIdle` 또는 `status === 'idle'` : 현재 idle 상태이거나 fresh/reset 상태입니다.
- `isPending` 또는 `status === 'pending'` : 현재 실행 중입니다.
- `isError` 또는 `status === 'error'` : 에러가 발생했습니다.
- `isSuccess` 또는 `status === 'success'` : 성공했고 mutation data를 사용할 수 있습니다.



기본 상태 외에도 추가 정보가 제공됩니다.

- `error` : mutation이 error 상태이면 `error` 프로퍼티로 확인할 수 있습니다.
- `data` : mutation이 success 상태이면 `data` 프로퍼티로 확인할 수 있습니다.



위 예시처럼 `mutate` 함수에 단일 변수 또는 객체를 전달해 mutation function에 값을 넘길 수 있습니다.

변수만 전달하는 수준에서는 특별해 보이지 않을 수 있지만, `onSuccess` 옵션과 Query Client의 `invalidateQueries`, `setQueryData` 메서드를 함께 사용하면 mutation은 매우 강력한 도구가 됩니다.

> IMPORTANT: `mutate` 함수는 비동기 함수입니다. 따라서 React 16 이하에서는 이벤트 콜백에 직접 사용할 수 없습니다. `onSubmit`에서 이벤트에 접근해야 한다면 다른 함수로 감싸야 합니다. 이는 React event pooling 때문입니다.

```tsx
// This will not work in React 16 and earlier
const CreateTodo = () => {
  const mutation = useMutation({
    mutationFn: (event) => {
      event.preventDefault()
      return fetch('/api', new FormData(event.target))
    },
  })

  return <form onSubmit={mutation.mutate}>...</form>
}

// This will work
const CreateTodo = () => {
  const mutation = useMutation({
    mutationFn: (formData) => {
      return fetch('/api', formData)
    },
  })

  const onSubmit = (event) => {
    event.preventDefault()
    mutation.mutate(new FormData(event.target))
  }

  return <form onSubmit={onSubmit}>...</form>
}
```

## 1. Resetting Mutation State

mutation 요청의 `error`나 `data`를 지워야 할 때가 있습니다.

이 경우 `reset` 함수를 사용할 수 있습니다.

```tsx
const CreateTodo = () => {
  const [title, setTitle] = useState('')
  const mutation = useMutation({ mutationFn: createTodo })

  const onCreateTodo = (e) => {
    e.preventDefault()
    mutation.mutate({ title })
  }

  return (
    <form onSubmit={onCreateTodo}>
      {mutation.error && (
        <h5 onClick={() => mutation.reset()}>{mutation.error}</h5>
      )}

      <input
        type="text"
        value={title}
        onChange={(e) => setTitle(e.target.value)}
      />

      <br />
      <button type="submit">Create Todo</button>
    </form>
  )
}
```

## 2. Mutation Side Effects

`useMutation`은 mutation lifecycle의 각 단계에서 빠르게 사이드 이펙트를 처리할 수 있는 헬퍼 옵션을 제공합니다.

이 옵션은 아래 경우에 유용합니다.

- mutation 이후 query invalidation 및 refetch
- optimistic update

```tsx
useMutation({
  mutationFn: addTodo,
  onMutate: (variables, context) => {
    // A mutation is about to happen!
    // Optionally return a result containing data to use when for example rolling back
    return { id: 1 }
  },
  onError: (error, variables, onMutateResult, context) => {
    // An error happened!
    console.log(`rolling back optimistic update with id ${onMutateResult.id}`)
  },
  onSuccess: (data, variables, onMutateResult, context) => {
    // Boom baby!
  },
  onSettled: (data, error, variables, onMutateResult, context) => {
    // Error or success... doesn't matter!
  },
})
```

<br>

콜백 함수 중 하나에서 promise를 반환하면, 다음 콜백이 호출되기 전에 해당 promise를 먼저 기다립니다.

```tsx
useMutation({
  mutationFn: addTodo,
  onSuccess: async () => {
    console.log("I'm first!")
  },
  onSettled: async () => {
    console.log("I'm second!")
  },
})
```

<br>

`mutate`를 호출할 때 `useMutation`에 정의한 콜백 외에 추가 콜백을 실행하고 싶을 수 있습니다.

이 방식은 컴포넌트별 사이드 이펙트에 사용할 수 있습니다.

mutation variable 뒤에 동일한 콜백 옵션을 전달하면 됩니다.

지원되는 옵션은 아래와 같습니다.

- `onSuccess`
- `onError`
- `onSettled`

단, mutation이 끝나기 전에 컴포넌트가 unmount되면 이 추가 콜백은 실행되지 않습니다.

```tsx
useMutation({
  mutationFn: addTodo,
  onSuccess: (data, variables, onMutateResult, context) => {
    // I will fire first
  },
  onError: (error, variables, onMutateResult, context) => {
    // I will fire first
  },
  onSettled: (data, error, variables, onMutateResult, context) => {
    // I will fire first
  },
})

mutate(todo, {
  onSuccess: (data, variables, onMutateResult, context) => {
    // I will fire second!
  },
  onError: (error, variables, onMutateResult, context) => {
    // I will fire second!
  },
  onSettled: (data, error, variables, onMutateResult, context) => {
    // I will fire second!
  },
})
```

<br>

### 2.1 Consecutive mutations

연속된 mutation에서는 `onSuccess`, `onError`, `onSettled` 처리 방식에 약간의 차이가 있습니다.

`mutate` 함수에 직접 전달한 콜백은 아래 조건에서만 한 번 실행됩니다.

- 한 번만 실행됨
- 컴포넌트가 여전히 mounted 상태일 때만 실행됨

이유는 `mutate`가 호출될 때마다 mutation observer가 제거되고 다시 subscribe되기 때문입니다.

반대로 `useMutation`에 정의한 핸들러는 각 `mutate` 호출마다 실행됩니다.

> `useMutation`에 전달한 `mutationFn`은 대체로 비동기입니다. 이 경우 mutation이 완료되는 순서는 `mutate` 호출 순서와 다를 수 있습니다.

```tsx
useMutation({
  mutationFn: addTodo,
  onSuccess: (data, variables, onMutateResult, context) => {
    // Will be called 3 times
  },
})

const todos = ['Todo 1', 'Todo 2', 'Todo 3']

todos.forEach((todo) => {
  mutate(todo, {
    onSuccess: (data, variables, onMutateResult, context) => {
      // Will execute only once, for the last mutation (Todo 3),
      // regardless which mutation resolves first
    },
  })
})
```

## 3. Promises

`mutate` 대신 `mutateAsync`를 사용하면 success 시 resolve되고 error 시 throw되는 promise를 받을 수 있습니다.

이 방식은 사이드 이펙트를 조합할 때 사용할 수 있습니다.

```tsx
const mutation = useMutation({ mutationFn: addTodo })

try {
  const todo = await mutation.mutateAsync(todo)
  console.log(todo)
} catch (error) {
  console.error(error)
} finally {
  console.log('done')
}
```

<br>

## 4. Retry

기본적으로 TanStack Query는 mutation error에 대해 retry하지 않습니다.

하지만 `retry` 옵션으로 설정할 수 있습니다.

```tsx
const mutation = useMutation({
  mutationFn: addTodo,
  retry: 3,
})
```

device가 offline이라 mutation이 실패한 경우에는, 다시 연결되면 같은 순서로 retry됩니다.

<br>

## 5. Persist mutations

필요하다면 mutation을 storage에 persist하고 나중에 다시 resume할 수 있습니다.

이 작업은 hydration 함수로 처리할 수 있습니다.

```tsx
const queryClient = new QueryClient()

// Define the "addTodo" mutation
queryClient.setMutationDefaults(['addTodo'], {
  mutationFn: addTodo,
  onMutate: async (variables, context) => {
    // Cancel current queries for the todos list
    await context.client.cancelQueries({ queryKey: ['todos'] })

    // Create optimistic todo
    const optimisticTodo = { id: uuid(), title: variables.title }

    // Add optimistic todo to todos list
    context.client.setQueryData(['todos'], (old) => [...old, optimisticTodo])

    // Return a result with the optimistic todo
    return { optimisticTodo }
  },
  onSuccess: (result, variables, onMutateResult, context) => {
    // Replace optimistic todo in the todos list with the result
    context.client.setQueryData(['todos'], (old) =>
      old.map((todo) =>
        todo.id === onMutateResult.optimisticTodo.id ? result : todo,
      ),
    )
  },
  onError: (error, variables, onMutateResult, context) => {
    // Remove optimistic todo from the todos list
    context.client.setQueryData(['todos'], (old) =>
      old.filter((todo) => todo.id !== onMutateResult.optimisticTodo.id),
    )
  },
  retry: 3,
})

// Start mutation in some component:
const mutation = useMutation({ mutationKey: ['addTodo'] })
mutation.mutate({ title: 'title' })

// If the mutation has been paused because the device is for example offline,
// Then the paused mutation can be dehydrated when the application quits:
const state = dehydrate(queryClient)

// The mutation can then be hydrated again when the application is started:
hydrate(queryClient, state)

// Resume the paused mutations:
queryClient.resumePausedMutations()
```

### 5.1 Persisting Offline mutations

`persistQueryClient` 플러그인으로 offline mutation을 persist하는 경우, 기본 mutation function을 제공하지 않으면 페이지가 다시 로드됐을 때 mutation을 resume할 수 없습니다.

이것은 기술적 제한입니다.

외부 storage에 persist할 때는 mutation의 state만 저장되고, 함수는 serialize할 수 없기 때문입니다.

hydration 이후에는 mutation을 발생시킨 컴포넌트가 mounted되어 있지 않을 수 있으므로, `resumePausedMutations` 호출 시 `No mutationFn found` 에러가 발생할 수 있습니다.

```tsx
const persister = createSyncStoragePersister({
  storage: window.localStorage,
})

const queryClient = new QueryClient({
  defaultOptions: {
    queries: {
      gcTime: 1000 * 60 * 60 * 24, // 24 hours
    },
  },
})

// we need a default mutation function so that paused mutations can resume after a page reload
queryClient.setMutationDefaults(['todos'], {
  mutationFn: ({ id, data }) => {
    return api.updateTodo(id, data)
  },
})

export default function App() {
  return (
    <PersistQueryClientProvider
      client={queryClient}
      persistOptions={{ persister }}
      onSuccess={() => {
        // resume mutations after initial restore from localStorage was successful
        queryClient.resumePausedMutations()
      }}
    >
      <RestOfTheApp />
    </PersistQueryClientProvider>
  )
}
```

queries와 mutations를 함께 다루는 offline example도 제공됩니다.

## 6. Mutation Scopes

기본적으로 모든 mutation은 병렬로 실행됩니다.

같은 mutation에 대해 `.mutate()`를 여러 번 호출해도 기본 동작은 병렬 실행입니다.

이 동작을 피하려면 `scope.id`를 지정할 수 있습니다.

같은 `scope.id`를 가진 mutation은 직렬로 실행됩니다.

즉, 해당 scope에서 이미 진행 중인 mutation이 있으면 새 mutation은 아래 상태로 시작합니다.

- `isPaused: true`

이 mutation들은 queue에 들어가고, 순서가 오면 자동으로 resume됩니다.

```tsx
const mutation = useMutation({
  mutationFn: addTodo,
  scope: {
    id: 'todo',
  },
})
```
