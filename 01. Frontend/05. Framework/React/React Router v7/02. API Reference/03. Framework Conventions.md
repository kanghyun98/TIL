# Framework Conventions

## 1️⃣ root.tsx

- "root" route (`app/root.tsx`)는 React Router 애플리케이션에서 **유일하게 필수인 라우트**이다.
- 이 라우트는 **모든 라우트의 부모(parent)** 역할을 하며, **루트 `<html>` document**를 렌더링하는 책임을 가진다.

```tsx
// app/root.tsx
import { Outlet, Scripts } from "react-router";

import "./global-styles.css";

export default function App() {
  return (
    <html lang="en">
      <head>
        <link rel="icon" href="/favicon.ico" />
      </head>
      <body>
        <Outlet />
        <Scripts />
      </body>
    </html>
  );
}
```

### ▶️ Components to Renderer

- root route는 애플리케이션의 document를 관리하기 때문에, React Router가 제공하는 문서 수준(document-level) 컴포넌트들을 렌더링하기에 적절한 위치이다. (e.g. `meta`)

### ▶️ Layout Export

- root route는 모든 [route module export](https://reactrouter.com/start/framework/route-module)를 지원한다.
- root route는 추가로 선택적인 `Layout` export도 지원한다.

- `Layout` 컴포넌트는 두 가지 목적을 가진다:
    1. root component, `HydrateFallback`, `ErrorBoundary` 사이에서 문서의 “app shell”을 중복 작성하지 않도록 한다.
    2. root component / `HydrateFallback` / `ErrorBoundary` 간 전환 시 React가 app shell 요소들을 re-mount 하지 않도록 한다.
        - React가 `<Links>` 컴포넌트의 `<link rel="stylesheet">` 태그를 제거했다가 다시 추가하면 [**FOUC(Flash of Unstyled Content)**](https://dev.to/lyqht/what-the-fouc-is-happening-flash-of-unstyled-content-413j)가 발생할 수 있다.

- Layout은 하나의 children prop을 받는다.
    - default export (예: `App`)
    - `HydrateFallback`
    - `ErrorBoundary`

**A note on useLoaderData in the Layout Component**

- `useLoaderData`는 loader가 성공적으로 실행되어 값을 반환했다는 가정을 기반으로 타입이 정의되어 있기 때문에 `ErrorBoundary` 컴포넌트에서 사용할 수 없다.
- `ErrorBoundary`에서 loader 데이터에 접근하려면 `useRouteLoaderData`를 사용해야 한다.
- `Layout` 컴포넌트도 성공 흐름과 에러 흐름 모두에서 사용되기 때문에 동일한 제한이 적용된다.
    - 성공 여부에 따라 로직을 분기해야 한다면 다음을 사용할 수 있다:
        - `useRouteLoaderData("root")`
        - `useRouteError()`

## 2️⃣ routes.ts

- `routes.ts` 파일은 애플리케이션의 **라우트 구성을 정의하는 파일**이다.
- 라우트는 **객체 배열 형태**로 구성한다.

```tsx
// app/routes.ts
import {
  type RouteConfig,
  route,
} from "@react-router/dev/routes";

export default [
  route("some/path", "./some/file.tsx"),
  // pattern ^           ^ module file
] satisfies RouteConfig;
```

route config entries를 만들기 위해 다음 helper 함수들을 사용할 수 있다.

- `route` — 일반 라우트 항목을 생성하는 helper
- `index` — index route 항목을 생성하는 helper
- `layout` — layout route 항목을 생성하는 helper
- `prefix` — 부모 라우트를 만들지 않고 여러 라우트에 경로 prefix를 추가하는 helper
- `relative` — 지정한 디렉토리를 기준으로 파일 경로를 resolve하는 helper

### ▶️ File-based Routing

- 라우트를 설정 파일 대신 **파일 이름 규칙**으로 정의할 수도 있다.
- `@react-router/fs-routes` 패키지에서 [file system routing convention](https://reactrouter.com/how-to/file-route-conventions)을 제공한다.

```tsx
import { type RouteConfig } from "@react-router/dev/routes";
import { flatRoutes } from "@react-router/fs-routes";

export default flatRoutes() satisfies RouteConfig;
```

## 3️⃣ react-router.config.ts

- React Router 프레임워크 설정을 정의하는 파일이다.
- SSR, directory locations, build settings 등 애플리케이션의 여러 측면을 커스터마이즈할 수 있다.

```tsx
// react-router.config.ts
import type { Config } from "@react-router/dev/config";

export default {
  appDirectory: "app",
  buildDirectory: "build",
  ssr: true,
  prerender: ["/", "/about"],
} satisfies Config;
```

### ▶️ Options

- `allowedActionOrigins`
    - 다른 origin에서의 action 요청을 허용할 origin 목록을 지정
- `appDirectory`
    - 애플리케이션 소스 코드가 위치한 디렉토리를 지정
- `basename`
    - basename 지정
    - 기본값 : `/`
- `buildDirectory`
    - 빌드 디렉토리의 경로
    - 기본값: `build`
- `buildEnd`
    - 빌드 완료 후 호출되는 함수
- `future`
    - 다가올 기능들을 활성화 할 수 있는 설정
- `prerender`
    - 빌드 시 pre-render할 경로를 설정
- `presets`
    - React Router plugin config presets 배열
    - 다른 플랫폼이나 도구들과 통합을 위함
- `routeDiscovery`
    - 라우트가 클라이언트에서 어떻게 탐색되고 로드되는지를 설정
    - Defaults to `mode: "lazy"` with `manifestPath: "/__manifest”`
- `serverBuildFile`
    - 서버 빌드 파일의 이름 지정
    - 기본값: `index.js`
- `serverBundles`
    - 라우트를 여러 서버 번들(server bundles) 로 할당하기 위한 함수
    - 서버 번들 ID를 반환해야 하며, 해당 값은 서버 빌드 디렉토리 내부에서 번들의 디렉토리 이름으로 사용됨
- `serverModuleFormat`
    - 서버 빌드의 출력 형식(output format) 을 지정
    - 기본값: `"esm"`
- `ssr`
    - SSR 사용 여부 설정
    - 기본값: `true`

## 4️⃣ entry.client.tsx

- 이 파일은 브라우저의 진입점(entry point) 이다.
- server entry module에서 생성된 마크업을 hydration 하는 역할을 한다.
- 다음과 같은 클라이언트 전용 코드를 초기화할 수 있다
    - 클라이언트 사이드 라이브러리 초기화
    - 클라이언트 전용 provider 추가
    - 기타 브라우저 전용 설정

```tsx
import { startTransition, StrictMode } from "react";
import { hydrateRoot } from "react-dom/client";
import { HydratedRouter } from "react-router/dom";

startTransition(() => {
  hydrateRoot(
    document,
    <StrictMode>
      <HydratedRouter />
    </StrictMode>
  );
});
```

## 5️⃣ entry.server.tsx

- 이 파일은 서버 측 진입점(entry point) 이다.
- 서버에서 React Router 애플리케이션이 HTTP 응답을 생성하는 방식을 제어한다.
- 이 모듈은 현재 요청에 대한 `context`와 `url`을 사용하여 `<ServerRouter>` 요소로 현재 페이지의 마크업을 렌더링해야 한다.
- 이 마크업은 이후 브라우저에서 JavaScript가 로드되면 client entry module(`entry.client.tsx`)을 통해 선택적으로 hydration 된다.

## 6️⃣ `.client` modules

- 브라우저에서 모듈 사이드 이펙트(module side effects) 를 사용하는 파일이나 의존성이 있을 수 있다.
- 파일 이름에 `*.client.ts`를 사용하거나 `.client` 디렉토리 안에 파일을 넣으면 해당 파일을 서버 번들에서 제외할 수 있다.
- 이 모듈에서 export된 값들은 서버에서는 모두 undefined가 되기 때문에 useEffect나 클릭 핸들러 같은 사용자 이벤트 안에서만 사용해야 한다.

## 7️⃣ `.server` modules

- 서버 전용(server-only) 모듈을 명시적으로 표시하는 방법이다.
- 클라이언트 번들에 포함되지 않으며 서버에서만 실행된다.
