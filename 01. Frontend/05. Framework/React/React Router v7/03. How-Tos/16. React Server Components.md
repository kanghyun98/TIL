# React Server Components

> React Server Components 지원은 실험적이며, 마이너/패치 릴리스에서도 호환성이 깨지는 변경이 발생할 수 있습니다. 사용할 때는 주의가 필요하며, 관련 변경 사항은 릴리스 노트를 매우 면밀히 확인해야 합니다.

<br>

React Server Components(RSC)는 일반적으로 React 19부터 제공되는 아키텍처와 API 집합을 가리킵니다.

문서에서는 다음과 같이 설명합니다.

> Server Components are a new type of Component that renders ahead of time, before bundling, in an environment separate from your client app or SSR server.

React Router는 RSC 호환 번들러와 통합하기 위한 API 집합을 제공하며, 이를 통해 React Router 애플리케이션에서 Server Components와 Server Functions를 활용할 수 있습니다. 이러한 React 기능이 익숙하지 않다면, React Router의 RSC API를 사용하기 전에 먼저 관련 문서를 읽어보는 것을 권장합니다.

RSC 지원은 Framework Mode와 Data Mode 모두에서 사용할 수 있습니다. 두 모드의 개념적 차이에 대해서는 Picking a Mode를 참고할 수 있습니다. 다만, 이 가이드에서 다루듯 RSC 모드와 비 RSC 모드는 API와 기능 측면에서 차이가 있습니다.

<br>

## 1. Quick Start

가장 빠르게 시작하는 방법은 템플릿 중 하나를 사용하는 것입니다.

이 템플릿들은 React Router RSC API가 이미 구성되어 있으며, 다음과 같은 기능을 바로 사용할 수 있습니다.

- Server Component Routes
- Server Side Rendering (SSR)
- Client Components (`"use client"` 지시어 사용)
- Server Functions (`"use server"` 지시어 사용)

<br>

### 1.1. RSC Framework Mode Template

RSC Framework Mode 템플릿은 unstable React Router RSC Vite 플러그인과 실험적인 `@vitejs/plugin-rsc` 플러그인을 함께 사용합니다.

```shell
$ npx create-react-router@latest --template remix-run/react-router-templates/unstable_rsc-framework-mode
```

<br>

### 1.2. RSC Data Mode Templates

Vite RSC Data Mode 템플릿은 실험적인 Vite `@vitejs/plugin-rsc` 플러그인을 사용합니다.

```shell
$ npx create-react-router@latest --template remix-run/react-router-templates/unstable_rsc-data-mode-vite
```

## 2. RSC Framework Mode

RSC Framework Mode의 대부분의 API와 기능은 비 RSC Framework Mode와 동일하므로, 이 가이드는 차이점에 집중합니다.

<br>

### 2.1. New React Router RSC Vite Plugin

RSC Framework Mode는 비 RSC Framework Mode와 다른 Vite 플러그인을 사용하며, 현재 `unstable_reactRouterRSC`로 export됩니다.

이 새로운 Vite 플러그인은 실험적인 `@vitejs/plugin-rsc` 플러그인에 대한 peer dependency도 가집니다. 또한 Vite 설정에서 `@vitejs/plugin-rsc` 플러그인은 React Router RSC 플러그인 뒤에 위치해야 합니다.

```tsx
import { defineConfig } from "vite";
import { unstable_reactRouterRSC as reactRouterRSC } from "@react-router/dev/vite";
import rsc from "@vitejs/plugin-rsc";

export default defineConfig({
  plugins: [reactRouterRSC(), rsc()],
});
```

<br>

### 2.2. Build Output

RSC Framework Mode의 서버 빌드 파일(`build/server/index.js`)은 이제 document/data 요청을 위한 `default` 요청 핸들러 함수(`(request: Request) => Promise<Response>`)를 export합니다.

필요하다면 이 함수를 표준 Node.js request listener로 변환해서 Node의 내장 `http.createServer` 함수나 이를 지원하는 다른 환경에서 사용할 수 있습니다. 이를 위해 `@remix-run/node-fetch-server`의 `createRequestListener` 함수를 사용할 수 있습니다.

예를 들어 Express에서는 다음과 같습니다.

```tsx
import express from "express";
import requestHandler from "./build/server/index.js";
import { createRequestListener } from "@remix-run/node-fetch-server";

const app = express();

app.use(
  "/assets",
  express.static("build/client/assets", {
    immutable: true,
    maxAge: "1y",
  }),
);
app.use(express.static("build/client"));
app.use(createRequestListener(requestHandler));
app.listen(3000);
```

<br>

### 2.3. React Elements From Loaders/Actions

RSC Framework Mode에서는 loader와 action이 다른 데이터와 함께 React element도 반환할 수 있습니다.

이 element들은 항상 서버에서만 렌더링됩니다.

```tsx
import type { Route } from "./+types/route";

export async function loader() {
  return {
    message: "Message from the server!",
    element: <p>Element from the server!</p>,
  };
}

export default function Route({
  loaderData,
}: Route.ComponentProps) {
  return (
    <>
      <h1>{loaderData.message}</h1>
      {loaderData.element}
    </>
  );
}
```

loader/action에서 반환한 React element 안에서 client 전용 기능(예: Hooks, event handler)을 사용해야 한다면, 해당 기능을 사용하는 컴포넌트를 client module로 분리해야 합니다.

```tsx
"use client";

export function Counter() {
  const [count, setCount] = useState(0);
  return (
    <button onClick={() => setCount(count + 1)}>
      Count: {count}
    </button>
  );
}
```

```tsx
import type { Route } from "./+types/route";
import { Counter } from "./counter";

export async function loader() {
  return {
    message: "Message from the server!",
    element: (
      <>
        <p>Element from the server!</p>
        <Counter />
      </>
    ),
  };
}

export default function Route({
  loaderData,
}: Route.ComponentProps) {
  return (
    <>
      <h1>{loaderData.message}</h1>
      {loaderData.element}
    </>
  );
}
```

### 2.4. Server Component Routes

라우트가 일반적인 `default` 컴포넌트 export 대신 `ServerComponent`를 export하면, 이 컴포넌트와 다른 라우트 컴포넌트(`ErrorBoundary`, `HydrateFallback`, `Layout`)는 일반적인 client component가 아니라 server component가 됩니다.

```tsx
import type { Route } from "./+types/route";
import { Outlet } from "react-router";
import { getMessage } from "./message";

export async function loader() {
  return {
    message: await getMessage(),
  };
}

export function ServerComponent({
  loaderData,
}: Route.ComponentProps) {
  return (
    <>
      <h1>Server Component Route</h1>
      <p>Message from the server: {loaderData.message}</p>
      <Outlet />
    </>
  );
}
```

server-first 라우트 안에서 client 전용 기능(예: Hooks, event handler)을 사용해야 한다면, 해당 기능을 사용하는 컴포넌트를 client module로 분리해야 합니다.

```tsx
"use client";

export function Counter() {
  const [count, setCount] = useState(0);
  return (
    <button onClick={() => setCount(count + 1)}>
      Count: {count}
    </button>
  );
}
```

```tsx
import { Counter } from "./counter";

export function ServerComponent() {
  return (
    <>
      <h1>Counter</h1>
      <Counter />
    </>
  );
}
```

### 2.5. `.server`/`.client` Modules

RSC의 `"use server"`와 `"use client"` 지시어와 혼동을 피하기 위해, RSC Framework Mode에서는 `.server` module과 `.client` module 지원이 더 이상 내장되어 있지 않습니다.

파일 네이밍 규칙에 의존하지 않는 대안으로, `@vitejs/plugin-rsc`가 제공하는 `"server-only"`와 `"client-only"` import 사용이 권장됩니다. 예를 들어 어떤 module이 client build에 실수로 포함되지 않도록 하려면, server 전용 module에서 side effect로 `"server-only"`를 import하면 됩니다.

```ts
import "server-only";

// Rest of the module...
```

React 팀이 만든 공식 npm 패키지 `server-only`와 `client-only`도 있지만, 이를 직접 설치할 필요는 없습니다. `@vitejs/plugin-rsc`가 내부적으로 이 import를 처리하고, runtime error 대신 build-time validation을 제공합니다.

기존 `.server` 및 `.client` 파일 네이밍 규칙에 의존하는 코드를 빠르게 마이그레이션하고 싶다면 `vite-env-only` 플러그인을 직접 사용하는 것이 권장됩니다. 예를 들어 `.server` module이 client build에 실수로 포함되지 않도록 하려면 다음과 같이 설정할 수 있습니다.

```tsx
import { defineConfig } from "vite";
import { denyImports } from "vite-env-only";
import { unstable_reactRouterRSC as reactRouterRSC } from "@react-router/dev/vite";
import rsc from "@vitejs/plugin-rsc";

export default defineConfig({
  plugins: [
    denyImports({
      client: { files: ["**/.server/*", "**/*.server.*"] },
    }),
    reactRouterRSC(),
    rsc(),
  ],
});
```

### 2.6. MDX Route Support

RSC Framework Mode에서는 `@mdx-js/rollup` v3.1.1+를 사용할 때 MDX route를 지원합니다.

MDX route에서 export되는 모든 컴포넌트는 RSC 환경에서도 유효해야 하므로, Hooks 같은 client 전용 기능을 사용할 수 없습니다. 이런 기능이 필요한 컴포넌트는 client module로 분리해야 합니다.

<br>

### 2.7. Custom Entry Files

RSC Framework Mode는 custom entry file을 지원하며, 이를 통해 RSC 서버, SSR 서버, client entry point의 동작을 커스터마이즈할 수 있습니다.

플러그인은 `app` 디렉터리에서 다음 custom entry file들을 자동으로 감지합니다.

- `app/entry.rsc.ts` (또는 `.tsx`) - Custom RSC server entry
- `app/entry.ssr.ts` (또는 `.tsx`) - Custom SSR server entry
- `app/entry.client.tsx` - Custom client entry

이 파일들이 없으면 React Router는 프레임워크가 제공하는 기본 entry를 사용합니다.

<br>

#### 2.7.1. Basic Override Pattern

기본 동작을 감싸거나 확장하는 custom entry file을 만들 수 있습니다. 예를 들어 RSC entry에 custom logging을 추가하려면 다음과 같이 작성할 수 있습니다.

```ts
import defaultEntry from "@react-router/dev/config/default-rsc-entries/entry.rsc";
import { RouterContextProvider } from "react-router";

export default {
  fetch(request: Request): Promise<Response> {
    console.log(
      "Custom RSC entry handling request:",
      request.url,
    );
    const requestContext = new RouterContextProvider();

    return defaultEntry.fetch(request, requestContext);
  },
};

if (import.meta.hot) {
  import.meta.hot.accept();
}
```

같은 방식으로 SSR entry도 커스터마이즈할 수 있습니다.

```ts
import { generateHTML as defaultGenerateHTML } from "@react-router/dev/config/default-rsc-entries/entry.ssr";

export function generateHTML(
  request: Request,
  serverResponse: Response,
): Promise<Response> {
  console.log(
    "Custom SSR entry generating HTML for:",
    request.url,
  );

  return defaultGenerateHTML(request, serverResponse);
}
```

client의 경우는 다음과 같습니다.

```ts
import "@react-router/dev/config/default-rsc-entries/entry.client";
```

<br>

#### 2.7.2. Copying Default Entries

더 고급스러운 커스터마이징이 필요하다면 기본 entry를 복사해서 원하는 대로 수정할 수 있습니다.

기본 entry를 찾는 방법은 다음과 같습니다.

1. IDE에서 기본 entry import에 대해 "Go to Definition" 또는 Cmd/Ctrl+Click을 사용합니다.

```ts
import defaultEntry from "@react-router/dev/config/default-rsc-entries/entry.rsc";
```

2. 기본 entry 코드를 custom file로 복사합니다.
3. 필요에 맞게 수정합니다.

기본 entry는 다음 위치에 있습니다.

- `@react-router/dev/config/default-rsc-entries/entry.rsc`
- `@react-router/dev/config/default-rsc-entries/entry.ssr`
- `@react-router/dev/config/default-rsc-entries/entry.client`

또는 `node_modules/@react-router/dev/dist/config/default-rsc-entries/` 아래에서 직접 확인할 수도 있습니다.

기본 entry를 복사할 때는 필요한 export를 유지해야 합니다.

- `entry.rsc.ts`는 `fetch` 메서드를 가진 default object를 export해야 합니다.
- `entry.ssr.ts`는 `generateHTML` 함수를 export해야 합니다.
- `entry.client.tsx`는 client-side hydration을 처리해야 합니다.

<br>

### 2.8. Unsupported Config Options

초기 unstable 릴리스에서는 `react-router.config.ts`의 다음 옵션들이 RSC Framework Mode에서 아직 지원되지 않습니다.

- `buildEnd`
- `prerender`
- `presets`
- `routeDiscovery`
- `serverBundles`
- `ssr: false` (SPA Mode)
- `future.v8_splitRouteModules`
- `future.unstable_subResourceIntegrity`

## 3. RSC Data Mode

위에서 설명한 RSC Framework Mode API는 더 낮은 수준의 RSC Data Mode API 위에 구축되어 있습니다.

RSC Data Mode는 RSC Framework Mode의 일부 기능(`routes.ts` 설정과 파일 시스템 라우팅, HMR, Hot Data Revalidation 등)이 빠져 있지만, 더 유연하며 자체 번들러와 서버 추상화에 통합할 수 있습니다.

<br>

### 3.1. Configuring Routes

라우트는 `matchRSCServerRequest`의 인자로 설정합니다. 최소한 path와 component가 필요합니다.

```tsx
function Root() {
  return <h1>Hello world</h1>;
}

matchRSCServerRequest({
  // ...other options
  routes: [{ path: "/", Component: Root }],
});
```

컴포넌트를 inline으로 정의할 수도 있지만, 초기 성능과 코드 구성 측면에서 `lazy()` 옵션과 Route Module 정의 사용이 권장됩니다.

지금까지 Route Module API는 Framework Mode 전용 기능이었습니다. 하지만 RSC route config의 `lazy` 필드는 Route Module export와 동일한 export를 기대하므로, 두 API가 더 통합됩니다.

```tsx
import type { unstable_RSCRouteConfig as RSCRouteConfig } from "react-router";

export function routes() {
  return [
    {
      id: "root",
      path: "",
      lazy: () => import("./root/route"),
      children: [
        {
          id: "home",
          index: true,
          lazy: () => import("./home/route"),
        },
        {
          id: "about",
          path: "about",
          lazy: () => import("./about/route"),
        },
      ],
    },
  ] satisfies RSCRouteConfig;
}
```

<br>

### 3.2. Server Component Routes

기본적으로 각 라우트의 `default` export는 Server Component를 렌더링합니다.

```tsx
export default function Home() {
  return (
    <main>
      <article>
        <h1>Welcome to React Router RSC</h1>
        <p>
          You won't find me running any JavaScript in the
          browser!
        </p>
      </article>
    </main>
  );
}
```

Server Component의 좋은 점 중 하나는 비동기 함수로 만들어 컴포넌트 안에서 직접 데이터를 가져올 수 있다는 것입니다.

```tsx
export default async function Home() {
  let user = await getUserData();
  return (
    <main>
      <article>
        <h1>Welcome to React Router RSC</h1>
        <p>
          You won't find me running any JavaScript in the
          browser!
        </p>
        <p>
          Hello, {user ? user.name : "anonymous person"}!
        </p>
      </article>
    </main>
  );
}
```

Server Components는 loader와 action에서도 반환할 수 있습니다. 일반적으로 RSC로 애플리케이션을 구성할 때 loader는 주로 `status` code를 설정하거나 `redirect`를 반환하는 용도로 유용합니다.

loader에서 Server Components를 사용하는 방식은 RSC를 점진적으로 도입할 때 도움이 될 수 있습니다.

<br>

### 3.3. Server Functions

Server Functions는 서버에서 실행되는 비동기 함수를 호출할 수 있게 해주는 React 기능입니다. `"use server"` 지시어로 정의합니다.

```tsx
"use server";

export async function updateFavorite(formData: FormData) {
  let movieId = formData.get("id");
  let intent = formData.get("intent");
  if (intent === "add") {
    await addFavorite(Number(movieId));
  } else {
    await removeFavorite(Number(movieId));
  }
}
```

```tsx
import { updateFavorite } from "./action.ts";

export async function AddToFavoritesForm({
  movieId,
}: {
  movieId: number;
}) {
  let isFav = await isFavorite(movieId);
  return (
    <form action={updateFavorite}>
      <input type="hidden" name="id" value={movieId} />
      <input
        type="hidden"
        name="intent"
        value={isFav ? "remove" : "add"}
      />
      <AddToFavoritesButton isFav={isFav} />
    </form>
  );
}
```

Server Function이 호출된 뒤에는 React Router가 자동으로 라우트를 revalidate하고, 새로운 서버 콘텐츠로 UI를 업데이트합니다. 별도로 cache invalidation을 처리할 필요는 없습니다.

<br>

### 3.4. Client Properties

라우트는 런타임에 서버에서 정의되지만, client reference와 `"use client"`를 활용해 `clientLoader`, `clientAction`, `shouldRevalidate`를 여전히 제공할 수 있습니다.

```tsx
"use client";

export function clientAction() {}

export function clientLoader() {}

export function shouldRevalidate() {}
```

이후 lazy loaded route module에서 이를 다시 export할 수 있습니다.

```tsx
export {
  clientAction,
  clientLoader,
  shouldRevalidate,
} from "./route.client";

export default function Root() {
  // ...
}
```

이 방식은 전체 라우트를 Client Component로 만드는 방법이기도 합니다.

```tsx
import { default as ClientRoot } from "./route.client";

export {
  clientAction,
  clientLoader,
  shouldRevalidate,
} from "./route.client";

export default function Root() {
  // Adding a Server Component at the root is required by bundlers
  // if you're using css side-effects imports.
  return <ClientRoot />;
}
```

### 3.5. Bundler Configuration

React Router는 RSC 호환 번들러와 쉽게 통합할 수 있는 여러 API를 제공합니다. 이는 React Router Data Mode를 사용해 자체 custom framework를 만드는 경우에 유용합니다.

다음 단계는 React Router 애플리케이션에서 Server Components(RSC)를 사용해 페이지를 server-render(SSR)하고, single-page app(SPA) navigation을 위해 hydrate하는 방법을 보여줍니다. SSR이나 client-side hydration을 꼭 사용해야 하는 것은 아닙니다. 원한다면 HTML 생성을 Static Site Generation(SSG)이나 Incremental Static Regeneration(ISR)에 활용할 수도 있습니다. 이 가이드는 일반적인 RSC 기반 애플리케이션에서 여러 API를 어떻게 연결하는지 설명하는 데 목적이 있습니다.

<br>

### 3.6. Entry points

라우트 정의 외에도 다음 항목들을 설정해야 합니다.

1. 들어오는 요청을 처리하고, RSC payload를 가져와 HTML로 변환하는 서버
2. RSC payload를 생성하는 React 서버
3. 생성된 HTML을 hydrate하고, hydration 이후 server action을 지원하도록 `callServer` 함수를 설정하는 브라우저 핸들러

다음 네이밍 규칙은 익숙함과 단순함을 위해 선택된 것입니다. 필요에 따라 entry point 이름과 구성을 자유롭게 정할 수 있습니다.

아래 예제는 각각의 entry point에 대해 번들러 문서를 참고할 수 있도록 구성되어 있습니다.

이 예제들은 모두 서버와 request handling을 위해 `express`와 `@remix-run/node-fetch-server`를 사용합니다.

**Routes**

Configuring Routes를 참고합니다.

**Server**

SSR을 반드시 사용할 필요는 없습니다. RSC를 사용해 HTML을 "prerender"하여 Static Site Generation(SSG)이나 Incremental Static Regeneration(ISR) 같은 방식으로 활용할 수 있습니다.

`entry.ssr.tsx`는 서버의 entry point입니다. 이 파일은 요청을 처리하고, RSC 서버를 호출하며, document 요청에서 RSC payload를 HTML로 변환하는 역할을 합니다.

Relevant APIs:

- `routeRSCServerRequest`
- `RSCStaticRouter`

**RSC Server**

"React Server"와 요청 처리/SSR을 담당하는 서버가 따로 존재하더라도, 실제로는 반드시 두 개의 물리적 서버가 필요한 것은 아닙니다. 같은 서버 안에 두 개의 module graph를 두는 방식으로도 구성할 수 있습니다. 이는 React가 RSC payload를 생성할 때와 클라이언트에서 hydrate할 HTML을 생성할 때 다르게 동작하기 때문에 중요합니다.

`entry.rsc.tsx`는 React Server의 entry point입니다. 요청을 라우트에 매칭하고 RSC payload를 생성하는 역할을 합니다.

Relevant APIs:

- `matchRSCServerRequest`

**Browser**

`entry.browser.tsx`는 클라이언트의 entry point입니다. 생성된 HTML을 hydrate하고, hydration 이후 server action을 지원하도록 `callServer` 함수를 설정하는 역할을 합니다.

Relevant APIs:

- `createCallServer`
- `getRSCStream`
- `RSCHydratedRouter`

#### 3.7.2. `entry.ssr.tsx`

다음은 단순화된 Vite SSR Server 예제입니다.

```tsx
import { createFromReadableStream } from "@vitejs/plugin-rsc/ssr";
import { renderToReadableStream as renderHTMLToReadableStream } from "react-dom/server.edge";
import {
  unstable_routeRSCServerRequest as routeRSCServerRequest,
  unstable_RSCStaticRouter as RSCStaticRouter,
} from "react-router";

export async function generateHTML(
  request: Request,
  serverResponse: Response,
): Promise<Response> {
  return await routeRSCServerRequest({
    // The incoming request.
    request,
    // The React Server response
    serverResponse,
    // Provide the React Server touchpoints.
    createFromReadableStream,
    // Render the router to HTML.
    async renderHTML(getPayload) {
      const payload = getPayload();
      const bootstrapScriptContent =
        await import.meta.viteRsc.loadBootstrapScriptContent(
          "index",
        );

      return await renderHTMLToReadableStream(
        <RSCStaticRouter getPayload={getPayload} />,
        {
          bootstrapScriptContent,
          formState: payload.formState,
        },
      );
    },
  });
}
```

#### 3.7.3. `entry.rsc.tsx`

다음은 단순화된 Vite RSC Server 예제입니다.

```tsx
import {
  createTemporaryReferenceSet,
  decodeAction,
  decodeFormState,
  decodeReply,
  loadServerAction,
  renderToReadableStream,
} from "@vitejs/plugin-rsc/rsc";
import { unstable_matchRSCServerRequest as matchRSCServerRequest } from "react-router";

import { routes } from "./routes/config";

function fetchServer(request: Request) {
  return matchRSCServerRequest({
    // Provide the React Server touchpoints.
    createTemporaryReferenceSet,
    decodeAction,
    decodeFormState,
    decodeReply,
    loadServerAction,
    // The incoming request.
    request,
    // The app routes.
    routes: routes(),
    // Encode the match with the React Server implementation.
    generateResponse(match) {
      return new Response(
        renderToReadableStream(match.payload),
        {
          status: match.statusCode,
          headers: match.headers,
        },
      );
    },
  });
}

export default async function handler(request: Request) {
  // Import the generateHTML function from the client environment
  const ssr = await import.meta.viteRsc.loadModule<
    typeof import("./entry.ssr")
  >("ssr", "index");

  return ssr.generateHTML(
    request,
    await fetchServer(request),
  );
}
```

#### 3.7.4. `entry.browser.tsx`

```tsx
import {
  createFromReadableStream,
  createTemporaryReferenceSet,
  encodeReply,
  setServerCallback,
} from "@vitejs/plugin-rsc/browser";
import { startTransition, StrictMode } from "react";
import { hydrateRoot } from "react-dom/client";
import {
  unstable_createCallServer as createCallServer,
  unstable_getRSCStream as getRSCStream,
  unstable_RSCHydratedRouter as RSCHydratedRouter,
  type unstable_RSCPayload as RSCServerPayload,
} from "react-router";

// Create and set the callServer function to support post-hydration server actions.
setServerCallback(
  createCallServer({
    createFromReadableStream,
    createTemporaryReferenceSet,
    encodeReply,
  }),
);

// Get and decode the initial server payload.
createFromReadableStream<RSCServerPayload>(
  getRSCStream(),
).then((payload) => {
  startTransition(async () => {
    const formState =
      payload.type === "render"
        ? await payload.formState
        : undefined;

    hydrateRoot(
      document,
      <StrictMode>
        <RSCHydratedRouter
          createFromReadableStream={
            createFromReadableStream
          }
          payload={payload}
        />
      </StrictMode>,
      {
        formState,
      },
    );
  });
});
```
