# Instrumentation (unstable)

Instrumentation은 실제 route handler를 수정하지 않고도 로깅, 에러 리포팅, 성능 추적을 추가할 수 있게 해줍니다. 이를 통해 서버와 클라이언트 모두에서 애플리케이션의 관측 가능성(observability)을 확보할 수 있습니다.  

<br>

## 1. Overview

Instrumentation API를 사용하면 request handler, router 동작, route middleware, route handler 등을 감싸는(wrapper) 함수를 정의할 수 있습니다. 



이를 통해 다음과 같은 작업이 가능합니다:

- 애플리케이션 성능 모니터링
- 로깅 추가
- observability 플랫폼(Sentry, DataDog, New Relic 등)과 통합
- OpenTelemetry tracing 구현
- 사용자 행동 및 navigation 패턴 추적 



Instrumentation의 중요한 설계 원칙은 **read-only**입니다.

- 실행 흐름을 관찰할 수는 있지만
- route handler의 인자나 반환값을 수정하여 런타임 동작을 변경할 수는 없습니다 

또한 instrumentation은 추가적인 코드 실행을 포함하므로, 애플리케이션 성능에 영향을 줄 수 있습니다. 필요에 따라 조건부로 적용해야 합니다.  

<br>

## 2. Quick Start (Framework Mode)

### 2.1. Server-side Instrumentation

`entry.server.tsx`에 instrumentation을 추가합니다.

```ts
export const unstable_instrumentations = [
  {
    handler(handler) {
      handler.instrument({
        async request(handleRequest, { request }) {
          let url = `${request.method} ${request.url}`;
          console.log(`Request start: ${url}`);
          await handleRequest();
          console.log(`Request end: ${url}`);
        },
      });
    },

    route(route) {
      if (route.id === "root") return;

      route.instrument({
        async loader(callLoader, { request }) {
          let url = `${request.method} ${request.url}`;
          console.log(`Loader start: ${url} - ${route.id}`);
          await callLoader();
          console.log(`Loader end: ${url} - ${route.id}`);
        },
        // async action() {}
        // async middleware() {}
        // async lazy() {}
      });
    },
  },
];

export default function handleRequest(/* ... */) {
  // 기존 handleRequest 구현
}
```

<br>

### 2.2. Client-side Instrumentation

`entry.client.tsx`에 instrumentation을 추가합니다.

```ts
const unstable_instrumentations = [
  {
    router(router) {
      router.instrument({
        async navigate(callNavigate, { currentUrl, to }) {
          let nav = `${currentUrl} → ${to}`;
          console.log(`Navigation start: ${nav}`);
          await callNavigate();
          console.log(`Navigation end: ${nav}`);
        },
      });
    },
  },
];
```

## 4. Core Concepts

### 4.1. Instrumentation Levels

instrumentation은 여러 레벨에서 가능하고, 각 실행 상황에 맞는 정보(info)를 같이 넘겨준다.

- Handler Level (Server)
- Router Level (Client)
- Route Level (Server + Client)

<br>

### 4.2. Read-only Design

Instrumentation은 실행 흐름을 관찰하기 위한 목적이며, 다음과 같은 제약이 있습니다:

- handler에 전달되는 인자를 수정할 수 없습니다
- handler의 반환값을 변경할 수 없습니다
- 애플리케이션의 런타임 동작을 변경할 수 없습니다

<br>

### 4.3. Error Handling

Instrumentation은 에러 처리에 대해 다음과 같은 특성을 가집니다:

- handler 실행 중 발생한 에러는 instrumentation에서 관찰할 수 있습니다
- instrumentation 내부에서 발생한 에러는 애플리케이션 동작에 영향을 주지 않도록 처리됩니다

<br>

### 4.4. Composition

여러 개의 instrumentation을 함께 사용할 수 있으며, 각 instrumentation은 순차적으로 적용됩니다.

각 wrapper는 이전 wrapper를 감싸는 형태로 실행되며, nested execution 구조를 형성합니다.

<br>

### 4.5. Conditional Instrumentation

조건에 따라 instrumentation을 적용할 수 있습니다.

예를 들어 특정 환경이나 특정 route에만 instrumentation을 적용할 수 있습니다.
