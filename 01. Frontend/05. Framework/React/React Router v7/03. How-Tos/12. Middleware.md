# Middleware

Middleware는 매칭된 경로의 Response 생성 전후에 코드를 실행할 수 있게 해줍니다. 이를 통해 인증, 로깅, 에러 처리, 데이터 전처리 같은 공통 패턴을 재사용 가능한 방식으로 구현할 수 있습니다.

Middleware는 중첩된 체인으로 실행됩니다. route handler로 내려갈 때는 parent route에서 child route 순서로 실행되고, Response가 생성된 뒤에는 child route에서 parent route 순서로 다시 올라옵니다.

예를 들어 `GET /parent/child` 요청에서는 다음 순서로 실행됩니다.

    - Root middleware start
      - Parent middleware start
        - Child middleware start
          - Run loaders, generate HTML Response
        - Child middleware end
      - Parent middleware end
    - Root middleware end

>  서버의 middleware(Framework Mode)와 클라이언트의 middleware(Framework/Data Mode) 사이에는 약간의 차이가 있습니다. 이 문서의 대부분의 예시는 다른 HTTP 서버의 middleware에 익숙한 사용자를 위해 Server Middleware를 기준으로 설명합니다.

<br>

## 1. Quick Start (Framework mode)

### 1.1. Enable the middleware flag

먼저 React Router config에서 middleware를 활성화합니다.

```ts
// react-router.config.ts
import type { Config } from "@react-router/dev/config";

export default {
  future: {
    v8_middleware: true,
  },
} satisfies Config;
```

>  middleware 기능을 활성화하면 `action`과 `loader`의 `context` 파라미터 타입이 바뀝니다. 
>  현재 `context`를 사용 중이라면 아래의 `getLoadContext` 섹션을 주의해서 봐야 합니다.

<br>

### 1.2. Create a context

Middleware는 middleware 체인 아래로 데이터를 전달하기 위해 `context` provider 인스턴스를 사용합니다. `createContext`를 사용하면 타입 안전한 context 객체를 만들 수 있습니다.

```ts
import { createContext } from "react-router";
import type { User } from "~/types";

export const userContext = createContext<User | null>(null);
```

<br>

### 1.3. Export middleware from your routes

```ts
import { redirect } from "react-router";
import { userContext } from "~/context";

// Server-side Authentication Middleware
async function authMiddleware({ request, context }) {
  const user = await getUserFromSession(request);
  if (!user) {
    throw redirect("/login");
  }
  context.set(userContext, user);
}

export const middleware: Route.MiddlewareFunction[] = [
  authMiddleware,
];

// Client-side timing middleware
async function timingMiddleware({ context }, next) {
  const start = performance.now();
  await next();
  const duration = performance.now() - start;
  console.log(`Navigation took ${duration}ms`);
}

export const clientMiddleware: Route.ClientMiddlewareFunction[] =
  [timingMiddleware];

export async function loader({
  context,
}: Route.LoaderArgs) {
  const user = context.get(userContext);
  const profile = await getProfile(user);
  return { profile };
}

export default function Dashboard({
  loaderData,
}: Route.ComponentProps) {
  return (
    <div>
      <h1>Welcome {loaderData.profile.fullName}!</h1>
      <Profile profile={loaderData.profile} />
    </div>
  );
}
```

<br>

### 1.4. Update your `getLoadContext` function (if applicable)

custom server와 `getLoadContext` 함수를 사용 중이라면, JavaScript 객체 대신 `RouterContextProvider` 인스턴스를 반환하도록 구현을 바꿔야 합니다.

```ts
import {
 createContext,
  RouterContextProvider,
} from "react-router";
import { createDb } from "./db";

+const dbContext = createContext<Database>();

function getLoadContext(req, res) {
  //  return { db: createDb() };
  const context = new RouterContextProvider();
  context.set(dbContext, createDb());
  return context;
}
```

## 3. Core Concepts

### 3.1. Server vs Client Middleware

**Server middleware**는 Framework mode의 서버에서 **HTML Document 요청과 이후  `.data` 요청에 대해 실행**됩니다. 
서버 middleware는 HTTP Request에 대한 응답으로 실행되므로, `next` 함수를 통해 middleware 체인을 따라 다시 HTTP Response를 위로 전달합니다.

```ts
async function serverMiddleware({ request }, next) {
  console.log(request.method, request.url);
  let response = await next();
  console.log(response.status, request.method, request.url);
  return response;
}

// Framework mode only
export const middleware: Route.MiddlewareFunction[] = [
  serverMiddleware,
];
```

<br>

**Client middleware**는 Framework mode와 Data mode의 브라우저에서 **client-side navigation과 fetcher 호출에 대해 실행**됩니다. 
클라이언트에는 HTTP Request가 없기 때문에, server middleware처럼 다시 bubble up할 `Response`가 없습니다. 대부분의 경우 클라이언트에서는 `next`의 반환값을 무시하고 middleware에서 아무것도 반환하지 않아도 됩니다.

```ts
async function clientMiddleware({ request }, next) {
  console.log(request.method, request.url);
  await next();
  console.log(response.status, request.method, request.url);
}

// Framework mode
export const clientMiddleware: Route.ClientMiddlewareFunction[] =
  [clientMiddleware];

// Or, Data mode
const route = {
  path: "/",
  middleware: [clientMiddleware],
  loader: rootLoader,
  Component: Root,
};
```

<br>

경우에 따라 loader/action 결과를 바탕으로 후처리를 하고 싶을 수 있습니다. 클라이언트 middleware에서는 `Response` 대신 **현재 활성화된 `dataStrategy`가 반환한 값(`Record<string, DataStrategyResult>`, route id 기준 key)을 위로 전달**합니다. 이를 통해 실행된 `loader`/`action`의 결과를 기준으로 조건부 로직을 수행할 수 있습니다.

<br>

### 3.2. When Middleware Runs

Middleware가 언제 실행되는지를 이해하는 것은 애플리케이션이 의도대로 동작하는지 판단하는 데 매우 중요합니다.

<br>

#### 3.2.1. Server Middleware

**Server Middleware는 “서버 요청이 있을 때만” 실행되고, 불필요한 요청을 만들지 않습니다.** Server Middleware가 있다고 해서 페이지 이동 시, 무조건 서버를 호출하지 않습니다.

여기서 React Router에서 **"handler"가 무엇인지가 문제**가 됩니다.

> 👉 handler = “요청을 처리해서 응답을 만드는 실행 단위”
> 👉 handler가 있어야 middleware가 실행됩니다.

- **Document 요청 (SSR)** : 응답 = HTML = loader 결과 + component 렌더링 => **전체 route가 하나의 handler 역할**
  - `GET /blog`

- **Data 요청 (SPA navigation)** : response에 포함되는 것이 `action`/`loader`뿐 => **loader / action**
  - `GET /route.data`




따라서:

- document 요청은 `loader`가 없어도 UI를 렌더링하는 "handler" 안에 있으므로 server middleware가 실행됩니다.
- client-side navigation은 서버에 `.data` 요청이 발생해 `action`/`loader`를 실행할 때만 server middleware가 실행됩니다.

<br>

#### 3.2.2. Client Middleware

client middleware는 더 단순합니다. 이미 클라이언트에 있고, navigation 시 항상 router에 대한 "request"를 만들기 때문입니다. client middleware는 실행할 `loader`가 있는지와 상관없이 모든 client navigation에서 실행됩니다.

<br>

### 3.3. Context API

새로운 context 시스템은 타입 안전성을 제공하고 naming conflict를 방지하며, nested middleware와 `action`/`loader` 함수에 데이터를 전달할 수 있게 해줍니다.

    // ✅ Type-safe
    import { createContext } from "react-router";
    const userContext = createContext<User>();
    
    // Later in middleware/`loader`s
    context.set(userContext, user); // Must be `User` type
    const user = context.get(userContext); // Returns `User` type
    
    // ❌ Old way (no type safety)
    context.user = user; // Could be anything

<br>

### 3.4. The `next` function

`next` 함수의 동작은 어떤 route middleware에서 호출되었는지에 따라 달라집니다.

- leaf가 아닌 middleware에서 호출되면 체인상의 다음 middleware를 실행합니다.
- leaf middleware에서 호출되면 route handler를 실행하고 요청에 대한 최종 `Response`를 생성합니다.


```ts
const middleware = async ({ context }, next) => {
  // Code here runs BEFORE handlers
  console.log("Before");

  const response = await next();

  // Code here runs AFTER handlers
  console.log("After");

  return response; // Optional on client, required on server
};
```



각 middleware에서는 `next()`를 한 번만 호출할 수 있습니다. 여러 번 호출하면 에러가 발생합니다.

<br>

### 3.5. Skipping `next()`

handler 이후에 실행할 코드가 필요 없다면 `next()`를 직접 호출하지 않아도 됩니다.

```ts
const authMiddleware = async ({ request, context }) => {
  const user = await getUser(request);
  if (!user) {
    throw redirect("/login");
  }
  context.set(userContext, user);
  // next() is called automatically
};
```

<br>

### 3.6. `next()` and Error Handling

React Router는 route `ErrorBoundary` export를 통한 built-in error handling을 가지고 있습니다. `action`/`loader`가 throw할 때와 마찬가지로, **`middleware`가 throw해도 적절한 `ErrorBoundary`에서 잡혀 처리되고** ancestor의 `next()` 호출을 통해 `Response`가 반환됩니다.

즉, **`next()` 함수는 throw하지 않고 항상 `Response`를 반환해야 하므로, `try/catch`로 감쌀 필요가 없습니다.**

이 동작은 root `middleware`에서 outgoing response에 필요한 header를 자동으로 설정하는 패턴, 예를 들어 session commit 같은 작업을 가능하게 하는 데 중요합니다. 만약 `middleware`의 에러로 인해 `next()`가 throw해버린다면, 되돌아오는 과정에서 ancestor middleware가 실행되지 못하고 필요한 header도 설정되지 못할 것입니다.

```ts
// routes/parent.tsx
export const middleware: Route.MiddlewareFunction[] = [
  async (_, next) => {
    let res = await next();
    //  ^ res.status = 500
    // This response contains the ErrorBoundary
    return res;
  },
];

// routes/parent.child.tsx
export const middleware: Route.MiddlewareFunction[] = [
  async (_, next) => {
    let res = await next();
    //  ^ res.status = 200
    // This response contains the successful UI render
    throw new Error("Uh oh, something went wrong!");
  },
];
```

어떤 `ErrorBoundary`가 렌더링되는지는 `next()`를 호출하기 전에 throw했는지, 호출한 뒤에 throw했는지에 따라 달라집니다. `next()` 호출 후에 throw하면 loader가 throw한 것과 같은 방식으로 그 route에서 위로 bubble up합니다. 이 시점에는 이미 loader가 실행되었고 route component를 렌더링하는 데 필요한 `loaderData`도 존재하기 때문입니다.

반면 `next()` 호출 전에 에러를 던지면 아직 어떤 loader도 실행되지 않았고 `loaderData`도 없습니다. 이 경우에는 `loader`를 가진 가장 높은 route까지 거슬러 올라간 다음, 그 지점부터 `ErrorBoundary`를 찾기 시작해야 합니다. `loaderData`가 없기 때문에 그 레벨 이하의 route component는 렌더링할 수 없습니다.
