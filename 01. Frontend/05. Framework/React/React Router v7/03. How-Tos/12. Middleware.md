# Middleware

> Framework Mode에서는 middleware가 `getLoadContext` 함수와 loader/action의 `context` 파라미터에 minor breaking change를 포함하고 있기 때문에 `future.v8_middleware` 플래그로 명시적으로 활성화해야 합니다.

Middleware는 매칭된 경로의 Response 생성 전후에 코드를 실행할 수 있게 해줍니다. 이를 통해 인증, 로깅, 에러 처리, 데이터 전처리 같은 공통 패턴을 재사용 가능한 방식으로 구현할 수 있습니다.

Middleware는 중첩된 체인으로 실행됩니다. route handler로 내려갈 때는 parent route에서 child route 순서로 실행되고, Response가 생성된 뒤에는 child route에서 parent route 순서로 다시 올라옵니다.

예를 들어 `GET /parent/child` 요청에서는 다음 순서로 실행됩니다.

    - Root middleware start
      - Parent middleware start
        - Child middleware start
          - Run loaders, generate HTML Response
        - Child middleware end
      - Parent middleware end
    - Root middleware end

>  서버의 middleware(Framework Mode)와 클라이언트의 middleware(Framework/Data Mode) 사이에는 약간의 차이가 있습니다. 이 문서의 대부분의 예시는 다른 HTTP 서버의 middleware에 익숙한 사용자를 위해 Server Middleware를 기준으로 설명합니다.

<br>

## 1. Quick Start (Framework mode)

### 1.1. Enable the middleware flag

먼저 React Router config에서 middleware를 활성화합니다.

```ts
// react-router.config.ts
import type { Config } from "@react-router/dev/config";

export default {
  future: {
    v8_middleware: true,
  },
} satisfies Config;
```

>  middleware 기능을 활성화하면 `action`과 `loader`의 `context` 파라미터 타입이 바뀝니다. 현재 `context`를 사용 중이라면 아래의 `getLoadContext` 섹션을 주의해서 봐야 합니다.

<br>

### 1.2. Create a context

Middleware는 middleware 체인 아래로 데이터를 전달하기 위해 `context` provider 인스턴스를 사용합니다. `createContext`를 사용하면 타입 안전한 context 객체를 만들 수 있습니다.

```ts
import { createContext } from "react-router";
import type { User } from "~/types";

export const userContext = createContext<User | null>(null);
```

<br>

### 1.3. Export middleware from your routes

```ts
import { redirect } from "react-router";
import { userContext } from "~/context";

// Server-side Authentication Middleware
async function authMiddleware({ request, context }) {
  const user = await getUserFromSession(request);
  if (!user) {
    throw redirect("/login");
  }
  context.set(userContext, user);
}

export const middleware: Route.MiddlewareFunction[] = [
  authMiddleware,
];

// Client-side timing middleware
async function timingMiddleware({ context }, next) {
  const start = performance.now();
  await next();
  const duration = performance.now() - start;
  console.log(`Navigation took ${duration}ms`);
}

export const clientMiddleware: Route.ClientMiddlewareFunction[] =
  [timingMiddleware];

export async function loader({
  context,
}: Route.LoaderArgs) {
  const user = context.get(userContext);
  const profile = await getProfile(user);
  return { profile };
}

export default function Dashboard({
  loaderData,
}: Route.ComponentProps) {
  return (
    <div>
      <h1>Welcome {loaderData.profile.fullName}!</h1>
      <Profile profile={loaderData.profile} />
    </div>
  );
}
```

<br>

### 1.4. Update your `getLoadContext` function (if applicable)

custom server와 `getLoadContext` 함수를 사용 중이라면, JavaScript 객체 대신 `RouterContextProvider` 인스턴스를 반환하도록 구현을 바꿔야 합니다.

```ts
import {
 createContext,
  RouterContextProvider,
} from "react-router";
import { createDb } from "./db";

+const dbContext = createContext<Database>();

function getLoadContext(req, res) {
  //  return { db: createDb() };
  const context = new RouterContextProvider();
  context.set(dbContext, createDb());
  return context;
}
```

## 3. Core Concepts

### 3.1. Server vs Client Middleware

Server middleware는 Framework mode의 서버에서 HTML Document 요청과 이후 navigation/fetcher 호출을 위한 `.data` 요청에 대해 실행됩니다. 서버 middleware는 HTTP Request에 대한 응답으로 실행되므로, `next` 함수를 통해 middleware 체인을 따라 다시 HTTP Response를 위로 전달합니다.

```ts
async function serverMiddleware({ request }, next) {
  console.log(request.method, request.url);
  let response = await next();
  console.log(response.status, request.method, request.url);
  return response;
}

// Framework mode only
export const middleware: Route.MiddlewareFunction[] = [
  serverMiddleware,
];
```

<br>

Client middleware는 Framework mode와 Data mode의 브라우저에서 client-side navigation과 fetcher 호출에 대해 실행됩니다. 클라이언트에는 HTTP Request가 없기 때문에, server middleware처럼 다시 bubble up할 `Response`가 없습니다. 대부분의 경우 클라이언트에서는 `next`의 반환값을 무시하고 middleware에서 아무것도 반환하지 않아도 됩니다.

```ts
async function clientMiddleware({ request }, next) {
  console.log(request.method, request.url);
  await next();
  console.log(response.status, request.method, request.url);
}

// Framework mode
export const clientMiddleware: Route.ClientMiddlewareFunction[] =
  [clientMiddleware];

// Or, Data mode
const route = {
  path: "/",
  middleware: [clientMiddleware],
  loader: rootLoader,
  Component: Root,
};
```

<br>

경우에 따라 loader/action 결과를 바탕으로 후처리를 하고 싶을 수 있습니다. 클라이언트 middleware에서는 `Response` 대신 현재 활성화된 `dataStrategy`가 반환한 값(`Record<string, DataStrategyResult>`, route id 기준 key)을 위로 전달합니다. 이를 통해 실행된 `loader`/`action`의 결과를 기준으로 조건부 로직을 수행할 수 있습니다.

다음은 client-side middleware로 구현한 CMS Redirect on 404 예시입니다.

```ts
async function cmsFallbackMiddleware({ request }, next) {
  const results = await next();

  // Check if we got a 404 from any of our routes and if so, look for a
  // redirect in our CMS
  const found404 = Object.values(results).some(
    (r) =>
      isRouteErrorResponse(r.result) &&
      r.result.status === 404,
  );

  if (found404) {
    const cmsRedirect = await checkCMSRedirects(
      request.url,
    );
    if (cmsRedirect) {
      throw redirect(cmsRedirect, 302);
    }
  }
}
```

server middleware에서는 `Response` body를 만지면 안 되고, status/headers를 읽거나 header를 설정하는 정도만 해야 합니다. 클라이언트 middleware에서도 이 값은 resulting navigation의 "body" 또는 "data"를 나타내므로 **read-only**로 간주해야 합니다. navigation의 결과는 middleware가 아니라 loader/action이 결정해야 하기 때문입니다. 그래서 클라이언트 middleware에서는 `await next()`의 결과를 읽어야 했더라도 다시 반환할 필요가 없는 경우가 대부분입니다.

<br>

### 3.2. When Middleware Runs

Middleware가 언제 실행되는지를 이해하는 것은 애플리케이션이 의도대로 동작하는지 판단하는 데 매우 중요합니다.

<br>

#### 3.2.1. Server Middleware

hydrated Framework Mode 앱에서 server middleware는 기본적으로 SPA 동작을 우선시하도록 설계되어 있어서 불필요한 네트워크 요청을 만들지 않습니다. middleware는 기존 요청을 감싸며, 실제로 서버에 접근해야 할 때만 실행됩니다.

여기서 React Router에서 "handler"가 무엇인지가 문제가 됩니다. route인지, `loader`인지에 대해서는 상황에 따라 다릅니다.

- document 요청(`GET /route`)에서는 response가 `loader`와 route component를 모두 포함하므로 handler는 route입니다.
- client-side navigation의 data 요청(`GET /route.data`)에서는 response에 포함되는 것이 `action`/`loader`뿐이므로 handler는 `action`/`loader`입니다.

따라서:

- document 요청은 `loader`가 없어도 UI를 렌더링하는 "handler" 안에 있으므로 server middleware가 실행됩니다.
- client-side navigation은 서버에 `.data` 요청이 발생해 `action`/`loader`를 실행할 때만 server middleware가 실행됩니다.

<br>

이 동작은 request annotation 계열 middleware, 예를 들어 요청 시간 로깅, session 확인/설정, outgoing caching header 설정 등에 중요합니다. 애초에 서버에 갈 이유가 없는데 이런 middleware를 실행하려고 서버로 가는 것은 무의미하며, 서버 부하와 noisy log만 증가시킵니다.

    // This middleware won't run on client-side navigations without a `.data` request
    async function loggingMiddleware({ request }, next) {
      console.log(`Request: ${request.method} ${request.url}`);
      let response = await next();
      console.log(
        `Response: ${response.status} ${request.method} ${request.url}`,
      );
      return response;
    }
    
    export const middleware: Route.MiddlewareFunction[] = [
      loggingMiddleware,
    ];

반대로 특정 route에 `loader`가 없어도 모든 client-navigation마다 어떤 server middleware를 실행하고 싶은 경우가 있을 수 있습니다. 예를 들어 인증이 필요한 섹션의 form 페이지에서 사용자가 form을 작성한 뒤 `action` 제출 시점이 아니라, 페이지에 진입할 때 auth middleware로 미리 리다이렉트하고 싶을 수 있습니다.

이런 경우에는 해당 middleware가 있는 route에 `loader`를 추가하면, 그 route가 포함된 모든 client-side navigation에서 항상 서버를 호출하게 만들 수 있습니다.

    function authMiddleware({ request }, next) {
      if (!isLoggedIn(request)) {
        throw redirect("/login");
      }
    }
    
    export const middleware: Route.MiddlewareFunction[] = [
      authMiddleware,
    ];
    
    // By adding a `loader`, we force the `authMiddleware` to run on every
    // client-side navigation involving this route.
    export async function loader() {
      return null;
    }

<br>

#### 3.2.2. Client Middleware

client middleware는 더 단순합니다. 이미 클라이언트에 있고, navigation 시 항상 router에 대한 "request"를 만들기 때문입니다. client middleware는 실행할 `loader`가 있는지와 상관없이 모든 client navigation에서 실행됩니다.

<br>

### 3.3. Context API

새로운 context 시스템은 타입 안전성을 제공하고 naming conflict를 방지하며, nested middleware와 `action`/`loader` 함수에 데이터를 전달할 수 있게 해줍니다.

    // ✅ Type-safe
    import { createContext } from "react-router";
    const userContext = createContext<User>();
    
    // Later in middleware/`loader`s
    context.set(userContext, user); // Must be `User` type
    const user = context.get(userContext); // Returns `User` type
    
    // ❌ Old way (no type safety)
    context.user = user; // Could be anything

<br>

#### 3.3.1. `Context` and `AsyncLocalStorage`

Node는 비동기 실행 컨텍스트를 따라 값을 전달할 수 있는 `AsyncLocalStorage` API를 제공합니다. 이 API는 Node 전용이지만, 대부분의 현대 런타임도 대부분 지원합니다.

이론상 middleware에서 child route로 값을 넘기는 방식으로 `AsyncLocalStorage`를 직접 사용할 수도 있었지만, 100% 크로스플랫폼 호환성이 보장되지 않는 점 때문에 런타임에 독립적으로 동작하는 재사용 가능한 middleware 패키지를 만들 수 있도록 first-class `context` API를 제공하기로 했습니다.

그렇더라도 이 API는 React Router middleware와 함께 매우 잘 동작하며, `context` API를 대체하거나 함께 사용할 수도 있습니다.

<br>

### 3.4. The `next` function

`next` 함수의 동작은 어떤 route middleware에서 호출되었는지에 따라 달라집니다.

- leaf가 아닌 middleware에서 호출되면 체인상의 다음 middleware를 실행합니다.
- leaf middleware에서 호출되면 route handler를 실행하고 요청에 대한 최종 `Response`를 생성합니다.


```ts
const middleware = async ({ context }, next) => {
  // Code here runs BEFORE handlers
  console.log("Before");

  const response = await next();

  // Code here runs AFTER handlers
  console.log("After");

  return response; // Optional on client, required on server
};
```



각 middleware에서는 `next()`를 한 번만 호출할 수 있습니다. 여러 번 호출하면 에러가 발생합니다.

<br>

### 3.5. Skipping `next()`

handler 이후에 실행할 코드가 필요 없다면 `next()`를 직접 호출하지 않아도 됩니다.

```ts
const authMiddleware = async ({ request, context }) => {
  const user = await getUser(request);
  if (!user) {
    throw redirect("/login");
  }
  context.set(userContext, user);
  // next() is called automatically
};
```

<br>

### 3.6. `next()` and Error Handling

React Router는 route `ErrorBoundary` export를 통한 built-in error handling을 가지고 있습니다. `action`/`loader`가 throw할 때와 마찬가지로, `middleware`가 throw해도 적절한 `ErrorBoundary`에서 잡혀 처리되고 ancestor의 `next()` 호출을 통해 `Response`가 반환됩니다.

즉, `next()` 함수는 throw하지 않고 항상 `Response`를 반환해야 하므로, `try/catch`로 감쌀 필요가 없습니다.

이 동작은 root `middleware`에서 outgoing response에 필요한 header를 자동으로 설정하는 패턴, 예를 들어 session commit 같은 작업을 가능하게 하는 데 중요합니다. 만약 `middleware`의 에러로 인해 `next()`가 throw해버린다면, 되돌아오는 과정에서 ancestor middleware가 실행되지 못하고 필요한 header도 설정되지 못할 것입니다.

```ts
// routes/parent.tsx
export const middleware: Route.MiddlewareFunction[] = [
  async (_, next) => {
    let res = await next();
    //  ^ res.status = 500
    // This response contains the ErrorBoundary
    return res;
  },
];

// routes/parent.child.tsx
export const middleware: Route.MiddlewareFunction[] = [
  async (_, next) => {
    let res = await next();
    //  ^ res.status = 200
    // This response contains the successful UI render
    throw new Error("Uh oh, something went wrong!");
  },
];
```

어떤 `ErrorBoundary`가 렌더링되는지는 `next()`를 호출하기 전에 throw했는지, 호출한 뒤에 throw했는지에 따라 달라집니다. `next()` 호출 후에 throw하면 loader가 throw한 것과 같은 방식으로 그 route에서 위로 bubble up합니다. 이 시점에는 이미 loader가 실행되었고 route component를 렌더링하는 데 필요한 `loaderData`도 존재하기 때문입니다.

반면 `next()` 호출 전에 에러를 던지면 아직 어떤 loader도 실행되지 않았고 `loaderData`도 없습니다. 이 경우에는 `loader`를 가진 가장 높은 route까지 거슬러 올라간 다음, 그 지점부터 `ErrorBoundary`를 찾기 시작해야 합니다. `loaderData`가 없기 때문에 그 레벨 이하의 route component는 렌더링할 수 없습니다.

<br>

## 4. Changes to `getLoadContext`/`AppLoadContext`

이 섹션은 custom server와 custom `getLoadContext` 함수를 사용하는 경우에만 해당합니다.

Middleware는 `getLoadContext`가 생성하고 `action`/`loader`에 전달하는 `context` 파라미터에 breaking change를 도입합니다. 기존의 module augmentation 기반 `AppLoadContext` 방식은 실질적으로 타입 안전하지 않고 TypeScript에게 "그냥 믿어"라고 말하는 수준이었습니다.

Middleware는 클라이언트의 `clientMiddleware`에서도 동일한 성격의 `context`가 필요하지만, 서버에서 이미 만족스럽지 않았던 패턴을 클라이언트에 복제하고 싶지는 않았기 때문에, 타입 안전성을 확보할 수 있는 새로운 API가 도입되었습니다.

middleware를 opt-in하면 `context` 파라미터는 `RouterContextProvider` 인스턴스로 바뀝니다.

```ts
let dbContext = createContext<Database>();
let context = new RouterContextProvider();
context.set(dbContext, getDb());
//                     ^ type-safe
let db = context.get(dbContext);
//  ^ Database
```

<br>

### 4.1. Migration from `AppLoadContext`

현재 `AppLoadContext`를 사용 중이라면, 기존 module augmentation을 `AppLoadContext` 대신 `RouterContextProvider`에 적용하는 방식으로 점진적으로 마이그레이션할 수 있습니다. 그 다음 `getLoadContext`가 `RouterContextProvider` 인스턴스를 반환하도록 바꾸면 됩니다.

```ts
declare module "react-router" {
-  interface AppLoadContext {
+  interface RouterContextProvider {
    db: Database;
    user: User;
  }
}

function getLoadContext() {
  const loadContext = {...};
-  return loadContext;
+  let context = new RouterContextProvider();
+  Object.assign(context, loadContext);
+  return context;
}
```

이렇게 하면 middleware를 처음 도입할 때는 `action`/`loader`를 수정하지 않고도 그대로 둘 수 있습니다. 기존처럼 `context.db`처럼 직접 값을 읽을 수 있기 때문입니다.

다만 이 방식은 **React Router v7에서 middleware를 도입하면서 `context.set`/`context.get`로 점진적으로 옮겨가기 위한 migration strategy**일 뿐입니다. 다음 major version에서도 계속 동작한다고 가정하면 안 됩니다.

`RouterContextProvider` 클래스는 클라이언트의 `<HydratedRouter getContext>`와 `<RouterProvider getContext>`의 `context` 파라미터에도 사용됩니다.

`AppLoadContext`는 본래 HTTP 서버에서 React Router handler로 값을 넘겨주는 목적이 강하기 때문에, 이렇게 augmentation한 필드는 `clientMiddleware`, `clientLoader`, `clientAction`에서는 사용할 수 없다는 점을 알아야 합니다. 다만 클라이언트에서도 `getContext`를 통해 직접 값을 제공했다면 예외입니다.

## 5. Common Patterns

### 5.1. Authentication

```ts
import { redirect } from "react-router";
import { userContext } from "~/context";
import { getSession } from "~/sessions.server";

export const authMiddleware = async ({
  request,
  context,
}) => {
  const session = await getSession(request);
  const userId = session.get("userId");

  if (!userId) {
    throw redirect("/login");
  }

  const user = await getUserById(userId);
  context.set(userContext, user);
};

import { authMiddleware } from "~/middleware/auth";

export const middleware: Route.MiddlewareFunction[] = [
  authMiddleware,
];

export async function loader({
  context,
}: Route.LoaderArgs) {
  const user = context.get(userContext); // Guaranteed to exist
  return { user };
}
```

### 5.2. Logging

```ts
import { requestIdContext } from "~/context";

export const loggingMiddleware = async (
  { request, context },
  next,
) => {
  const requestId = crypto.randomUUID();
  context.set(requestIdContext, requestId);

  console.log(
    `[${requestId}] ${request.method} ${request.url}`,
  );

  const start = performance.now();
  const response = await next();
  const duration = performance.now() - start;

  console.log(
    `[${requestId}] Response ${response.status} (${duration}ms)`,
  );

  return response;
};
```

### 5.3. CMS Redirect on 404

```ts
export const cmsFallbackMiddleware = async (
  { request },
  next,
) => {
  const response = await next();

  // Check if we got a 404
  if (response.status === 404) {
    // Check CMS for a redirect
    const cmsRedirect = await checkCMSRedirects(
      request.url,
    );
    if (cmsRedirect) {
      throw redirect(cmsRedirect, 302);
    }
  }

  return response;
};
```
