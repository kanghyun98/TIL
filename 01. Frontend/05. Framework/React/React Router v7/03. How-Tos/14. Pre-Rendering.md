# Pre-rendering

Pre-Rendering은 런타임이 아니라 빌드 시점에 페이지를 렌더링하여 정적 콘텐츠의 페이지 로드 속도를 높일 수 있게 해줍니다.

<br>

## 1. Configuration

Pre-rendering은 `react-router.config.ts`에서 `prerender` 옵션으로 활성화됩니다.

가장 간단한 설정은 boolean `true`이며, 이는 `routes.ts`를 기준으로 애플리케이션의 모든 정적 경로를 pre-render 합니다.

    import type { Config } from "@react-router/dev/config";
    
    export default {
      prerender: true,
    } satisfies Config;

boolean `true`는 파라미터 값을 알 수 없기 때문에 동적 경로(예: `/blog/:slug`)는 포함하지 않습니다.

<br>

동적 값을 포함해 특정 경로를 설정하려면 경로 배열을 지정할 수 있습니다.

```ts
import type { Config } from "@react-router/dev/config";

let slugs = getPostSlugs();

export default {
  prerender: [
    "/",
    "/blog",
    ...slugs.map((s) => `/blog/${s}`),
  ],
} satisfies Config;
```

<br>

경로를 결정하기 위해 더 복잡한 로직이나 비동기 로직이 필요하다면, 경로 배열을 반환하는 함수를 제공할 수도 있습니다. 이 함수는 애플리케이션의 모든 정적 경로를 직접 추가하지 않도록 `getStaticPaths` 메서드를 제공합니다.

```ts
import type { Config } from "@react-router/dev/config";

export default {
  async prerender({ getStaticPaths }) {
    let slugs = await getPostSlugsFromCMS();
      return [
        ...getStaticPaths(), // "/" and "/blog"
        ...slugs.map((s) => `/blog/${s}`),
      ];
    },
} satisfies Config;
```

<br>

### 1.1. Concurrency (unstable)

> 이 API는 실험적이며, minor/patch 릴리스에서 호환성이 깨지는 변경이 있을 수 있습니다. 주의해서 사용하고, 관련 변경 사항이 있는 릴리스 노트를 매우 주의 깊게 확인해야 합니다.

기본적으로 페이지는 한 번에 하나의 경로씩 pre-render 됩니다. concurrency를 활성화하면 여러 경로를 병렬로 pre-render 할 수 있으며, 많은 경우 빌드 시간을 단축할 수 있습니다. 애플리케이션에 가장 좋은 성능을 제공하는 값을 직접 실험해봐야 합니다.

concurrency를 지정하려면 `prerender` config를 `prerender.paths` 필드로 내리고, `prerender.unstable_concurrency`에 값을 지정하면 됩니다.

```ts
import type { Config } from "@react-router/dev/config";

let slugs = getPostSlugs();

export default {
  prerender: {
    paths: [
      "/",
      "/blog",
      ...slugs.map((s) => `/blog/${s}`),
    ],
    unstable_concurrency: 4, // 4개의 경로 동시 처리
  },
} satisfies Config;
```

<br>

## 2. Pre-Rendering with/without a Runtime Server

Pre-Rendering은 `ssr` config 값에 따라 두 가지 방식으로 사용할 수 있습니다:

- `ssr:true`(기본값)와 함께 런타임 SSR 서버를 사용하는 방식

- `ssr:false`로 정적 파일 서버에 배포하는 방식

<br>

### 2.1. Pre-rendering with `ssr:true`

`ssr:true`에서 pre-rendering을 사용한다는 것은 런타임 서버는 계속 유지하되, 더 빠른 응답 시간을 위해 특정 경로를 pre-render 하겠다는 의미입니다.

```ts
import type { Config } from "@react-router/dev/config";

export default {
  // Can be omitted - defaults to true
  ssr: true,
  prerender: ["/", "/blog", "/blog/popular-post"],
} satisfies Config;
```

<br>

#### 2.1.1. Data Loading and Pre-rendering

Pre-rendering을 위한 별도의 애플리케이션 API는 없습니다. pre-render 되는 라우트는 서버 렌더링과 동일한 route `loader` 함수를 사용합니다.

```ts
export async function loader({ request, params }) {
  let post = await getPost(params.slug);
  return post;
}

export function Post({ loaderData }) {
  return <div>{loaderData.title}</div>;
}
```

배포된 서버에서 요청이 라우트로 들어오는 대신, 빌드 과정에서 `new Request()`를 생성하고 서버가 하듯이 앱을 통과시켜 실행합니다.

서버 렌더링 시에는 pre-render 되지 않은 경로에 대한 요청도 평소처럼 서버 렌더링됩니다.

<br>

#### 2.1.2. Static File Output

렌더링 결과는 `build/client` 디렉터리에 기록됩니다. 각 경로마다 두 개의 파일이 생성됩니다:

- 초기 document 요청을 위한 `[url].html` HTML 파일

- 브라우저의 클라이언트 사이드 네비게이션 요청을 위한 `[url].data` 파일

<br>

빌드 출력에는 어떤 파일이 pre-render 되었는지가 표시됩니다.

```
> react-router build
vite v5.2.11 building for production...
...
vite v5.2.11 building SSR bundle for production...
...
Prerender: Generated build/client/index.html
Prerender: Generated build/client/blog.data
Prerender: Generated build/client/blog/index.html
Prerender: Generated build/client/blog/my-first-post.data
Prerender: Generated build/client/blog/my-first-post/index.html
...
```

개발 중에는 pre-rendering 결과를 public 디렉터리에 저장하지 않으며, 이 동작은 `react-router build`에서만 수행됩니다.

<br>

### 2.2. Pre-rendering with `ssr:false`

위 예제들은 런타임 서버를 배포하면서, 일부 정적 페이지를 pre-render 하여 서버를 거치지 않게 하고 더 빠른 로드를 얻는 경우를 가정합니다.

런타임 SSR을 비활성화하고 정적 파일 서버에서 제공되도록 pre-rendering을 설정하려면 `ssr:false` config 플래그를 설정할 수 있습니다.

```ts
import type { Config } from "@react-router/dev/config";

export default {
  ssr: false, // disable runtime server rendering
  prerender: true, // pre-render all static routes
} satisfies Config;
```

`ssr:false`를 `prerender` config 없이 지정하면, React Router는 이를 SPA Mode라고 부릅니다. SPA Mode에서는 애플리케이션의 어떤 경로에서도 hydration 할 수 있는 단일 HTML 파일을 렌더링합니다. 이것이 가능한 이유는 HTML 파일 안에는 `root` 라우트만 렌더링하고, hydration 중 브라우저 URL을 기준으로 어떤 child route를 로드할지 결정하기 때문입니다.

즉, `root` 라우트에는 `loader`를 사용할 수 있지만, 다른 라우트에는 사용할 수 없습니다. 어떤 라우트를 로드해야 하는지 브라우저 hydration 전까지 알 수 없기 때문입니다.

<br>

`ssr:false`에서 경로를 pre-render 하려는 경우, 해당 경로에 매칭되는 라우트는 `loader`를 가질 수 있습니다. 이 경우 `root`만이 아니라 해당 경로에 매칭되는 모든 라우트를 pre-render 하기 때문입니다. `ssr:false`가 설정되면 어떤 라우트에서도 `actions` 또는 `headers` 함수를 포함할 수 없습니다. 이를 실행할 런타임 서버가 없기 때문입니다.

<br>

#### 2.2.1. Pre-rendering with a SPA Fallback

`ssr:false`를 사용하되 모든 라우트를 pre-render 하고 싶지 않은 경우도 가능합니다. 어떤 경로는 pre-rendering의 성능/SEO 이점이 필요하지만, 다른 페이지는 SPA로 충분할 수 있습니다.

이 역시 config 조합으로 할 수 있습니다. pre-render 하려는 경로만 `prerender` config에 제한하면, React Router는 다른 경로를 hydrate 하기 위해 "SPA Fallback" HTML 파일도 함께 출력합니다. 이 방식은 SPA Mode와 동일한 접근을 사용합니다.

이 파일은 다음 경로 중 하나에 기록됩니다:

- `/` 경로를 pre-render 하지 않은 경우: `build/client/index.html`

- `/` 경로를 pre-render 한 경우: `build/client/__spa-fallback.html`

```ts
import type { Config } from "@react-router/dev/config";

export default {
  ssr: false,

  // SPA fallback will be written to build/client/index.html
  prerender: ["/about-us"],

  // SPA fallback will be written to build/client/__spa-fallback.html
  prerender: ["/", "/about-us"],
} satisfies Config;
```

<br>

#### 2.2.2. Invalid Exports

`ssr:false`에서 pre-rendering을 사용할 때 React Router는 쉽게 놓칠 수 있는 실수를 방지하기 위해, 유효하지 않은 export가 있으면 빌드 시점에 에러를 발생시킵니다.

- 런타임 서버가 없기 때문에 모든 라우트에서 `headers`/`action` 함수는 금지됩니다

- `prerender` config 없이 `ssr:false`를 사용하는 경우 (SPA Mode), `loader`는 `root` 라우트에만 허용됩니다

- `prerender` config와 함께 `ssr:false`를 사용하는 경우, `loader`는 `prerender` 경로에 매칭되는 어떤 라우트에서도 허용됩니다

  - child route를 가진 pre-rendered route에서 `loader`를 사용하고 있다면, 다음 중 하나를 통해 parent `loaderData`를 런타임에 올바르게 결정할 수 있도록 해야 합니다:

    - 모든 child route를 pre-render 해서, 각 child route 경로에 대해 빌드 시점에 parent `loader`가 호출되고 `.data` 파일에 렌더링되도록 하기

    - pre-render 되지 않은 child 경로에 대해 런타임에 호출될 수 있는 `clientLoader`를 parent에 사용하기
