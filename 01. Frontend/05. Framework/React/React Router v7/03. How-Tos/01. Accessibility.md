# Accessibility

React Router 애플리케이션에서 접근성(accessibility)은 일반적인 웹 접근성과 크게 다르지 않습니다.  
적절한 **시맨틱 마크업(semantic markup)**을 사용하고 **WCAG(Web Content Accessibility Guidelines)**를 따르는 것만으로도 대부분의 접근성 요구 사항을 충족할 수 있습니다.

React Router는 가능한 경우 접근성 모범 사례를 기본 동작으로 제공하며, 필요한 경우 이를 구현할 수 있는 API도 제공합니다.

<br>

## 1. Links

### 1.1 `<Link>` 컴포넌트

React Router의 `<Link>` 컴포넌트는 **표준 HTML `<a>` 태그를 렌더링합니다.**  
즉, 브라우저가 기본적으로 제공하는 접근성 동작을 그대로 활용할 수 있습니다.

이 덕분에 다음과 같은 기능을 별도의 구현 없이 사용할 수 있습니다.

- 키보드 네비게이션
- 스크린 리더 지원
- 브라우저 기본 접근성 동작

<br>

### 1.2 `<NavLink>` 컴포넌트

React Router는 `<NavLink>` 컴포넌트도 제공합니다.

`<NavLink>`는 `<Link>`와 동일하게 동작하지만, **현재 페이지를 가리키는 링크인지 여부를 보조 기술(assistive technology)에 전달하는 추가 정보를 제공합니다.**

이 기능은 다음과 같은 UI에서 특히 유용합니다.

- 네비게이션 메뉴
- breadcrumb
- 현재 위치 표시

<br>

## 2. Routing

클라이언트 사이드 라우팅이 활성화되면 접근성 측면에서 몇 가지 중요한 사항을 고려해야 합니다.

전통적인 **멀티 페이지 웹사이트**에서는 다음과 같이 동작합니다.

1. 앱이 `<a>` 태그를 렌더링  
2. 브라우저가 페이지 이동을 처리  
3. 페이지 전체가 새로 로드  

또한 사용자가 **JavaScript를 비활성화하더라도 React Router 애플리케이션은 기본적으로 이 방식으로 동작해야 합니다.**

하지만 클라이언트 스크립트가 로드되면 React Router가 라우팅을 제어하고 브라우저의 기본 동작을 막습니다.

이때 React Router는 **라우트 변경 시 UI에 대해 아무런 가정을 하지 않기 때문에** 개발자가 접근성을 직접 고려해야 합니다.

<br>

### 2.1 Focus Management

라우트가 변경될 때 **어떤 요소가 focus를 받아야 하는지 결정해야 합니다.**

이는 특히 다음 사용자에게 중요합니다.

- 키보드 사용자
- 스크린 리더 사용자

적절한 포커스 관리는 페이지 전환 시 사용자 경험을 크게 개선합니다.

<br>

### 2.2 Live-Region Announcements

스크린 리더 사용자에게는 **라우트 변경 사실을 알리는 것이 도움이 됩니다.**

예를 들어 다음 상황에서 알림을 제공할 수 있습니다.

- 라우트 변경
- 전환 상태
- 로딩 시간이 긴 경우

이러한 알림은 **ARIA live region**을 통해 구현할 수 있습니다.
