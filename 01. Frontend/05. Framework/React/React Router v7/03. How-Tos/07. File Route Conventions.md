# File Route Conventions

파일 기반 라우팅은 라우트 구성을 코드가 아니라 **파일 이름 규칙**으로 표현하는 방식입니다.  
이 방식을 사용하면 `@react-router/fs-routes` 패키지가 파일 구조를 읽어 라우트 설정으로 변환해 줍니다.

즉, `app/routes` 아래의 파일과 폴더 구조가 그대로 URL 구조와 중첩 레이아웃 구조로 이어집니다.

<br>

## 1. Setting up

먼저 `@react-router/fs-routes` 패키지를 설치합니다.

```bash
npm i @react-router/fs-routes
```



그 다음 `app/routes.ts`에서 `flatRoutes()`를 사용해 라우트 구성을 제공합니다.

```ts
import { type RouteConfig } from "@react-router/dev/routes";
import { flatRoutes } from "@react-router/fs-routes";

export default flatRoutes() satisfies RouteConfig;
```



기본적으로 `app/routes` 디렉토리 안의 모든 모듈이 애플리케이션의 라우트가 됩니다.

특정 파일을 라우트에서 제외하고 싶다면 `ignoredRouteFiles` 옵션을 사용할 수 있습니다.

```ts
import { type RouteConfig } from "@react-router/dev/routes";
import { flatRoutes } from "@react-router/fs-routes";

export default flatRoutes({
  ignoredRouteFiles: ["home.tsx"],
}) satisfies RouteConfig;
```

기본 라우트 디렉토리는 `app/routes`이지만, `rootDirectory` 옵션으로 변경할 수도 있습니다.

```ts
import { type RouteConfig } from "@react-router/dev/routes";
import { flatRoutes } from "@react-router/fs-routes";

export default flatRoutes({
  rootDirectory: "file-routes",
}) satisfies RouteConfig;
```

이 글에서는 이후 예시를 모두 기본 디렉토리인 `app/routes` 기준으로 설명합니다.

<br>

## 2. Basic Routes

기본적으로 **파일 이름이 그대로 URL pathname에 매핑**됩니다.  
예외는 `_index.tsx`로, 이 파일은 루트 라우트의 index route가 됩니다.

사용할 수 있는 확장자는 다음과 같습니다.

- `.js`, `.jsx`, `.ts`, a`.tsx`

```txt
app/
├── routes/
│   ├── _index.tsx
│   └── about.tsx
└── root.tsx
```

매칭 결과는 다음과 같습니다.

- `/` → `app/routes/_index.tsx`
- `/about` → `app/routes/about.tsx`

이 라우트들은 중첩 라우팅 규칙에 따라 `app/root.tsx`의 `<Outlet />` 안에서 렌더링됩니다.

<br>

## 3. Dot Delimiters

파일 이름에 `.`을 추가하면 URL에서는 `/`로 해석됩니다.

```txt
app/
├── routes/
│   ├── _index.tsx
│   ├── about.tsx
│   ├── concerts.trending.tsx
│   ├── concerts.salt-lake-city.tsx
│   └── concerts.san-diego.tsx
└── root.tsx
```

매칭 결과는 다음과 같습니다.

- `/` → `app/routes/_index.tsx`
- `/about` → `app/routes/about.tsx`
- `/concerts/trending` → `app/routes/concerts.trending.tsx`
- `/concerts/salt-lake-city` → `app/routes/concerts.salt-lake-city.tsx`
- `/concerts/san-diego` → `app/routes/concerts.san-diego.tsx`

또한 dot delimiter는 **URL 구분뿐 아니라 중첩 구조도 함께 만듭니다.**

<br>

## 4. Dynamic Segments

실제 애플리케이션의 URL은 정적이지 않고 데이터 기반인 경우가 많습니다.  
Dynamic segment를 사용하면 URL의 일부를 매칭하고, 그 값을 코드에서 사용할 수 있습니다.

Dynamic segment는 `$` prefix로 만듭니다.

```txt
app/
├── routes/
│   ├── _index.tsx
│   ├── about.tsx
│   ├── concerts.$city.tsx
│   └── concerts.trending.tsx
└── root.tsx
```

매칭 결과는 다음과 같습니다.

- `/` → `app/routes/_index.tsx`
- `/about` → `app/routes/about.tsx`
- `/concerts/trending` → `app/routes/concerts.trending.tsx`
- `/concerts/salt-lake-city` → `app/routes/concerts.$city.tsx`
- `/concerts/san-diego` → `app/routes/concerts.$city.tsx`

이 값은 URL에서 파싱되어 다양한 API에 전달됩니다.  
이 값을 **URL Parameters**라고 부릅니다.

가장 유용하게 접근하는 위치는 `loader`와 `action`입니다.

```ts
export async function loader({ params }) {
  return fakeDb.getAllConcertsForCity(params.city);
}
```

파일 이름의 `$city.tsx`는 그대로 `params.city`가 됩니다.

또한 여러 dynamic segment를 함께 사용할 수도 있습니다.

```ts
export async function loader({ params }) {
  return fake.db.getConcerts({
    date: params.date,
    city: params.city,
  });
}
```

예를 들어 `concerts.$city.$date`라면 `params.city`, `params.date`로 접근합니다.

<br>

## 5. Nested Routes

Nested routing은 **URL의 세그먼트 구조를 컴포넌트 계층과 데이터 구조에 연결하는 방식**입니다.

중첩 라우트는 dot delimiter를 통해 만듭니다.  
`.` 앞의 파일명이 다른 라우트 파일과 일치하면 자동으로 해당 부모 라우트의 자식이 됩니다.

```txt
app/
├── routes/
│   ├── _index.tsx
│   ├── about.tsx
│   ├── concerts._index.tsx
│   ├── concerts.$city.tsx
│   ├── concerts.trending.tsx
│   └── concerts.tsx
└── root.tsx
```

`app/routes/concerts.`로 시작하는 모든 파일은 `app/routes/concerts.tsx`의 자식 라우트가 되며, 부모 라우트의 `<Outlet />` 안에서 렌더링됩니다.

매칭 결과는 다음과 같습니다.

- `/` → `app/routes/_index.tsx` → `app/root.tsx`
- `/about` → `app/routes/about.tsx` → `app/root.tsx`
- `/concerts` → `app/routes/concerts._index.tsx` → `app/routes/concerts.tsx`
- `/concerts/trending` → `app/routes/concerts.trending.tsx` → `app/routes/concerts.tsx`
- `/concerts/salt-lake-city` → `app/routes/concerts.$city.tsx` → `app/routes/concerts.tsx`

중첩 라우트를 만들 때는 보통 부모 URL에 직접 접근했을 때도 무언가 렌더링될 수 있도록 index route를 함께 추가합니다.

예를 들어 URL이 `/concerts/salt-lake-city`라면 UI 계층은 다음과 같습니다.

```txt
<Root>
  <Concerts>
    <City />
  </Concerts>
</Root>
```

<br>

## 6. Nested URLs without Layout Nesting

URL은 중첩되지만 자동 레이아웃 중첩은 원하지 않는 경우가 있습니다.  
이럴 때는 **부모 세그먼트 뒤에 trailing underscore**를 사용해 중첩을 끌 수 있습니다.

```txt
app/
├── routes/
│   ├── _index.tsx
│   ├── about.tsx
│   ├── concerts.$city.tsx
│   ├── concerts.trending.tsx
│   ├── concerts.tsx
│   └── concerts_.mine.tsx
└── root.tsx
```

매칭 결과는 다음과 같습니다.

- `/` → `app/routes/_index.tsx` → `app/root.tsx`
- `/about` → `app/routes/about.tsx` → `app/root.tsx`
- `/concerts/mine` → `app/routes/concerts_.mine.tsx` → `app/root.tsx`
- `/concerts/trending` → `app/routes/concerts.trending.tsx` → `app/routes/concerts.tsx`
- `/concerts/salt-lake-city` → `app/routes/concerts.$city.tsx` → `app/routes/concerts.tsx`

즉 `/concerts/mine`은 더 이상 `app/routes/concerts.tsx` 아래에 중첩되지 않고 `app/root.tsx` 아래에 렌더링됩니다.

`trailing_` underscore는 **path segment는 만들지만 layout nesting은 만들지 않습니다.**

<br>

## 7. Nested Layouts without Nested URLs

이번에는 반대로 **URL 세그먼트는 추가하지 않지만 레이아웃은 공유**하고 싶은 경우입니다.  
이를 **Pathless Routes**라고 부릅니다.

대표적인 예시는 인증 관련 페이지입니다.  
예를 들어 로그인/회원가입 페이지는 퍼블릭 페이지와는 다른 header/footer를 쓰지만 URL에 별도 세그먼트를 추가하고 싶지 않을 수 있습니다.

이 경우 `_leading` underscore를 사용합니다.

```txt
app/
├── routes/
│   ├── _auth.login.tsx
│   ├── _auth.register.tsx
│   ├── _auth.tsx
│   ├── _index.tsx
│   ├── concerts.$city.tsx
│   └── concerts.tsx
└── root.tsx
```

매칭 결과는 다음과 같습니다.

- `/` → `app/routes/_index.tsx` → `app/root.tsx`
- `/login` → `app/routes/_auth.login.tsx` → `app/routes/_auth.tsx`
- `/register` → `app/routes/_auth.register.tsx` → `app/routes/_auth.tsx`
- `/concerts` → `app/routes/concerts.tsx` → `app/routes/concerts.tsx`
- `/concerts/salt-lake-city` → `app/routes/concerts.$city.tsx` → `app/routes/concerts.tsx`

`_leading` underscore는 **파일명을 URL에서 숨기고 레이아웃만 남기는 역할**을 합니다.

<br>

## 8. Optional Segments

라우트 세그먼트를 괄호로 감싸면 **optional segment**가 됩니다.

```txt
app/
├── routes/
│   ├── ($lang)._index.tsx
│   ├── ($lang).$productId.tsx
│   └── ($lang).categories.tsx
└── root.tsx
```

매칭 결과는 다음과 같습니다.

- `/` → `app/routes/($lang)._index.tsx`
- `/categories` → `app/routes/($lang).categories.tsx`
- `/en/categories` → `app/routes/($lang).categories.tsx`
- `/fr/categories` → `app/routes/($lang).categories.tsx`
- `/american-flag-speedo` → `app/routes/($lang)._index.tsx`
- `/en/american-flag-speedo` → `app/routes/($lang).$productId.tsx`
- `/fr/american-flag-speedo` → `app/routes/($lang).$productId.tsx`

여기서 `/american-flag-speedo`가 `($lang).$productId.tsx`가 아니라 `($lang)._index.tsx`에 매칭되는 이유는, optional dynamic param 뒤에 또 다른 dynamic param이 오면 단일 세그먼트 URL에서 무엇을 매칭해야 할지 안정적으로 판단할 수 없기 때문입니다.

Optional segment는 eager하게 매칭되므로 `/:lang`으로 해석됩니다.

이런 구조에서는 `($lang)._index.tsx`의 loader 안에서 `params.lang`을 확인한 뒤, 유효한 언어 코드가 아니라면 현재 기본 언어 기준 URL로 redirect하는 방식이 권장됩니다.

<br>

## 9. Splat Routes

Dynamic segment는 한 개의 path segment만 매칭하지만, **splat route는 URL의 나머지 전체를 매칭**합니다.  
여기에는 `/`도 포함됩니다.

```txt
app/
├── routes/
│   ├── _index.tsx
│   ├── $.tsx
│   ├── about.tsx
│   └── files.$.tsx
└── root.tsx
```

매칭 결과는 다음과 같습니다.

- `/` → `app/routes/_index.tsx`
- `/about` → `app/routes/about.tsx`
- `/beef/and/cheese` → `app/routes/$.tsx`
- `/files` → `app/routes/files.$.tsx`
- `/files/talks/react-conf_old.pdf` → `app/routes/files.$.tsx`
- `/files/talks/react-conf_final.pdf` → `app/routes/files.$.tsx`
- `/files/talks/react-conf-FINAL-MAY_2024.pdf` → `app/routes/files.$.tsx`

Dynamic params와 비슷하게 splat route에서도 `params["*"]`로 값을 읽을 수 있습니다.

```ts
export async function loader({ params }) {
  const filePath = params["*"];
  return fake.getFileInfo(filePath);
}
```

<br>

## 10. Catch-all Route

다른 어떤 라우트와도 매칭되지 않는 요청을 처리하는 **catch-all route**를 만들고 싶다면 `$.tsx` 파일을 routes 디렉토리 안에 만들면 됩니다.

매칭 예시는 다음과 같습니다.

- `/` → `app/routes/_index.tsx`
- `/about` → `app/routes/about.tsx`
- `/any-invalid-path-will-match` → `app/routes/$.tsx`

기본적으로 이 라우트는 200 응답을 반환하므로, 실제 404 페이지로 사용하려면 status code를 직접 바꿔주어야 합니다.

```ts
export async function loader() {
  return data({}, 404);
}
```

<br>

## 11. Escaping Special Characters

라우트 규칙에서 사용하는 특수 문자를 실제 URL의 일부로 사용하고 싶다면 `[]`로 escape할 수 있습니다.  
이 방식은 특히 URL에 확장자를 포함하는 resource route에서 유용합니다.

예시는 다음과 같습니다.

- `app/routes/sitemap[.]xml.tsx` → `/sitemap.xml`
- `app/routes/[sitemap.xml].tsx` → `/sitemap.xml`
- `app/routes/weird-url.[_index].tsx` → `/weird-url/_index`
- `app/routes/dolla-bills-[$].tsx` → `/dolla-bills-$`
- `app/routes/[[so-weird]].tsx` → `/[so-weird]`
- `app/routes/reports.$id[.pdf].ts` → `/reports/123.pdf`

<br>

## 12. Folders for Organization

라우트는 파일뿐 아니라 **폴더 + `route.tsx` 조합**으로도 구성할 수 있습니다.  
이 경우 폴더 안의 나머지 파일들은 라우트가 되지 않습니다.

이 방식의 장점은 **해당 라우트가 사용하는 코드를 같은 폴더 안에 함께 둘 수 있다는 점**입니다.  
즉 기능 이름을 다른 디렉토리에서 반복하지 않고도 코드를 더 가깝게 정리할 수 있습니다.

중요한 점은 다음과 같습니다.

- 폴더 내부 파일 이름은 라우트 path에 영향을 주지 않음
- 라우트 path는 오직 **폴더 이름**으로 결정됨

예를 들어 기존에 파일 기반으로 구성된 라우트를 다음처럼 폴더 구조로 바꿀 수 있습니다.

```txt
app/
├── routes/
│   ├── app.tsx
│   └── app.projects.tsx
```

이를 다음처럼 바꿀 수 있습니다.

```txt
app/
├── routes/
│   ├── app/
│   │   ├── footer.tsx
│   │   ├── primary-nav.tsx
│   │   └── route.tsx
│   └── app.projects/
│       ├── get-projects.server.ts
│       ├── project-buttons.tsx
│       ├── project-card.tsx
│       └── route.tsx
```

즉 다음 두 파일은 같은 라우트입니다.

```txt
app/routes/app.tsx
app/routes/app/route.tsx
```

그리고 다음 두 파일도 같은 라우트입니다.

```txt
app/routes/app._index.tsx
app/routes/app._index/route.tsx
```
