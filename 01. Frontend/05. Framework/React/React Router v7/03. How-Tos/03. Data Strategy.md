# React Router Data Strategy

React Router는 `loader`와 `action`을 통해 라우트 단위로 데이터를 처리합니다.  
기본적으로 React Router는 매칭된 라우트의 loader와 action을 자동으로 실행하며, 대부분의 애플리케이션에서는 이 기본 동작만으로 충분합니다.

하지만 다음과 같은 상황에서는 **데이터 실행 방식을 직접 제어해야 할 수도 있습니다.**

- middleware 실행
- custom context 전달
- caching layer 구현
- 로깅 및 observability 추가
- loader 실행 방식 제어

이러한 요구사항을 위해 React Router는 **`dataStrategy`라는 low-level API**를 제공합니다.  
`dataStrategy`를 사용하면 **loader와 action이 어떻게 실행될지 직접 정의할 수 있습니다.**

<br>

## 1. Usage

`dataStrategy`는 `createBrowserRouter`를 생성할 때 옵션으로 설정할 수 있습니다.

```ts
const router = createBrowserRouter(routes, {
  async dataStrategy({
    matches,
    request,
    runClientMiddleware,
    fetcherKey,
  }) {
    // custom execution logic
  },
});
```

`dataStrategy` 함수는 loader/action 실행을 제어하기 위한 여러 값을 전달합니다.

- **matches** – 현재 request와 매칭된 라우트 배열  
- **runClientMiddleware** – 매칭된 라우트에 대한 client middleware 실행 helper  
- **fetcherKey** – 요청이 navigation이 아니라 fetcher 요청일 경우 전달되는 key  

<br>

## 2. DataStrategyMatch

`matches` 배열의 각 항목은 `DataStrategyMatch` 객체입니다.  
이 객체는 loader/action 실행을 제어하기 위한 메서드를 제공합니다.

- **shouldCallHandler()** – 현재 요청에서 이 라우트의 handler(loader/action)를 실행해야 하는지 판단  
- **resolve()** – 실제 route handler(loader/action)를 실행하는 함수  

예를 들어 다음과 같은 네비게이션이 발생했다고 가정합니다.

```
/parent/child/a → /parent/child/b
```

이 경우 React Router는 다음과 같이 동작합니다.

- `parent` loader → 이미 데이터 존재 → 실행하지 않음  
- `child` loader → 이미 데이터 존재 → 실행하지 않음  
- `b` loader → 새로운 라우트 → 실행  

이처럼 필요한 handler만 실행하도록 제어할 수 있습니다.

```ts
await match.resolve();
```

`resolve()`를 호출하면 해당 라우트의 handler가 실행됩니다.

<br>

## 3. Example

다음은 handler 실행을 직접 제어하는 간단한 예시입니다.

```ts
let router = createBrowserRouter(routes, {
  async dataStrategy({
    matches,
    request,
    runClientMiddleware,
  }) {
    const matchesToLoad = matches.filter((m) =>
      m.shouldCallHandler(),
    );

    const results = {};

    await runClientMiddleware(() =>
      Promise.all(
        matchesToLoad.map(async (match) => {
          results[match.route.id] = await match.resolve();
        }),
      ),
    );

    return results;
  },
});
```

이 전략은 다음 순서로 실행됩니다.

1. 실행해야 할 라우트 선택  
2. client middleware 실행  
3. 각 라우트의 loader/action 실행  
4. 결과 반환  

<br>

## 4. Calling Route Middleware

라우트에 `middleware`가 정의되어 있는 경우  
`runClientMiddleware`를 사용해야 합니다.

```ts
await runClientMiddleware(() =>
  Promise.all(
    matchesToLoad.map(async (match) => {
      results[match.route.id] = await match.resolve();
    }),
  ),
);
```

`runClientMiddleware`는 `dataStrategy`와 동일한 인자를 받기 때문에  
독립적인 전략과 쉽게 조합할 수 있습니다.

<br>

## 5. Advanced Handler Execution

`resolve()`는 callback을 전달하여  
**handler 실행 방식을 커스터마이즈할 수 있습니다.**

```ts
await Promise.all(
  matchesToLoad.map((match) =>
    match.resolve((handler) => {
      let customContext = getCustomContext();
      return handler(customContext);
    }),
  ),
);
```

이 방식은 handler 실행 시 custom context를 전달할 때 사용할 수 있습니다.

<br>

## 6. Custom Revalidation Behavior

revalidation 동작을 변경하려면  
`match.shouldRevalidateArgs`를 사용할 수 있습니다.

```ts
const matchesToLoad = matches.filter((match) => {
  let defaultShouldRevalidate = customShouldRevalidate(
    match.shouldRevalidateArgs,
  );
  return match.shouldCallHandler(defaultShouldRevalidate);
});
```

이를 통해 loader 재실행 조건을 커스터마이즈할 수 있습니다.

<br>

## 7. Migrating away from shouldLoad

이전 API에서는 `shouldLoad`라는 boolean 값이 사용되었습니다.

하지만 이 방식은 revalidation 전략을 커스터마이즈하기 어려웠기 때문에  
현재는 **`shouldCallHandler` 사용이 권장됩니다.**

<br>

## 8. Advanced Use Cases

`dataStrategy`는 다양한 고급 데이터 처리 전략을 구현할 때 사용할 수 있습니다.

예를 들어 custom middleware 시스템을 직접 구현할 수 있습니다.

```ts
for (const match of matches) {
  if (match.route.handle?.middleware) {
    await match.route.handle.middleware(
      { request, params },
      context,
    );
  }
}
```

또한 모든 라우트 데이터를 하나의 GraphQL 요청으로 처리하는 전략도 구현할 수 있습니다.

```ts
let gql = getFragmentsFromRouteHandles(matchesToLoad);
let data = await fetchGql(gql);
let results = parseResultsFromGql(matchesToLoad, data);
return results;
```
