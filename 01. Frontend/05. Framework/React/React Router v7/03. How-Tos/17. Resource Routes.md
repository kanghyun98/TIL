# Resource Routes

서버 렌더링 시, 라우트는 컴포넌트를 렌더링하는 대신 이미지, PDF, JSON payload, webhook 같은 "resource"를 제공할 수 있습니다.

<br>

## 1. Defining a Resource Route

모듈이 `loader` 또는 `action`을 export하고 `default` 컴포넌트를 export하지 않으면, 관례에 따라 해당 라우트는 리소스 라우트가 됩니다.

UI 대신 PDF를 제공하는 라우트는 다음과 같습니다.

```ts
route("/reports/pdf/:id", "pdf-report.ts");
```

```tsx
// pdf-report.ts
import type { Route } from "./+types/pdf-report";

export async function loader({ params }: Route.LoaderArgs) {
  const report = await getReport(params.id);
  const pdf = await generateReportPDF(report);
  return new Response(pdf, {
    status: 200,
    headers: {
      "Content-Type": "application/pdf",
    },
  });
}
```

여기에는 `default` export가 없습니다. 그래서 이 라우트는 리소스 라우트가 됩니다.

<br>

## 2. Linking to Resource Routes

리소스 라우트로 연결할 때는 `<a>` 또는 `<Link reloadDocument>`를 사용합니다. 그렇지 않으면 React Router가 클라이언트 사이드 라우팅을 시도하고 payload를 가져오려고 합니다. 이 실수를 하면 도움이 되는 에러 메시지가 표시됩니다.

```tsx
<Link reloadDocument to="/reports/pdf/123">
  View as PDF
</Link>
```

<br>

## 3. Handling different request methods

GET 요청은 `loader`가 처리하고, POST, PUT, PATCH, DELETE는 `action`이 처리합니다.

```tsx
import type { Route } from "./+types/resource";

export function loader(_: Route.LoaderArgs) {
  return Response.json({ message: "I handle GET" });
}

export function action(_: Route.ActionArgs) {
  return Response.json({
    message: "I handle everything else",
  });
}
```

<br>

## 4. Return Types

리소스 라우트는 반환 타입 측면에서 유연합니다. `Response` 인스턴스나 `data()` 객체를 반환할 수 있습니다.

어떤 타입을 사용할지 결정할 때 참고할 수 있는 일반적인 기준은 다음과 같습니다.

- 외부 소비를 의도한 리소스 라우트라면 `Response` 인스턴스를 반환합니다.
  - 결과 응답의 인코딩이 코드에서 명시적으로 유지되며, React Router가 내부적으로 `data() -> Response`를 어떻게 변환하는지 고민할 필요가 없습니다.
- `fetchers` 또는 `<Form>` 제출에서 리소스 라우트에 접근한다면 `data()`를 반환합니다.
  - UI 라우트의 loader/action과 일관성을 유지할 수 있습니다.
  - `data()`/`Await`를 통해 promise를 UI로 스트리밍할 수 있습니다.

<br>

## 5. Error Handling

리소스 라우트에서 `Error`를 throw하거나 `Response`/`data()`가 아닌 값을 throw하면 `handleError`가 트리거되고 500 HTTP Response가 반환됩니다.

```tsx
export function action() {
  let db = await getDb();
  if (!db) {
    // Fatal error - return a 500 response and trigger `handleError`
    throw new Error("Could not connect to DB");
  }
  // ...
}
```

리소스 라우트가 `Response`를 생성하면 `new Response()` 또는 `data()`를 통해 생성된 경우라도 성공적으로 실행된 것으로 간주되며 `handleError`를 트리거하지 않습니다. HTTP 요청에 대해 API가 성공적으로 `Response`를 생성했기 때문입니다. 이 규칙은 4xx/5xx status code를 가진 응답을 throw한 경우와 return한 경우 모두에 적용됩니다. 이 동작은 4xx/5xx `Response`에 대해 rejected promise를 반환하지 않는 `fetch()`와 일치합니다.

```tsx
export function action() {
  // Non-fatal error - don't trigger `handleError`:
  throw new Response(
    { error: "Unauthorized" },
    { status: 401 },
  );

  // These 3 are equivalent to the above
  return new Response(
    { error: "Unauthorized" },
    { status: 401 },
  );

  throw data({ error: "Unauthorized" }, { status: 401 });

  return data({ error: "Unauthorized" }, { status: 401 });
}
```

<br>

### 5.1 Error Boundaries

`Error Boundaries`는 리소스 라우트가 UI에서 접근될 때만 적용됩니다. 예를 들어 `fetcher` 호출이나 `<Form>` 제출이 여기에 해당합니다. 이런 경우 리소스 라우트에서 `throw`하면, 에러는 UI에서 가장 가까운 `ErrorBoundary`까지 버블링됩니다.
