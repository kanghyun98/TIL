# Single Page App (SPA)

> 이 가이드는 Framework mode에서 싱글 페이지 앱을 구축하는 방법에 초점을 맞춥니다. Declarative mode나 Data mode에서 사용하고 있다면, 자체적인 SPA 아키텍처를 설계할 수 있습니다.

프레임워크로 사용할 때는 `react-router.config.ts` 파일에서 `ssr: false`를 설정해 "SPA Mode"를 활성화할 수 있습니다. 이렇게 하면 런타임 서버 렌더링이 비활성화되고, 빌드 시점에 `index.html`이 생성되어 이를 SPA로 서빙하고 hydrate할 수 있습니다.

일반적인 싱글 페이지 앱은 대부분 비어 있는 `index.html` 템플릿을 전송하며, 보통은 비어 있는 `<div id="root"></div>` 정도만 포함합니다. 반면 SPA Mode에서 `react-router build`는 빌드 시점에 루트 라우트를 `index.html` 파일로 미리 렌더링합니다. 이는 다음을 의미합니다.

- 비어 있는 `<div>`보다 더 많은 내용을 보낼 수 있습니다
- 애플리케이션 셸 데이터를 불러오기 위해 루트 `loader`를 사용할 수 있습니다
- 사용자가 처음 보게 되는 페이지를 React 컴포넌트로 생성할 수 있습니다(루트 `HydrateFallback`)
- UI를 변경하지 않고 나중에 서버 렌더링을 다시 활성화할 수 있습니다

> SPA Mode는 애플리케이션의 모든 경로를 동일한 HTML 파일에서 제공할 수 있게 해주는 특별한 형태의 "Pre-Rendering"입니다. 더 광범위한 사전 렌더링이 필요하다면 Pre-Rendering 가이드를 참고하면 됩니다.

<br>

## 1. 런타임 서버 렌더링 비활성화

서버 렌더링은 기본적으로 활성화되어 있습니다. 이를 비활성화하려면 `react-router.config.ts`에서 `ssr` 플래그를 `false`로 설정합니다.

```ts
import { type Config } from "@react-router/dev/config";

export default {
  ssr: false,
} satisfies Config;
```

이 값을 `false`로 설정하면 서버 빌드는 더 이상 생성되지 않습니다.

중요한 점은 `ssr: false`가 런타임 서버 렌더링만 비활성화한다는 것입니다. 빌드 시점에 `index.html` 파일을 생성하기 위해 루트 라우트는 여전히 서버 렌더링됩니다. 그래서 프로젝트에는 여전히 `@react-router/node` 의존성이 필요하고, 라우트는 SSR에 안전해야 합니다. 즉, 서버 렌더링을 비활성화했더라도 초기 렌더링 중에는 `window` 같은 브라우저 전용 API를 호출할 수 없습니다.

<br>

## 2. 루트 라우트에 `HydrateFallback`과 선택적 `loader` 추가

SPA Mode는 빌드 시점에 `index.html` 파일을 생성하며, 이 파일을 SPA의 진입점으로 서빙할 수 있습니다. 이때 애플리케이션의 어떤 경로에서도 런타임 hydration이 가능하도록 루트 라우트만 렌더링됩니다.

비어 있는 `<div>`보다 더 나은 로딩 UI를 제공하려면 루트 라우트에 `HydrateFallback` 컴포넌트를 추가할 수 있습니다. 이렇게 하면 빌드 시점에 로딩 UI가 `index.html`에 렌더링되고, SPA가 로딩되고 hydrate되는 동안 즉시 사용자에게 표시됩니다.

```tsx
import LoadingScreen from "./components/loading-screen";

export function Layout() {
  return <html>{/* ... */}</html>;
}

export function HydrateFallback() {
  return <LoadingScreen />;
}

export default function App() {
  return <Outlet />;
}
```

루트 라우트는 빌드 시점에 서버 렌더링되므로, 필요하다면 루트 라우트에서 `loader`도 사용할 수 있습니다. 이 `loader`는 빌드 시점에 호출되며, 데이터는 선택적인 `HydrateFallback`의 `loaderData` prop으로 전달됩니다.

```tsx
import { Route } from "./+types/root";

export async function loader() {
  return {
    version: await getVersion(),
  };
}

export function HydrateFallback({
  loaderData,
}: Route.ComponentProps) {
  return (
    <div>
      <h1>Loading version {loaderData.version}...</h1>
      <AwesomeSpinner />
    </div>
  );
}
```

SPA Mode를 사용할 때는 다른 페이지들을 pre-rendering하지 않는 한, 앱의 다른 라우트에는 `loader`를 포함할 수 없습니다.

<br>

## 3. client loaders와 client actions 사용

서버 렌더링이 비활성화되어 있어도, 라우트 데이터와 변경 작업을 관리하기 위해 `clientLoader`와 `clientAction`을 사용할 수 있습니다.

```tsx
import { Route } from "./+types/some-route";

export async function clientLoader({
  params,
}: Route.ClientLoaderArgs) {
  let data = await fetch(`/some/api/stuff/${params.id}`);
  return data;
}

export async function clientAction({
  request,
}: Route.ClientActionArgs) {
  let formData = await request.formData();
  return await processPayment(formData);
}
```

## 4. 모든 URL을 `index.html`로 연결

`react-router build`를 실행한 뒤에는 `build/client` 디렉터리를 원하는 정적 호스팅 환경에 배포합니다.

일반적인 SPA 배포와 마찬가지로, 호스트가 애플리케이션의 모든 URL을 클라이언트 빌드의 `index.html`로 연결하도록 설정해야 합니다. 어떤 호스트는 이를 기본적으로 처리하지만, 그렇지 않은 경우도 있습니다. 예를 들어 `_redirects` 파일을 지원하는 호스트라면 다음과 같이 구성할 수 있습니다.

```txt
/*    /index.html   200
```

유효한 라우트에서 404가 발생한다면, 호스트 설정이 필요한 경우일 가능성이 높습니다.
