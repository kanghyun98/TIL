# Error Boundaries

React Router에서는 사용자에게 빈 페이지를 보여주지 않기 위해, 라우트 모듈에서 발생한 오류를 자동으로 잡고 **가장 가까운 `ErrorBoundary`**를 렌더링합니다.

다만 Error Boundary는 **폼 검증 오류를 렌더링하거나 에러 리포팅을 위한 용도는 아닙니다.**  
이런 경우에는 별도의 방식으로 처리해야 합니다.

<br>

## 1. Add a root error boundary

모든 애플리케이션은 최소한 **루트 에러 바운더리(root error boundary)**를 하나는 export하는 것이 좋습니다.

이 루트 에러 바운더리는 다음 세 가지 주요 경우를 처리합니다.

- status code와 text를 가진 `data`가 throw된 경우
- stack trace를 가진 `Error` 인스턴스
- 임의로 throw된 값

<br>

### 1.1 Framework Mode

Framework Mode에서는 오류가 라우트 수준의 error boundary에 **prop으로 전달**됩니다.  
따라서 별도의 훅을 사용해서 오류를 가져올 필요가 없습니다.

```ts
import { Route } from "./+types/root";

export function ErrorBoundary({
  error,
}: Route.ErrorBoundaryProps) {
  if (isRouteErrorResponse(error)) {
    return (
      <>
        <h1>
          {error.status} {error.statusText}
        </h1>
        <p>{error.data}</p>
      </>
    );
  } else if (error instanceof Error) {
    return (
      <div>
        <h1>Error</h1>
        <p>{error.message}</p>
        <p>The stack trace is:</p>
        <pre>{error.stack}</pre>
      </div>
    );
  } else {
    return <h1>Unknown Error</h1>;
  }
}
```

<br>

## 2. Write a bug

Error Boundary를 **의도적으로 렌더링시키기 위한 제어 흐름 수단으로 오류를 던지는 것은 권장되지 않습니다.**  
Error Boundary는 기본적으로 **의도하지 않은 오류를 잡기 위한 목적**입니다.

```ts
export async function loader() {
  return undefined();
}
```

이 경우 앞에서 작성한 UI 중 `instanceof Error` 분기가 렌더링됩니다.

이 동작은 loader에만 해당하지 않습니다.  다음과 같은 모든 route module API에서 동일하게 적용됩니다.

- loaders
- actions
- components
- headers
- links
- meta

<br>

## 3. Throw data in loaders/actions

`#2`의 규칙에는 예외가 있습니다.  대표적으로 **404 같은 상태 코드 처리**입니다.

loader가 페이지를 렌더링하는 데 필요한 데이터를 찾지 못한 경우에는  
적절한 status code를 담아 **의도적으로 `throw data()`**를 사용할 수 있습니다.

```ts
import { data } from "react-router";

export async function loader({ params }) {
  let record = await fakeDb.getRecord(params.id);
  if (!record) {
    throw data("Record Not Found", { status: 404 });
  }
  return record;
}
```

이 경우 앞에서 작성한 UI 중 `isRouteErrorResponse` 분기가 렌더링됩니다.

<br>

## 4. Nested error boundaries

오류가 발생하면 **가장 가까운 error boundary**가 렌더링됩니다.

<br>

### 4.1 Framework Mode

다음과 같은 중첩 라우트를 가정합니다.

```ts
// ✅ has error boundary
route("/app", "app.tsx", [
  // ❌ no error boundary
  route("invoices", "invoices.tsx", [
    // ✅ has error boundary
    route("invoices/:id", "invoice-page.tsx", [
      // ❌ no error boundary
      route("payments", "payments.tsx"),
    ]),
  ]),
]);
```

이때 오류 발생 위치에 따라 렌더링되는 error boundary는 다음과 같습니다.

| error origin       | rendered boundary  |
| ------------------ | ------------------ |
| `app.tsx`          | `app.tsx`          |
| `invoices.tsx`     | `app.tsx`          |
| `invoice-page.tsx` | `invoice-page.tsx` |
| `payments.tsx`     | `invoice-page.tsx` |

<br>

## 5. Error Sanitization

Framework Mode에서 프로덕션 빌드를 사용하는 경우, 서버에서 발생한 오류는 브라우저로 전달되기 전에 **자동으로 sanitize**됩니다.

이렇게 하면 stack trace 같은 민감한 서버 정보가 브라우저에 노출되는 것을 막을 수 있습니다.

즉, 브라우저의 프로덕션 환경에서는 throw된 `Error`가

- 일반적인(generic) 메시지를 가지게 되고
- stack trace는 포함하지 않게 됩니다.

원본 오류는 서버에서는 그대로 유지됩니다.

다만 `throw data(yourData)`로 전달한 데이터는 **렌더링을 의도한 데이터이기 때문에 sanitize되지 않습니다.**
